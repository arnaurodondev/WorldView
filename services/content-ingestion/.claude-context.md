# Content Ingestion Service (S4) — Agent Context

**Port**: 8004 | **DB**: `content_ingestion_db` | **Status**: Compliant

## Mission
Scheduled polling of EODHD global news, EODHD per-ticker news (`eodhd_ticker_news`, auto-created per equity by `TickerNewsSymbolSyncWorker`), SEC EDGAR filings, Finnhub, NewsAPI, and Polymarket prediction markets. Domain allowlists, rate limiting per source, fetch-level dedup, raw payload storage to MinIO bronze. Also consumes `nlp.document.ready.v1` to flip tenant uploads to `ready`. `SourceType` = `contracts.enums.ContentSourceType` (eodhd, eodhd_ticker_news, sec_edgar, finnhub, newsapi, polymarket, manual).

## EODHD Shared Quota Accounting (blind-spot fix 2026-07-01)
S4 is the LARGEST EODHD consumer (per-ticker news ~520 req/h) but historically wrote NO Valkey quota keys, so the account-wide monthly total (`eodhd:v1:quota:{YYYY-MM}:credits_used`) reflected ONLY market-ingestion (S2) → dashboards/alerts materially undercounted and June exhaustion fired with no freshness alert. Fix: both EODHD adapters (`eodhd/client.py::fetch_news`, `eodhd_ticker_news/adapter.py::_fetch_page`) now call `infrastructure/adapters/eodhd_quota.py::record_eodhd_request` per HTTP request, which increments the SAME shared counters S2 uses via `messaging.eodhd_quota.EodhdQuotaService.record_usage` (best-effort — a Valkey failure logs a WARNING, never breaks ingestion). Per-request cost = `eodhd.credits_per_request` (default 5, config `CONTENT_INGESTION_EODHD__CREDITS_PER_REQUEST`). The worker builds ONE `EodhdQuotaService` from its `ValkeyClient` and threads it into both adapters. **Requires a content-ingestion redeploy to activate.** The shared-lib `record_usage` addition is additive (S2 unaffected — no S2 redeploy needed).

**DAILY vs MONTHLY enforcement (INCIDENT 2026-07-03).** EODHD's REAL limit is **100k API calls PER UTC DAY**, not per month (verified via `GET /api/user`: `dailyRateLimit=100000`, `apiRequests`=used-today resets daily). The old guard hard-blocked on the MONTHLY counter with a 100k cap, so normal daily usage (~75k credits/day) blew past 100k ~1-1.5 days into every month and STAYED blocked until the monthly key's TTL expired — artificially starving ingestion (June "earnings empty", July-03 news halt). `EodhdQuotaService` now enforces the **DAILY** hard cap (`eodhd_daily_quota` / `CONTENT_INGESTION_EODHD_DAILY_QUOTA`, default 100k) against the per-UTC-day counter `eodhd:v1:quota:day:{YYYY-MM-DD}:credits_used`; the monthly counter (`eodhd_monthly_quota`, default 100k) is retained for cross-service **reporting/attribution ONLY and no longer blocks**. Both `_daily_quota` values must match S2's. `DailyBudgetTracker` now derives headroom from the real daily cap (no 30.4-day amortisation).

## Process Architecture (R22 — PLAN-0006)
5 independent processes, same Dockerfile, different `command` overrides:

| Process | Entry Point | Description |
|---------|------------|-------------|
| API | `uvicorn content_ingestion.app:create_app --factory` | HTTP endpoints only — no background processing |
| Scheduler | `python -m content_ingestion.infrastructure.scheduler.scheduler_main` | Tick-based (`SchedulerProcess`): evaluates sources, creates task rows; also runs `TickerNewsSymbolSyncWorker` |
| Worker | `python -m content_ingestion.infrastructure.workers.worker_main` | Claims tasks, executes fetch-and-write pipeline |
| Dispatcher | `python -m content_ingestion.infrastructure.messaging.outbox.dispatcher_main` | Publishes outbox events to Kafka |
| Document-ready consumer | `python -m content_ingestion.infrastructure.messaging.consumers.document_ready_consumer_main` | Consumes `nlp.document.ready.v1`, marks tenant uploads `ready` |

## Advisory Lock & Watermarks
- Advisory lock from shared `messaging.pg.advisory_lock` (SHA-256 hashing, deterministic)
- Lock held only during DB writes, NOT during external API fetches (prevents pool exhaustion)
- Watermarks: `source_adapter_state.last_watermark` drives incremental polling via `from_date` parameter
- Adapters accept `from_date` kwarg which overrides `source.config["from_date"]`
- **EODHD news batch sweep (QUOTA-OPT 2026-06-16)**: EODHD `/api/news` bills a FLAT 5 credits + 5/ticker PER REQUEST regardless of article count, so one request pulls the WHOLE batch since the watermark. Adapters build `from = watermark - eodhd.news_watermark_overlap_days`, `to = today`, `limit = eodhd.news_page_limit` (1000, EODHD max), and paginate via `offset` ONLY when a page comes back full — a normal incremental run is ONE call. First run / empty watermark uses a BOUNDED backfill (`today - backfill_initial_days`), never "since epoch". After a non-empty sweep, `execute_task._newest_published_at()` advances `last_watermark` to the NEWEST article's `published_at` (not wall-clock `now`). On HTTP 429 the adapter raises `ProviderRateLimited`/`AdapterError` (retryable, NOT fatal) → task RETRY with back-off and watermark NOT advanced. Intra-sweep url_hash dedup + downstream `FetchAndWriteUseCase` dedup absorb the overlap re-fetch.

## Pitfalls
- All TIMESTAMP columns must be TIMESTAMPTZ (not TIMESTAMP)
- DB + Kafka writes must be same transaction (outbox pattern)
- Always use `common.ids.new_uuid7()` for PKs, not `uuid.uuid4()`
- Always use `common.time.utc_now()`, never `datetime.now()` or `datetime.utcnow()`
- Mocking `asyncio.to_thread` must patch at call site
- FastAPI route files must NOT use `from __future__ import annotations` (breaks DI resolution)
- TokenBucket requires all 4 args: capacity, tokens, refill_rate, last_refill
- Never use Python `hash()` for advisory locks — use `hashlib.sha256` (BP-015)
- Outbox payload field names must match Avro schema exactly (BP-017)
- httpx.AsyncClient shared in lifespan, closed on shutdown — never per-request
- Session rollback after IntegrityError before continuing loop
- DLQ `move_to_dead_letter` stores `payload_json` for requeue
- PolymarketAdapter uses `fetch_log_exists_fn(market_id, snapshot_at)` — different dedup signature from standard `exists_fn(url_hash)` (BP-057: no DB session in adapter itself)
- POLYMARKET worker path bypasses `ExecuteContentTaskUseCase` — uses `_execute_polymarket_task()` directly in `WorkerProcess`
- `SECEdgarAdapter._is_market_hours()` requires UTC-aware datetime input; uses `zoneinfo.ZoneInfo` (never pytz)
- `SECEdgarAdapter.fetch()` (R1 Fix ①/②, 2026-07-04): fetches the **primary filing document**, not the `…-index.htm` page. Flow per filing: `client.fetch_filing_manifest()` (GET `{acc}/index.json`) → `resolve_primary_document()` (largest form-matching `.htm`, excludes XBRL/`R\d+.htm`/`_cal/_def/_lab/_pre/_ref`/exhibits) → `fetch_filing_document(primary)`. Filings with no groundable HTML are **skipped** (never store index boilerplate). `_synthesize_title()` sets `FetchResult.title` = `"{FORM} — {Company} ({Period})"` so `document_source_metadata.title` is populated (was NULL 100%). Coverage: `config["ciks"]` watchlist (list or comma-str) scopes EFTS **per-CIK** (`ciks=` param) — without it the global date-sorted search never surfaces watched companies like Apple. Migration `0010` seeds the mega-cap CIK watchlist. `contracts.trust.SOURCE_AUTHORITY` gained `sec_edgar=1.00` (was falling to `default` 0.50). Backfill for the ~4953 pre-fix docs: `scripts/ops/backfill_sec_edgar_primary_docs.py` (dry-run default).

## MinIO Compensating GC (PLAN-0001-B-R3)
- `BronzeStoragePort.delete_object(key)` removes a bronze object (best-effort orphan GC)
- `FetchAndWriteUseCase` tracks `pending_minio_keys` per batch; GC on any exception before commit
- GC failures are logged as WARNING and must not mask the original exception
- Pattern: append key after `put_object`, clear list after successful `commit_fn()`, loop-delete on failure
- Test: `tests/unit/application/test_minio_gc.py`

## ContentIngestionTask State Machine (PLAN-0006 Wave B-1)
- Entity: `domain/entities.py::ContentIngestionTask` — unit of work for scheduler-worker pattern
- States: PENDING → CLAIMED → RUNNING → SUCCEEDED / RETRY / FAILED
- `claim(worker_id, lease_seconds)`: PENDING/RETRY → CLAIMED
- `start()`: CLAIMED → RUNNING (increments attempt_count)
- `succeed()`: RUNNING → SUCCEEDED
- `fail(error)`: RUNNING → RETRY (if attempts left) or FAILED (terminal)
- Factory: `create_for_source(source, is_backfill, window_start)`
- Config: `db_url_read`, `scheduler_tick_interval_seconds`, `worker_batch_size`, `worker_lease_seconds`, `worker_concurrency`, etc.

  - **Existing**: `FetchLogPort` (+ `count_by_source_since`), `OutboxPort` (+ `count_pending`), `BronzeStoragePort`
  - **New**: `SourcePort`, `TaskPort`, `AdapterStatePort` (+ `get_all`), `DLQPort` (+ `count_failed`)
- `application/ports/metrics.py` — `MetricsPort` protocol (`record_fetch()`)
- `application/ports/source_adapter.py` — `SourceAdapterPort` ABC
- `application/ports/unit_of_work.py` — UoW properties typed as ports (not concrete repos)
- All ports are `@runtime_checkable` `Protocol` classes — mirrors 1:1 with infrastructure repos

## ExecuteContentTaskUseCase (PLAN-0009 Wave C-1)
- Keyword-only constructor with 7 injected dependencies — zero infrastructure imports (R25)
- `adapter_state_factory`, `fetch_log_factory`, `outbox_factory`: session → port callables (wired by worker)
- `bronze`: pre-built `BronzeStoragePort` (injected by worker)
- `adapter_builder`: `(SourceType, exists_fn) → SourceAdapterPort` (worker's `_build_adapter`)
- Metrics recording (`record_fetch`) moved to worker — called post-execute on success
- Worker's `_build_adapter` encapsulates all client construction, rate-limiter setup, and adapter registry lookup
