## Mission

## Key Pitfalls
- **Design decision: enriched vs. temporal event filtering (DP-F001)**: Enriched events drop mentions that resolve to neither a canonical entity nor a provisional queue id ŌĆö they would have no edge to anchor (see `infrastructure/messaging/consumers/blocks/enriched_event.py` `_build_raw_*`). Temporal events keep all events regardless of resolvable participants ŌĆö macro events (CPI, FOMC, sanctions) legitimately have none, and S7's `TemporalEventConsumer` accepts `exposed_entities=[]`. Do not "fix" this asymmetry; it is intentional and was flagged DP-F001 in `docs/audits/2026-05-24-investigation-qa-open-items.md`.
- **HNSW indexes must use partial predicate**: `WHERE (expires_at IS NULL OR expires_at > now())`; use `op.execute()` in Alembic (not native HNSW support)
- **Chunk and section embeddings use SEPARATE HNSW indexes** ŌĆö mixing granularities pollutes ANN recall
- **`routing_decisions.composite_score` is the routing score** ŌĆö do NOT duplicate into `document_source_metadata`; access via JOIN on `doc_id`
- **Routing signal v2 (PLAN-0093 C-1, 2026-05-23)**: `compute_routing_score()` accepts ONLY 5 live signals ŌĆö `entity_density` (0.35), `source_reliability` (0.30), `recency` (0.15), `document_type` (0.10), `extraction_yield` (0.10). The 3 dead signals (`watchlist`/`novelty`/`price_impact`) were dropped because they could never fire in single-pass routing. Tier thresholds recalibrated: DEEP=0.75, MEDIUM=0.45, LIGHT=0.20. Re-adding the dropped signals requires two-pass routing first.
- **entity_mentions persistence floor (PLAN-0093 C-2, 2026-05-23, F-NPL-005)**: `Settings.min_persist_floor` (default 0.6, env `NLP_PIPELINE_MIN_PERSIST_FLOOR`) gates writes to the `entity_mentions` table ŌĆö sub-floor mentions are dropped at `persist_artifacts()` write time AND filtered out by `EntityMentionRepository.get_unresolved_batch{,_with_context}()` SQL (`WHERE confidence >= :min_confidence`) for defense-in-depth. `chunks.entity_mentions` JSONB still uses `gliner_mention_floor` (same default 0.6) ŌĆö both should stay in sync.
- **Unknown-ticker skip-set (PLAN-0093 C-3, 2026-05-23, F-NPL-FUNDAMENTALS-001)**: `MarketDataClient._resolve_instrument_id` accepts an optional `valkey_client`; after 3 consecutive 404s for a ticker, the client writes a 7-day marker at `nlp:price_impact:unknown_tickers:<TICKER>` so subsequent worker cycles skip the round-trip. Per-ticker fail counter at `nlp:price_impact:unknown_tickers:fails:<TICKER>` with 1-day rolling TTL. All Valkey ops are best-effort ŌĆö a Valkey outage falls through to the network path (degraded but functional). Wire the Valkey client when constructing `MarketDataClient` in production; tests + dev workflows pass `valkey_client=None`.
- **impact_score writer (PLAN-0093 C-3, 2026-05-23, F-NPL-008)**: `ArticleRelevanceScoringWorker` now writes `document_source_metadata.impact_score = MAX(impact_score)` from `article_impact_windows` in the same phase-3 session (single UPDATEŌĆ”FROM batched per cycle). Stays NULL when no `article_impact_windows` row exists (PRD-0026 ┬¦6.5 ŌĆö "not yet labelled" vs "zero impact" must remain distinguishable in the weighted-display formula).
- **`ArticleRelevanceScoringWorker` must follow R24 3-phase**: read batch ŌåÆ close session ŌåÆ Ollama calls ŌåÆ open new session ŌåÆ batch UPDATE; never hold DB session open during Ollama calls
- **Worker routing tier filter uses `COALESCE(rd.final_routing_tier, rd.routing_tier)`** ŌĆö novelty-downgraded articles have `final_routing_tier='LIGHT'` but `routing_tier='DEEP'`
- **`day_t2/day_t5` are cumulative from `close_t0`** ŌĆö fetch day_t0 bar first; use its close as baseline
- **`DisplayRelevanceScore.value` uses `is not None` not `or 0.0`** ŌĆö `mi=0.0` (zero price movement) and `mi=None` (not yet labelled) have different semantics
- **`NLP_PIPELINE_SERVICE_ACCOUNT_TOKEN` required in production** (BP-303) ŌĆö without it, PriceImpactLabellingWorker falls back to `dev-login` which is blocked in production
- **Ollama JSON mode**: pass `"format": "json"` in API body ŌĆö prompt-only instruction is insufficient
- **`UNNEST pg_trgm batch`**: use `bindparam('terms', type_=sa.ARRAY(sa.Text()))` not `::type` cast syntax (asyncpg rejects it, BP-076)
- **`FailureInfo[None]` has NO `value`/`key`/`headers` fields** ŌĆö only `event_id`, `topic`, `partition`, `offset`, `attempt`, `last_error`, `record`
- **CanonicalTickersCache atomic swap uses Lua script** (`_ATOMIC_TICKER_SWAP`) not MULTI/EXEC ŌĆö eliminates partial-clear race on network drop between DEL and SADD (BP-422)
- **Legacy article passthrough (BP-575, PLAN-0096 W4)**: pre-PLAN-0086 Avro `content.article.stored.v1` payloads have no `tenant_id` field; migration 0020 made `entity_mentions.tenant_id` NOT NULL, so legacy messages used to crash every retry and stall the topic (94 stuck messages observed 2026-05-26). `ArticleProcessingConsumer.process_message` now substitutes `common.ids.PUBLIC_TENANT_ID` (`00000000-0000-0000-0000-000000000000`) when both headers and payload omit `tenant_id`, emitting `article_consumer.legacy_tenant_id_sentinel_applied` WARN. New producers MUST stamp a real tenant ŌĆö the sentinel only protects the in-flight backlog. Tooling: `scripts/replay_stuck_articles.py` for lag/peek; `infra/prometheus/rules/nlp_pipeline_retry_storm.yml` for the stall alert.
- **Pre-persist tenant_id substitution counter must be labelled by upstream block (PLAN-0099 W2 T-W2-04)**: the safety net at `article_consumer._run_pipeline` increments `nlp_pipeline_pre_persist_tenant_id_substituted_total{block_source=...}` once per substituted mention. `block_source` is a **bounded enum** (`ner` / `entity_resolution` / `novelty_backfill` / `deep_extraction` / `unknown`) declared in `infrastructure.metrics.prometheus.PRE_PERSIST_BLOCK_SOURCES` ŌĆö NEVER pass free-form strings. The classifier `_infer_block_source(mention)` in `article_consumer.py` maps to the enum by inspecting `provisional_queue_id` / `resolution_outcome` / `ner_model_id`; any new block that constructs `EntityMention` instances must either propagate `tenant_id` correctly (preferred) OR stamp a domain field that lets the classifier attribute the leak ŌĆö otherwise the substitution is bucketed into `"unknown"` and root-cause is opaque. Operators query e.g. `sum by (block_source) (rate(nlp_pipeline_pre_persist_tenant_id_substituted_total[5m]))` to identify the dominant null-tenant source for PLAN-0100 ┬¦13.4 remediation. Tests: `tests/unit/infrastructure/messaging/consumers/test_article_consumer_tenant_id_propagation.py::TestPrePersistTenantIdSubstitutedCounter`.
- **Per-mention tenant_id stamp must be unconditional + pre-persist guard (BP-586, PLAN-0098 W2 T-W2-01)**: `ArticleProcessingConsumer._run_pipeline` historically guarded the post-NER per-mention stamp with `if tenant_id is not None`, but the envelope-level sentinel substitution (BP-575 fix) already guarantees `tenant_id` is non-None at that point ŌĆö the guard was redundant AND left the door open for any *downstream* block (entity resolution, deep extraction, novelty backfill) to leak a mention with `tenant_id=None` into `ml.final_mentions` and through to the INSERT, where the NOT NULL constraint (migration 0020) fails the row, retries the message forever, and pins `entity_mentions` at 0 rows even when `routing_decisions` and `chunks` populate normally. Fix is two-part: (a) drop the redundant guard on the post-NER stamp; (b) add a pre-persist safety net immediately before `persist_artifacts(ml=ml, ...)` that iterates `ml.final_mentions`, substitutes the request-scoped tenant (falling back to `PUBLIC_TENANT_ID`) for any `None`, and emits `article_consumer.pre_persist_tenant_id_substituted` WARN with the offending mention IDs so an operator can trace the regressed block. Any new block constructing `EntityMention` instances MUST accept `tenant_id` as a parameter; the pre-persist guard is defence-in-depth, NOT a substitute for proper propagation. Tests: `tests/unit/infrastructure/messaging/consumers/test_article_consumer_tenant_id_propagation.py`.
- **R40 sentinel-tenant visibility ŌĆö three-leg disjunction REQUIRED on EVERY tenant SQL filter (BP-583, PLAN-0098)**: any WHERE clause that currently reads `tenant_id IS NULL OR tenant_id = :tenant_id` MUST also admit the PUBLIC_TENANT_ID literal `'00000000-0000-0000-0000-000000000000'::uuid` ŌĆö otherwise PLAN-0096 W4 fallback rows (BP-575) are invisible to every authenticated tenant. Canonical sites in nlp-pipeline: `news_query._ENTITY_ARTICLES_SQL` and `chunk_search.py` (`_search_chunks`, `_search_sections`, sections-COUNT, `lexical_search`). Use the hardcoded literal, NOT a bind variable, to match `news_query` (commit 8c3a9249). Before adding any new tenant-filtered SQL, `grep -rn "tenant_id IS NULL OR tenant_id = :" services/nlp-pipeline/src/` and confirm zero hits. Tests: `tests/unit/infrastructure/test_tenant_id_chunk_isolation.py::TestChunkANNRepositoryPublicTenantSentinel` and `test_news_query_tenant_filter.py::test_entity_articles_sql_includes_public_tenant_sentinel`.
- **MarketDataClient JWT-mint cold-start race (BP-562, PLAN-0096 W2 T-W2-01)**: when `PriceImpactLabellingWorker` boots ahead of api-gateway during `docker compose up`, the first JWT-mint `POST` surfaces `httpx.ConnectError`. The client retries up to 4 attempts with exponential backoff `(0s, 5s, 15s, 45s)` (~65s grace), then returns `None` so the caller falls through to the legacy 401-and-warn path instead of crashing the worker loop. Delays live in `_TOKEN_MINT_RETRY_DELAYS` in `infrastructure/http/market_data_client.py`; tests in `tests/unit/infrastructure/test_market_data_client_jwt_backoff.py` patch `asyncio.sleep`.
- **Relevance-scoring silent Ollama fallback (BP-563, PLAN-0096 W2 T-W2-02)**: `ArticleRelevanceScoringWorker` branches on `api_key`; when `NLP_PIPELINE_RELEVANCE_SCORING_API_KEY` is empty in `docker.env`, the worker silently falls back to Ollama and starves under any Ollama outage (0 articles scored per cycle, no error log). The deployed `docker.env` now exports the DeepInfra key and `docker.env.example` documents the contract. Provider-selection contract pinned by `tests/unit/infrastructure/workers/test_article_relevance_scoring_worker.py::TestProviderSelectionContract`. Verify on live with `docker compose exec nlp-pipeline-relevance-scoring env | grep RELEVANCE_SCORING`.
- **PLAN-0100 W1 BP-606 entity-articles JSONB-fallback (`news_query.py`)**: `_ENTITY_ARTICLES_SQL`'s `entity_article_ids` CTE UNIONs two discovery paths: (a) normalised `entity_mentions` table SELECT and (b) denormalised `chunks.entity_mentions @> jsonb_build_array(jsonb_build_object('entity_id', CAST(:entity_id AS TEXT)))` SELECT. The denormalised leg covers low-coverage tickers (MSTR canary: 8 chunks with JSONB entity_id, 0 rows in normalised table) ŌĆö without it the chat retrieval returns empty and the LLM substitutes a different ticker (HARMFUL Q2 fault). Both legs honour the R35 three-row-class tenant filter (NULL + tenant + PUBLIC_TENANT_ID sentinel); GIN index on `chunks.entity_mentions` accelerates Leg-2. Any future denormalised/normalised dual-write pair MUST have its read-side aggregator UNION both sources OR a CI parity fixture ŌĆö lineage gaps surface only post-facto in chat-eval. Companion: `scripts/backfill_entity_mentions_from_chunks_jsonb.py` (idempotent dry-run-default). Tests: `tests/unit/infrastructure/test_news_query_jsonb_fallback.py`.
