# NLP Pipeline Service (S6) — Agent Context

**Port**: 8006 | **DB**: `nlp_db` (owned) + `intelligence_db` (read/write, DDL NOT owned) | **Status**: Feature-complete (C-1 + C-2 + C-3 + C-4 done)

## Mission
Full intelligence enrichment: sectioning, GLiNER NER (**11 classes**), routing score, suppression, chunk/section embeddings (BGE 1024-dim), novelty gate, entity resolution cascade, deep LLM extraction (Qwen2.5-7B).

## Domain Entities
Section (with `speaker` field for transcripts), Chunk, EntityMention (with `resolution_stage`), MentionResolution (audit trail per cascade stage), DocumentEntityStats, ChunkEmbedding, SectionEmbedding, RoutingDecision (with `final_routing_tier`), **DocumentSourceMetadata** (citation cache for S8 RAG, Wave B-1)

- Qwen2.5-7B extraction via `libs/ml-clients` — containerised via Ollama at `OLLAMA_BASE_URL`

## Suppression Gate Paths
- `SUPPRESS` (<0.20) → `ProcessingPath.HALT`
- `LIGHT` (0.20–0.45) → `ProcessingPath.SECTION_EMBEDDINGS_ONLY`
- `MEDIUM`/`DEEP` (≥0.45) → `ProcessingPath.FULL_PIPELINE`
- Uses `final_routing_tier` over `routing_tier` when set (novelty correction)

## Consumer & Dispatcher (R22 standalone processes)
- **ArticleProcessingConsumer**: `BaseKafkaConsumer[None]`; at-least-once; `is_duplicate` always False; `dead_letter` writes to nlp_db DLQ
  - Entry point: `infrastructure/messaging/consumers/article_consumer_main.py`
- **NLPPipelineOutboxDispatcher**: custom (not BaseOutboxDispatcher); polls `outbox_events` via `claim_batch`; produces bytes directly; moves to DLQ after 5 attempts
  - Entry point: `infrastructure/messaging/outbox/dispatcher_main.py` (supervised restart with exponential backoff)
- **WatchlistEventConsumer**: updates Valkey SET on watchlist changes
  - Entry point: `infrastructure/messaging/consumers/watchlist_consumer_main.py`
- **Admin token**: `X-Admin-Token` validated via `hmac.compare_digest` against `NLP_PIPELINE_ADMIN_TOKEN`

## Critical Rules
- **NEVER** run Alembic against `intelligence_db` from this service (`ALEMBIC_ENABLED=false`)
- Use `libs/ml-clients` for ALL ML calls (EmbeddingClient, NERClient, ExtractionClient)
- Backpressure via `asyncio.Semaphore` on Ollama queue depth
- Zero NER mentions NEVER suppress a document — explicit invariant, tested
- NMS IoU threshold is **strictly > 0.5** (not >=)

## Block 7-10 Key Behaviours (Wave C-3)
- **Block 7**: `chunk_section()` uses `_SENTENCE_END_RE` to split sentences; never splits mid-sentence. Section embeddings for ALL tiers; chunk embeddings only on FULL_PIPELINE.
- **Block 8**: MinHash key = `s5:minhash:article:<doc_id>` in Valkey; ONLY DEEP tier is downgraded (MEDIUM never touched by novelty gate). novelty_score returned for Block 5 feature vector.
- **Block 9 (batch-optimised)**: 3 DB round-trips regardless of mention count (1 per batch stage):
  - Stage 1: `alias_repo.batch_exact_match(all_texts)` → `dict[str, UUID]` (IN clause)
  - Stage 2: `alias_repo.batch_ticker_isin_match(tickers, isins)` → `dict[str, UUID]` (1-2 queries)
  - Stage 3: `alias_repo.batch_fuzzy_trigram(stage3_texts, threshold, top_k)` → `dict[str, list[tuple[UUID,float]]]` (UNNEST lateral join)
  - Stage 4 (ANN HNSW): per-mention only for unresolved after stages 1-3; requires `clear_margin > 0.10`
  - UNRESOLVED mentions are NEVER deleted; stages 1-2 emit miss audit entries for complete trail
- **Block 10**: `ExtractionResult = dict[str, Any]`; evidence_date = `coalesce(published_at, extracted_at)` — never `now()`. Claims written to `nlp_db.outbox_events` via `ClaimsRepository` (never direct to `intelligence_db`).
- **Backpressure**: `BackpressureController(max_depth, resume_depth)` with `asyncio.Semaphore`. Context manager cleans up on exception. `is_paused` property used by Kafka consumer loop.

## Pitfalls
- HNSW indexes must include partial predicate `WHERE (expires_at IS NULL OR expires_at > now())` with embedding_status = 'ready' check
- Chunk and section embeddings use SEPARATE HNSW indexes
- Routing thresholds: deep >= 0.70, medium >= 0.45, light >= 0.20
- ORM: `MentionResolutionModel.metadata` column Python attr renamed to `resolution_metadata` (column name stays `metadata`) to avoid collision with SQLAlchemy `DeclarativeBase.metadata`
- Watchlist Valkey ops are best-effort — always return 0.0 on unavailability, never raise
- `BackpressureController.resume_depth` MUST be < `max_depth` — constructor raises `ValueError` otherwise
- `_split_sentences` uses `re.compile(r"(?<=[.!?])\s+")` — text with no punctuation becomes 1 sentence

## Pitfalls (Wave C-4 additions)
- `FailureInfo[None]` has NO `value`/`key`/`headers` fields — only `event_id`, `topic`, `partition`, `offset`, `attempt`, `last_error`, `record`
- `WatchlistCache(client=...)` expects raw `redis.asyncio.Redis`, NOT `ValkeyClient` — pass `valkey._redis`
- `GLiNERLocalAdapter(model_path=..., semaphore=...)` — no `model_id` kwarg
- `GLiNERHTTPAdapter(base_url=..., semaphore=...)` — HTTP adapter for containerised GLiNER server
- `OllamaEmbeddingAdapter/ExtractionAdapter(base_url, model_id, semaphore)` — all 3 required
- `_write_section_embeddings` / `_write_chunk_embeddings` now require `model_id` parameter
- Dispatcher uses custom `claim_batch` (not lease-based `BaseOutboxDispatcher`) — incompatible schemas
- `process_message_from_failure` is a no-op for `FailureInfo[None]` — original payload not recoverable
- **D-004 Eventual consistency gap**: Block 9 opens an `intel_sf()` session inside the outer `nlp_sf()` session. The intel_session commits before the nlp_session commit (~line 279 vs ~line 341). If the nlp commit fails, intel_db writes from entity resolution (Block 9) are already persisted and cannot be rolled back. Re-delivery will retry, potentially creating duplicate audit records. This is accepted eventual consistency. The `nlp_commit_failed_intel_writes_may_be_orphaned` warning log is emitted when the nlp commit fails.
- **GLiNER batching**: Block 4 (`run_ner_block`) calls `ner_client.batch_extract_entities(batch_inputs)` once per processing chunk — all sections in a batch share a single model forward pass. NMS applied per-section after receiving the batch result.
- **UNNEST pg_trgm batch**: `batch_fuzzy_trigram` uses `UNNEST(:terms::text[]) AS q(search_term)` lateral join — PostgreSQL casts the Python list to `text[]` automatically. Asyncpg rejects `::type` cast syntax in `text()` params — use `bindparam('terms', type_=sa.ARRAY(sa.Text()))` if `text()` casts fail (BP-076).
