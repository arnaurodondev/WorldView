## Routing Tier Thresholds
- `_assign_tier` module defaults: `TIER_DEEP=0.75 / TIER_MEDIUM=0.45 / TIER_LIGHT=0.20` (recalibrated for v2 ceiling).
- BUT the article consumer passes `settings.routing_tier_*`, so EFFECTIVE bounds are config-driven: config defaults `0.70 / 0.35 / 0.20`; **`configs/docker.env` overrides to `DEEP=0.45 / MEDIUM=0.35 / LIGHT=0.20`**.
- below LIGHT ŌåÆ `SUPPRESS` ŌåÆ `ProcessingPath.HALT`
- **Deep-extraction VALUE gate (2026-07-17 backlog-drain lever, `apply_deep_extraction_value_gate`)** ŌĆö MEDIUM & DEEP both map to `FULL_PIPELINE`, so ~69% of intake ran the full DeepInfra deep-extraction chain and the ~192k backlog never drained (`docs/audits/2026-07-17-article-backlog-lever.md`). The gate downgrades genuinely low-value docs (composite score `< NLP_PIPELINE_DEEP_EXTRACTION_SCORE_FLOOR`, default **0.45**) from `FULL_PIPELINE` ŌåÆ `SECTION_EMBEDDINGS_ONLY`: **chunk embeddings still written (searchable), only entity resolution + LLM extraction skipped**. Conservative ŌĆö never touches an already-cheap path, a filing (`_AUTHORITATIVE_FILING_SOURCES`), or a doc scoring Ōēź floor. Applied at initial-path AND re-applied in `ml_phase` post-novelty (authoritative). Keys on the STATIC composite score, which fully controls routing when the learned router is `off` (prod default). Toggle `NLP_PIPELINE_DEEP_EXTRACTION_VALUE_GATE_ENABLED` (default on). **Default floor 0.45 == the live-prod DEEP boundary (`NLP_PIPELINE_ROUTING_TIER_DEEP=0.45`), so NO DEEP-tier doc is gated ŌĆö only the `[0.35,0.45)` MEDIUM band (~26-28% of intake). Keep floor Ōēż the DEEP threshold** (a higher floor like 0.50 gates DEEP-tier docs in `[0.45,floor)` ŌåÆ wrongly skips real earnings/M&A/analyst/contract news). At the default, ~59% of intake skips the LLM chain (~41% deep-extraction volume/cost cut), flipping drain > intake.
  - **Follow-up (not built):** composite score is an entity-DENSITY proxy, not a VALUE proxy ŌĆö a focused single-company earnings/M&A/contract story scores low despite high value, so even 0.45 gates some substantive MEDIUM news. Durable fix = a VALUE signal (event-type detection: earnings/M&A/analyst/contract) to force-extract substantive MEDIUM docs regardless of density.

## Key Pitfalls
- **Never hardcode `estimated_cost_usd=0.0` for a paid model (PLAN-0117 W3, RC-1)** ŌĆö resolve cost via `ml_clients.pricing.resolve_cost(model_id, provider=ŌĆ”, tokens_in=ŌĆ”, tokens_out=ŌĆ”, provider_estimated_cost=ŌĆ”)` at EVERY LLM usage-log call site. It returns `(Decimal, cost_source)`: DeepInfra `usage.estimated_cost` (capture from the response body) wins ŌåÆ `"provider"`; Ollama/GLiNER ŌåÆ `"local"` ($0 legitimate); else the price matrix ŌåÆ `"pricematrix"`. Pass `cost_source` (and `user_id` ŌĆö NULL for background pipelines) to `SessionScopedNlpUsageLogger.log`. Wired at: `relevance_cascade.py`, `blocks/deep_extraction.py`, `workers/article_relevance_scoring_worker.py`, `workers/unresolved_resolution_worker.py` (4 sites). `pricing.py` is the SINGLE calculator; do not add local price math.
- **Silent-zero cost guard (PLAN-0117 W5, FR-7, BP-715)** ŌĆö the `NlpUsageLogRepository.log` INSERT now calls `observability.metrics.record_silent_zero_cost("nlp-pipeline", ŌĆ”)`, which trips `llm_usage_silent_zero_cost_total{service,model_id}` when a row has tokens>0, `$0` cost, and `cost_source NOT IN (local, aggregate)`. The article-consumer boot logs a WARNING for any configured model not in `ml_clients.pricing` / `LOCAL_FREE_MODELS` (`warn_unpriceable_models`). If you add a configured model id, also add it to `ml_clients.model_registry.PLATFORM_MODEL_REGISTRY` (CI test `test_all_configured_models_priceable` enforces it).
- **Public content is stamped with the PUBLIC_TENANT_ID sentinel, NOT SQL NULL** (BP-575) ŌĆö search tenant predicates MUST match `tenant_id IS NULL OR tenant_id = '00000000-ŌĆ”'::uuid` (see `_tenant_predicate` in `chunk_search.py`, and `news_query.py` R35). A bare `IS NULL` predicate hid ~86% of the public corpus (BUG-3: `source_types=['sec_edgar']` returned 0; no-filter ANN returned ~4/66k). The nil-UUID sentinel is `common.ids.PUBLIC_TENANT_ID`.
- **pgvector HNSW post-filters the candidate set** ŌĆö WHERE filters (source_type/tenant/entity/date) apply AFTER the index returns ~`hnsw.ef_search` (default 40) candidates, so any selective filter starves to ~0. `ChunkANNRepository` raises it per-query via `SELECT set_config('hnsw.ef_search', ŌĆ”, true)` from `chunk_ann_ef_search` (default 200).
- **ef_search alone can't surface a RARE source under a HARD filter (BUG-3 finish)** ŌĆö for a ~2%-density bucket (sec_edgar = ~1.4k of ~66.8k) the HNSW top-200 for a news/entity text probe contains **0** sec_edgar rows (proven live), so `source_types=['sec_edgar']` returned 0 even with ef_search=200. Fix: when a SELECTIVE filter (`source_types`/`entity_ids`/`entity_types`) is present, `ChunkANNRepository` FILTERS FIRST and runs an EXACT `ORDER BY embedding <=> query` over the filtered subset inside a `WITH filtered AS MATERIALIZED (ŌĆ”)` CTE fence (the CTE forces exact-sort, so the HNSW index is structurally unusable ŌĆö verified via EXPLAIN: no `idx_chunk_emb_hnsw`, top-N heapsort over the filtered CTE). Applies to both chunk and section ANN. Knobs: `chunk_ann_exact_when_filtered` (default True), `chunk_ann_exact_max_rows` (default 100k ŌĆö filtered sets above this fall back to HNSW, which is fine because a large filtered set means the filter isn't selective). The UNFILTERED path (98% case) keeps the fast HNSW ANN unchanged. pgvector is **0.7.2** here ŌåÆ `hnsw.iterative_scan` is NOT available (0.8+ only); do not use it. NB latency of the exact path is dominated by the unindexed `dsm.source_type` seq scan (~600ms, same cost the old path paid), not the distance sort.
- **The exact-KNN path is O(bucket-size) ŌĆö it became a TIMEOUT on the grown sec_edgar corpus (migration 0024 supersedes it for indexed sources)** ŌĆö the "few thousand rows = sub-ms" assumption broke when the R1 filings backfill grew `sec_edgar` to ~30.5k of ~110k ready embeddings. The MATERIALIZED-CTE exact sort then took **~19s** (EXPLAIN ANALYZE, `source_types=['sec_edgar']`, top_k=25) ŌĆö past rag-chat's `RAG_CHAT_UPSTREAM_TIMEOUT_SECONDS=10` ŌåÆ `get_filings` got 0 items ~100%, and NEWS intermittently emptied (same endpoint). Fix: **migration 0024 denormalizes `source_type` onto `chunk_embeddings`/`section_embeddings`** (backfilled + kept current by `BEFORE INSERT` triggers `trg_chunk_emb_source_type` / `trg_section_emb_source_type`) and adds **PARTIAL HNSW indexes** `idx_chunk_emb_hnsw_sec_edgar` (predicate `embedding_status='ready' AND source_type='sec_edgar'`) + `idx_section_emb_hnsw_sec_edgar`. For a SINGLE allow-listed `source_type` with no entity filter, `ChunkANNRepository` now emits a validated **LITERAL** `ce.source_type='sec_edgar'` predicate (Postgres `predicate_implied_by` only matches a partial index against a literal / single-element array, NEVER a bind param ŌĆö hence the literal, gated by `_SAFE_SOURCE_TYPE` regex + the `chunk_ann_indexed_source_types` allow-list to stay injection-safe). Result: **~20ms, Index Scan using idx_chunk_emb_hnsw_sec_edgar, recall@25 = 24-25/25** vs exact. Every OTHER filter shape (multi-source, entity filters, un-indexed source) keeps the exact/HNSW path, so recall is preserved. Knobs: `chunk_ann_indexed_source_types` (CSV, default `sec_edgar`), `chunk_ann_accel_ef_search` (default 400). To accelerate another bucket: add a partial index in a new migration AND the value to `chunk_ann_indexed_source_types` (both, or it silently falls back to exact).
- **FTS planner mis-costs a Seq Scan** for broad `tsv_english @@ tsquery` (evaluated `@@` on every row: ~55s vs ~3s GIN bitmap). `document_search.py` pins the GIN index via `set_config('enable_seqscan','off', true)` (config `fts_force_index_scan`, default True). Also: `ts_headline` is generated ONLY for the paginated page rows (in the SELECT leg), never for every matched chunk (was the ~30s ŌåÆ gateway-503 cost). Title/url/published_at come straight from `document_source_metadata` (dsm), not the S5 batch call (which 401s when no internal JWT is forwarded).
- **Startup stale-embedding expiry is PROVIDER-LABEL-AWARE + batched (PRE-1, 2026-07-06)** ŌĆö `_expire_stale_embeddings` (app.py, boot) expires embeddings whose `model_id` is not one of `current_embedding_model_ids(settings)` = **BOTH** `embedding_model_id` (`bge-large`, written by ollama / the `/embed` endpoint) AND `embedding_api_model_id` (`BAAI/bge-large-en-v1.5`, written by DeepInfra) ŌĆö the SAME physical bge-large model under two labels. The old query compared against only `embedding_model_id`, so it flagged the entire DeepInfra corpus (~13.2k unexpired chunk rows of ~110k) as stale; setting `expires_at` on each is a non-HOT update that forces an HNSW graph re-insert (~0.5s/row, measured), so the single unbounded UPDATE ran ~2h and tripped the 10-min `statement_timeout` on **every boot** (caught ŌåÆ `expire_stale_embeddings_failed`, never completed). Fix: `NOT IN (both labels)` ŌåÆ 0-row no-op on the live corpus (verified); drain in bounded `LIMIT :batch` UPDATEs each under its own `SET LOCAL statement_timeout`; run OFF the boot path as a cancel-on-shutdown background task. Knobs: `embedding_expiry_batch_size` (200), `embedding_expiry_statement_timeout_ms` (300000), `embedding_expiry_max_batches_per_run` (100000). NOTE: this expiry mechanism assumes a GENUINE model change writes a label in NEITHER config field; if you switch embedding models, update BOTH config values.
- **HNSW indexes must use partial predicate**: `WHERE (expires_at IS NULL OR expires_at > now())`; use `op.execute()` in Alembic (not native HNSW support)
- **Chunk and section embeddings use SEPARATE HNSW indexes** ŌĆö mixing granularities pollutes ANN recall
- **`routing_decisions.composite_score` is the routing score** ŌĆö do NOT duplicate into `document_source_metadata`; access via JOIN on `doc_id`
- **Learned router modes off|shadow|live (PLAN-0111 C-6/C-7/C-8)** ŌĆö `NLP_PIPELINE_LEARNED_ROUTER_MODE`. `LearnedRouter` (`application/blocks/learned_routing.py`) proposes a tier per article (logged `learned_router_shadow` + counted + persisted to `learned_*` columns). In `shadow` it NEVER changes routing. In `live` (C-8) the post-cascade learned tier CONTROLS processing: `_apply_live_learned_tier` writes it onto `final_routing_tier` (which `apply_suppression_gate` + downstream gates read); the STATIC tier stays in `routing_tier` for comparison. **Three live invariants**: (1) title-less docs (blind classifier ŌĆö reads title+lede) fall back to the static tier (`learned_router_titleless_fallback`); (2) static SUPPRESS is never resurrected; (3) regulatory/`tenant_upload` sources still forced ŌēźMEDIUM. **C-7 cascade** (`NLP_PIPELINE_LEARNED_ROUTER_CASCADE`): in-band `P(yield)Ōłł[0.48,0.68]` escalates to the REUSED Llama-8B relevance scorer (`relevance_scoring_api_model_id` + `ARTICLE_RELEVANCE_SCORER` prompt ŌĆö `relevance_cascade.py`); relevanceŌēź0.5ŌåÆMEDIUM else LIGHT; out-of-band skips the LLM. Model artifact + meta in `application/blocks/models/`; LightGBM needs `libgomp1` (Dockerfile runtime).
- **`ArticleRelevanceScoringWorker` must follow R24 3-phase**: read batch ŌåÆ close session ŌåÆ Ollama calls ŌåÆ open new session ŌåÆ batch UPDATE; never hold DB session open during Ollama calls
- **Worker routing tier filter uses `COALESCE(rd.final_routing_tier, rd.routing_tier)`** ŌĆö novelty-downgraded articles have `final_routing_tier='LIGHT'` but `routing_tier='DEEP'`
- **`day_t2/day_t5` are cumulative from `close_t0`** ŌĆö fetch day_t0 bar first; use its close as baseline
- **`DisplayRelevanceScore.value` uses `is not None` not `or 0.0`** ŌĆö `mi=0.0` (zero price movement) and `mi=None` (not yet labelled) have different semantics
- **`NLP_PIPELINE_SERVICE_ACCOUNT_TOKEN` required in production** (BP-303) ŌĆö without it, PriceImpactLabellingWorker falls back to `dev-login` which is blocked in production
