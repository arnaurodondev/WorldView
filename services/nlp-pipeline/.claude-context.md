## Routing Tier Thresholds
- `_assign_tier` module defaults: `TIER_DEEP=0.75 / TIER_MEDIUM=0.45 / TIER_LIGHT=0.20` (recalibrated for v2 ceiling).
- BUT the article consumer passes `settings.routing_tier_*`, so EFFECTIVE bounds are config-driven: config defaults `0.70 / 0.35 / 0.20`; **`configs/docker.env` overrides to `DEEP=0.45 / MEDIUM=0.35 / LIGHT=0.20`**.
- below LIGHT ‚Üí `SUPPRESS` ‚Üí `ProcessingPath.HALT`
- **Deep-extraction VALUE gate (2026-07-17 backlog-drain lever, `apply_deep_extraction_value_gate`)** ‚Äî MEDIUM & DEEP both map to `FULL_PIPELINE`, so ~69% of intake ran the full DeepInfra deep-extraction chain and the ~192k backlog never drained (`docs/audits/2026-07-17-article-backlog-lever.md`). The gate downgrades genuinely low-value docs (composite score `< NLP_PIPELINE_DEEP_EXTRACTION_SCORE_FLOOR`, default **0.45**) from `FULL_PIPELINE` ‚Üí `SECTION_EMBEDDINGS_ONLY`: **chunk embeddings still written (searchable), only entity resolution + LLM extraction skipped**. Conservative ‚Äî never touches an already-cheap path, a filing (`_AUTHORITATIVE_FILING_SOURCES`), or a doc scoring ‚â• floor. Applied at initial-path AND re-applied in `ml_phase` post-novelty (authoritative). Keys on the STATIC composite score, which fully controls routing when the learned router is `off` (prod default). Toggle `NLP_PIPELINE_DEEP_EXTRACTION_VALUE_GATE_ENABLED` (default on). **Default floor 0.45 == the live-prod DEEP boundary (`NLP_PIPELINE_ROUTING_TIER_DEEP=0.45`), so NO DEEP-tier doc is gated ‚Äî only the `[0.35,0.45)` MEDIUM band (~26-28% of intake). Keep floor ‚â§ the DEEP threshold** (a higher floor like 0.50 gates DEEP-tier docs in `[0.45,floor)` ‚Üí wrongly skips real earnings/M&A/analyst/contract news). At the default, ~59% of intake skips the LLM chain (~41% deep-extraction volume/cost cut), flipping drain > intake.
  - **Follow-up (not built):** composite score is an entity-DENSITY proxy, not a VALUE proxy ‚Äî a focused single-company earnings/M&A/contract story scores low despite high value, so even 0.45 gates some substantive MEDIUM news. Durable fix = a VALUE signal (event-type detection: earnings/M&A/analyst/contract) to force-extract substantive MEDIUM docs regardless of density.

## Key Pitfalls
- **Never hardcode `estimated_cost_usd=0.0` for a paid model (PLAN-0117 W3, RC-1)** ‚Äî resolve cost via `ml_clients.pricing.resolve_cost(model_id, provider=‚Ä¶, tokens_in=‚Ä¶, tokens_out=‚Ä¶, provider_estimated_cost=‚Ä¶)` at EVERY LLM usage-log call site. It returns `(Decimal, cost_source)`: DeepInfra `usage.estimated_cost` (capture from the response body) wins ‚Üí `"provider"`; Ollama/GLiNER ‚Üí `"local"` ($0 legitimate); else the price matrix ‚Üí `"pricematrix"`. Pass `cost_source` (and `user_id` ‚Äî NULL for background pipelines) to `SessionScopedNlpUsageLogger.log`. Wired at: `relevance_cascade.py`, `blocks/deep_extraction.py`, `workers/article_relevance_scoring_worker.py`, `workers/unresolved_resolution_worker.py` (4 sites). `pricing.py` is the SINGLE calculator; do not add local price math.
- **Silent-zero cost guard (PLAN-0117 W5, FR-7, BP-715)** ‚Äî the `NlpUsageLogRepository.log` INSERT now calls `observability.metrics.record_silent_zero_cost("nlp-pipeline", ‚Ä¶)`, which trips `llm_usage_silent_zero_cost_total{service,model_id}` when a row has tokens>0, `$0` cost, and `cost_source NOT IN (local, aggregate)`. The article-consumer boot logs a WARNING for any configured model not in `ml_clients.pricing` / `LOCAL_FREE_MODELS` (`warn_unpriceable_models`). If you add a configured model id, also add it to `ml_clients.model_registry.PLATFORM_MODEL_REGISTRY` (CI test `test_all_configured_models_priceable` enforces it).
- **Public content is stamped with the PUBLIC_TENANT_ID sentinel, NOT SQL NULL** (BP-575) ‚Äî search tenant predicates MUST match `tenant_id IS NULL OR tenant_id = '00000000-‚Ä¶'::uuid` (see `_tenant_predicate` in `chunk_search.py`, and `news_query.py` R35). A bare `IS NULL` predicate hid ~86% of the public corpus (BUG-3: `source_types=['sec_edgar']` returned 0; no-filter ANN returned ~4/66k). The nil-UUID sentinel is `common.ids.PUBLIC_TENANT_ID`.
- **pgvector HNSW post-filters the candidate set** ‚Äî WHERE filters (source_type/tenant/entity/date) apply AFTER the index returns ~`hnsw.ef_search` (default 40) candidates, so any selective filter starves to ~0. `ChunkANNRepository` raises it per-query via `SELECT set_config('hnsw.ef_search', ‚Ä¶, true)` from `chunk_ann_ef_search` (default 200).
- **ef_search alone can't surface a RARE source under a HARD filter (BUG-3 finish)** ‚Äî for a ~2%-density bucket (sec_edgar = ~1.4k of ~66.8k) the HNSW top-200 for a news/entity text probe contains **0** sec_edgar rows (proven live), so `source_types=['sec_edgar']` returned 0 even with ef_search=200. Fix: when a SELECTIVE filter (`source_types`/`entity_ids`/`entity_types`) is present, `ChunkANNRepository` FILTERS FIRST and runs an EXACT `ORDER BY embedding <=> query` over the filtered subset inside a `WITH filtered AS MATERIALIZED (‚Ä¶)` CTE fence (the CTE forces exact-sort, so the HNSW index is structurally unusable ‚Äî verified via EXPLAIN: no `idx_chunk_emb_hnsw`, top-N heapsort over the filtered CTE). Applies to both chunk and section ANN. Knobs: `chunk_ann_exact_when_filtered` (default True), `chunk_ann_exact_max_rows` (default 100k ‚Äî filtered sets above this fall back to HNSW, which is fine because a large filtered set means the filter isn't selective). The UNFILTERED path (98% case) keeps the fast HNSW ANN unchanged. pgvector is **0.7.2** here ‚Üí `hnsw.iterative_scan` is NOT available (0.8+ only); do not use it. NB latency of the exact path is dominated by the unindexed `dsm.source_type` seq scan (~600ms, same cost the old path paid), not the distance sort.
- **The exact-KNN path is O(bucket-size) ‚Äî it became a TIMEOUT on the grown sec_edgar corpus (migration 0024 supersedes it for indexed sources)** ‚Äî the "few thousand rows = sub-ms" assumption broke when the R1 filings backfill grew `sec_edgar` to ~30.5k of ~110k ready embeddings. The MATERIALIZED-CTE exact sort then took **~19s** (EXPLAIN ANALYZE, `source_types=['sec_edgar']`, top_k=25) ‚Äî past rag-chat's `RAG_CHAT_UPSTREAM_TIMEOUT_SECONDS=10` ‚Üí `get_filings` got 0 items ~100%, and NEWS intermittently emptied (same endpoint). Fix: **migration 0024 denormalizes `source_type` onto `chunk_embeddings`/`section_embeddings`** (backfilled + kept current by `BEFORE INSERT` triggers `trg_chunk_emb_source_type` / `trg_section_emb_source_type`) and adds **PARTIAL HNSW indexes** `idx_chunk_emb_hnsw_sec_edgar` (predicate `embedding_status='ready' AND source_type='sec_edgar'`) + `idx_section_emb_hnsw_sec_edgar`. For a SINGLE allow-listed `source_type` with no entity filter, `ChunkANNRepository` now emits a validated **LITERAL** `ce.source_type='sec_edgar'` predicate (Postgres `predicate_implied_by` only matches a partial index against a literal / single-element array, NEVER a bind param ‚Äî hence the literal, gated by `_SAFE_SOURCE_TYPE` regex + the `chunk_ann_indexed_source_types` allow-list to stay injection-safe). Result: **~20ms, Index Scan using idx_chunk_emb_hnsw_sec_edgar, recall@25 = 24-25/25** vs exact. Every OTHER filter shape (multi-source, entity filters, un-indexed source) keeps the exact/HNSW path, so recall is preserved. Knobs: `chunk_ann_indexed_source_types` (CSV, default `sec_edgar`), `chunk_ann_accel_ef_search` (default 400). To accelerate another bucket: add a partial index in a new migration AND the value to `chunk_ann_indexed_source_types` (both, or it silently falls back to exact).
- **FTS planner mis-costs a Seq Scan** for broad `tsv_english @@ tsquery` (evaluated `@@` on every row: ~55s vs ~3s GIN bitmap). `document_search.py` pins the GIN index via `set_config('enable_seqscan','off', true)` (config `fts_force_index_scan`, default True). Also: `ts_headline` is generated ONLY for the paginated page rows (in the SELECT leg), never for every matched chunk (was the ~30s ‚Üí gateway-503 cost). Title/url/published_at come straight from `document_source_metadata` (dsm), not the S5 batch call (which 401s when no internal JWT is forwarded).
- **Startup stale-embedding expiry is PROVIDER-LABEL-AWARE + batched (PRE-1, 2026-07-06)** ‚Äî `_expire_stale_embeddings` (app.py, boot) expires embeddings whose `model_id` is not one of `current_embedding_model_ids(settings)` = **BOTH** `embedding_model_id` (`bge-large`, written by ollama / the `/embed` endpoint) AND `embedding_api_model_id` (`BAAI/bge-large-en-v1.5`, written by DeepInfra) ‚Äî the SAME physical bge-large model under two labels. The old query compared against only `embedding_model_id`, so it flagged the entire DeepInfra corpus (~13.2k unexpired chunk rows of ~110k) as stale; setting `expires_at` on each is a non-HOT update that forces an HNSW graph re-insert (~0.5s/row, measured), so the single unbounded UPDATE ran ~2h and tripped the 10-min `statement_timeout` on **every boot** (caught ‚Üí `expire_stale_embeddings_failed`, never completed). Fix: `NOT IN (both labels)` ‚Üí 0-row no-op on the live corpus (verified); drain in bounded `LIMIT :batch` UPDATEs each under its own `SET LOCAL statement_timeout`; run OFF the boot path as a cancel-on-shutdown background task. Knobs: `embedding_expiry_batch_size` (200), `embedding_expiry_statement_timeout_ms` (300000), `embedding_expiry_max_batches_per_run` (100000). NOTE: this expiry mechanism assumes a GENUINE model change writes a label in NEITHER config field; if you switch embedding models, update BOTH config values.
- **Two INDEPENDENT DB timeouts ‚Äî server `statement_timeout` AND client `command_timeout` (dead-connection hang, 2026-07-21)** ‚Äî `statement_timeout` (server_settings, `statement_timeout_ms`=60000) is enforced by Postgres and does NOTHING once the server is dead. On 2026-07-21 Postgres OOM-crashed, a pooled asyncpg socket went half-open (dead but never RST), and a processing coroutine issued a query that HUNG for **2.4 h** (asyncpg has no client deadline by default) ‚Äî wedging the whole article pipeline until a manual restart. The sibling `_settle_message` retry-ceiling only guards the offset-commit path, NOT the main extraction‚Üíresolution‚ÜíDB path, so nothing bounded the wait. Fix: `build_connect_args` now also sets asyncpg's **top-level `command_timeout`** (client-side, `command_timeout_s`=600.0 / env `NLP_PIPELINE_COMMAND_TIMEOUT_S`) on BOTH nlp_db and intelligence_db engines, so any op on a dead connection RAISES `asyncio.TimeoutError` (a plain `Exception`, NOT a `FatalError`) ‚Üí the consumer's generic `except Exception` treats it as transient ‚Üí in-place retry / DLQ (at-least-once preserved, no silent drop). 600 s is deliberately ABOVE both legitimate ceilings that run on the SAME engine (60 s `statement_timeout`, 300 s embedding-expiry `SET LOCAL` batch) so it never preempts live work, and BELOW Kafka `max.poll.interval.ms` (30 min) so the article redelivers before the consumer is evicted. `command_timeout` MUST be a top-level connect kwarg, not a `server_settings` value. Set to 0 to disable (the pre-fix unbounded behaviour ‚Äî not recommended).
- **HNSW indexes must use partial predicate**: `WHERE (expires_at IS NULL OR expires_at > now())`; use `op.execute()` in Alembic (not native HNSW support)
- **Same literal-vs-bound-param partial-index footgun recurred a 2nd time (BP-730)** ‚Äî `entity_profile_embedding.py::ann_search` (on `entity_embedding_state`, shared w/ knowledge-graph) hit the identical `chunk_search.py`/BP-717 defect: a bound `:view_type` param can't satisfy `predicate_implied_by` on a partial HNSW index, so the planner Seq-Scans ‚Üí OOM. This check is still NOT in `.claude/review/heuristics/HIGH_RISK_PATTERNS.md`/`REVIEW_CHECKLIST.md` ‚Äî any new query against a partial-indexed column needs a literal (allow-listed) predicate or an EXPLAIN-verified Index Scan test, don't assume review catches it.
- **Chunk and section embeddings use SEPARATE HNSW indexes** ‚Äî mixing granularities pollutes ANN recall
- **`routing_decisions.composite_score` is the routing score** ‚Äî do NOT duplicate into `document_source_metadata`; access via JOIN on `doc_id`
