## Mission
Relation canonicalization (Block 11), graph materialization + evidence staging (Block 12 hot path), 15 async APScheduler workers (Block 13 — confidence recomputation, contradiction detection, summary generation, entity descriptions, embedding refresh, partition management, narrative generation, path pre-computation), Apache AGE shadow sync (Block 14), outbox dispatcher.

## Domain Entities
- `Relation` (frozen DC) — maps to hash-partitioned `relations` table (HASH ×8 on `subject_entity_id`)
- `RelationEvidence` (frozen DC) — `is_backfill` flag for historical loads
- `RelationSummary` — LLM-generated; `evidence_hash` for change-detection skip
- `ContradictionLink` — row in `relation_contradiction_links`
- `Contradiction` — event aggregate: subject-based, opposite+non-neutral polarity, 90-day window
- `ConfidenceComponents` — 4-step bounded formula result; call `.validate()` after construction
- `SemanticMode` StrEnum: `RELATION_STATE | TEMPORAL_CLAIM`
- `DecayClass` StrEnum: `STANDARD | TEMPORAL`
- `RelationType` — 16 code-level values (8 original + 3 PRD-0018 + 5 PLAN-0089 Lever-4 migration 0041: APPOINTED_AS, DIVESTED_FROM, DOWNGRADED_BY, FILED_LAWSUIT_AGAINST, REPORTED_REVENUE_OF); 32+ total in `relation_type_registry` DB seeds
- **QW-1 startup check** (`app.py` lifespan): warns `registry_embeddings_missing` if `relation_type_registry` rows have `embedding IS NULL`. Non-fatal but means new relation types won't participate in Step 2 ANN soft-mapping until embeddings are backfilled.
- `TemporalEvent` (frozen DC) — `lifecycle_phase` computed at access time (PENDING_ACTIVE/ACTIVE/RESIDUAL/EXPIRED)
- `EntityEventExposure` (frozen DC) — entity ↔ temporal event link
- `SimilarEntityResult` (frozen DC, slots=True) — ANN similarity result; gains `surprise_score: float | None`
- `EntityNarrativeVersion` — version-controlled LLM narrative with `health_score`

## Confidence Formula
4-step bounded: support (temporal-weighted avg of source_weights) → corroboration gain (distinct qualifying sources, cap 0.20) → contradiction penalty (top-3 decayed links, cap 0.60) → clamp [0,1].
- RELATION_STATE: uses `decay_alpha` from `decay_class_config` row
- TEMPORAL_CLAIM: always uses `0.02310` (30-day half-life)

## Provisional Enrichment Backoff (DEF-033)
```
next_retry_at = now() + min(base × 2^retry_count, max) minutes
```
Default: base=2, max=1440 (24h cap). `claim_batch` SELECT filters `next_retry_at IS NULL OR next_retry_at <= now()`.

## Key Pitfalls
- **`partition_key` STORED** — never in INSERT; DB raises error
- **`relation_evidence_raw` has NO `relation_id`** — JOIN on the triple
- **`relation_evidence_raw` has NO `canonicalized_evidence_text`** — only `relation_evidence` (processed table) has it
- **AGE Cypher injection** — entity IDs as `$params`; edge labels only exception (validate against `_VALID_EDGE_LABELS` whitelist)
- **AGE `LOAD 'age'` per connection** — handled by AGE adapter; do NOT use read-only sessions for Cypher
- **AGE O(n³) at depth=3 on hub entities** — 5s timeout enforced; use pre-computed `path_insights` for scale
- **`entity.dirtied.v1` post-commit ordering** — accumulate `entity_ids_to_dirty`, produce AFTER commit
- **`ensure_rows_exist()` entity type check** — `financial_instrument` → 3 rows; all others → 2 rows (`definition` + `narrative` only)
- **Ollama model dimension** — MUST use `bge-large:latest` (1024-dim); `nomic-embed-text` (768-dim) raises FatalError
- **GLOBAL temporal events** — link to sector/industry entities ONLY; never per-company rows
- **EODHD API fields that do NOT exist**: `General.Officers`, `Holders.Institutions`, `Financials.Revenue_Segment` — use Insider Transactions API instead
- **`create_or_get()` partial index BP-459** — unique index on `lower(canonical_name)` has `WHERE entity_type != 'financial_instrument'`; phantom duplicates possible for provisional entities with name variations
- **`EmbeddingClientProtocol` is duck-typed locally** — no ml-clients runtime dep in knowledge-graph; Python version boundary
- **`SummaryWorker` ARCH-003 3-phase** — Phase 1+2 on read replica, Phase 3 (LLM) holds no session, Phase 4 opens fresh write session per row
- **ETF sector backfill (PLAN-0103 W8 / BP-629)**: ETF canonical rows (`QQQ`, `XLE`, `IBIT`, `XLK`, …) have `metadata->>'sector' IS NULL` by default — the equities fundamentals consumer never runs on fund payloads. Run `scripts/backfill_etf_sectors.py --apply` after ingesting new ETF holdings; the `/internal/v1/entities/sectors` endpoint also synthesises a generic `"Equity ETF"` bucket when no row exists AND the entity is recognisably an ETF (asset_class metadata OR known ticker prefix `XL*`/`ARK*`) — never let an ETF holding fall out of the rag-chat risk aggregator.
- **`summary_worker_force_regen_batch_size=0` default** — only set to positive value for one tick after prompt template upgrades
- **`_NOISE_BLOCKLIST` location** — `infrastructure/workers/provisional_enrichment.py` module-level frozenset
- **`provisional_entity_queue.status` has 5 values**: `pending/processing/resolved/failed/noise` (CHECK constraint migration 0020)
- **`_DETERMINISTIC_CREATED_AT_FALLBACK` in `graph_write.py`** — claims use a fixed `created_at = 2024-01-01` as a sentinel to satisfy the `(claim_id, created_at)` PRIMARY KEY on the RANGE-partitioned `claims` table. This means all claims land in `claims_2024_01`. **Data loss risk**: if the retention script drops partitions older than 24 months, `claims_2024_01` will be eligible for DROP in May 2026. Also, the 90-day contradiction detection window (`created_at >= now() - 90 days`) silently excludes all sentinel-dated claims. Fix: thread `published_at` from the enriched article event through to `_insert_claim()` and use it as `created_at`.

## PLAN-0093 Pitfalls (added 2026-05-24)
- **PLAN-0093 B-1 AGE label bootstrap** — every relation type written to AGE must exist as a `label` first or `CREATE (a)-[:NEW_LABEL]->(b)` silently no-ops (BP-544). The bootstrap list lives in `infrastructure/age/age_bootstrap.py` and is replayed on every container start via the lifespan hook; missing labels surface as 0-edge graphs with no error. When adding a new relation type, append to the bootstrap list AND ship a migration that runs `SELECT create_elabel('worldview_kg', '<TYPE>')` so existing volumes catch up.
- **PLAN-0093 B-2 sentinel UUID stability** — migration 0044 seeds 5 sentinel entities (`unknown_entity`, `unresolved_entity`, plus three reserved slots) with hard-coded UUIDv7 IDs. Migration 0045 adds a `CHECK` constraint on `canonical_entities` that pins those 5 IDs. FIX-4 added the sync test `tests/integration/test_sentinel_uuid_sync.py` that fails fast if the migration constants drift from `domain/constants/sentinels.py`. Never change a sentinel UUID — extend with a NEW slot.
- **PLAN-0093 B-2 DEFERRABLE FK on outbox** — `outbox_events.aggregate_id → canonical_entities.entity_id` is declared `DEFERRABLE INITIALLY DEFERRED` so the same transaction can `INSERT canonical_entities` and `INSERT outbox_events` referencing it. Never change to `IMMEDIATE`; the dispatcher relies on this for batch writes from `ProvisionalEnrichmentWorker`.
- **PLAN-0093 B-3 evidence `claim_id`/`chunk_id` NOT NULL** — `relation_evidence` and `relation_evidence_raw` enforce NOT NULL on both at the application layer (the partitioned `claims` and `chunks` tables prevent DB-level FKs). Any writer that omits either column drops the row at the next bulk-load; never default to empty string or `'unknown'`. Validation lives in `application/writers/evidence_writer.py:_validate_required_keys`.
- **PLAN-0093 B-4 `entity_mentions.tenant_id` NOT NULL** — migration 0046 made `entity_mentions.tenant_id` NOT NULL with no NULL fallback. The previous `COALESCE(tenant_id, '00000000-0000-0000-0000-000000000000')` pattern was a cross-tenant leak vector (S6 mentions written under the system tenant became visible to every tenant's RAG retrieval). Use the producing tenant's id verbatim; reject rows that arrive without one.
- **PLAN-0093 D-1 atomic `claim_for_enrichment` (BP-548)** — the SELECT … FOR UPDATE SKIP LOCKED + UPDATE pair must run inside the same transaction; the in-memory counter increment must always be paired with the UPDATE write. The pre-BP-548 pattern incremented `enrichment_attempts` in Python memory and skipped the UPDATE when the row was already locked elsewhere, producing silent attempt loss and infinite retry loops. The fixed pattern uses `RETURNING enrichment_attempts` so the worker uses the DB's authoritative value.
- **PLAN-0096 W3 AGE label bootstrap session-cache (BP-574)** — `_bootstrap_age_labels` MUST call `await self._invalidate_session_connection(session)` after its `session.commit()`. Without it, PostgreSQL's plpgsql engine caches the pre-bootstrap AGE label catalog on the physical connection; the subsequent phase MERGE statements then silently no-op (the 0/14,822 TemporalEvent silent-drop bug). Reconciliation script for any existing backlog: `scripts/reconcile_age_temporal_events.py` (idempotent, `--dry-run` supported).
- **PLAN-0093 D-3 SummaryWorker ordering** — the catchup query uses `ORDER BY last_summary_attempt_at NULLS FIRST, dirtied_at ASC` so brand-new entities (NULL attempt) get summarised before stale ones, and the per-row counter resets on a successful write but persists across worker restarts. Rows with `last_summary_attempt_count > 3` are skipped without retry until a manual reset; do not "fix" stuck entities by clearing the counter alone — investigate the LLM failure first, then run `UPDATE … SET last_summary_attempt_count = 0 WHERE entity_id IN (…)` only after the root cause is patched.
