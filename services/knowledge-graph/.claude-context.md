## Confidence Formula
Current default path is `compute_confidence_beta` (`domain/confidence.py`, PLAN-0109 W3 — a Beta/subjective-logic posterior), NOT the older 4-step additive `compute_confidence` (still present, `temporal_claim_alpha` param ignored there too). Both consume `eff_alpha = decay_alpha` unconditionally for the temporal-decay term — the value is resolved BEFORE it reaches `confidence.py`:
- **Alpha resolution (PRD-0120, PLAN-0123)**: `relation_type_registry.py`'s `find_exact`/`find_by_embedding` resolve `decay_alpha` as `COALESCE(rtr.decay_alpha, dcc.decay_alpha)` — a per-type fitted value (written by the offline decay fitter, `application/analytics/decay_fitting/`) overrides the class-level prior (`decay_class_config`) when present; NULL falls back to the class value (today's pre-PRD-0120 behavior). `confidence.py` itself has ZERO diff from this — only the *source* of the alpha it reads changed.
- RELATION_STATE: `d = 1.0` until `valid_to` closes the window (interval-truthed, no exponential decay) — out of scope for per-type fitting (P-3).
- TEMPORAL_CLAIM: `d = exp(-alpha * age_days)` where `alpha` is the resolved value above (per-type fit if one has been written back, else the class value — NOT a fixed constant).

## Decay Fitter (PRD-0120, PLAN-0123) — `application/analytics/decay_fitting/`
Offline, read-mostly job that fits a per-type decay rate for `TEMPORAL_CLAIM` relation types, replacing the hand-asserted `decay_class_config` value where enough data exists (class stays as an empirical-Bayes prior for cold-start/sparse types).
- **Two lifetime definitions**: (a) corroboration-decay via `nhpp_estimator.fit_nhpp` — a non-homogeneous Poisson process fit (intensity ∝ `exp(-alpha*age)`), NOT an inter-arrival-gap MLE (that estimates mention *rate*, not decay *rate* — review finding SS-1; `test_nhpp_rejects_inter_arrival_proxy` is the regression guard); (b) time-to-supersession/contradiction via `supersession_estimator.fit_supersession` — a right-censored exponential MLE reading directly off `relations.first_evidence_at`/`latest_contra_at`/`valid_to` (NOT `relations_history`, which has no `decay_alpha` column).
- **Pipeline**: `lifetime_extraction.py` (read-only, `relation_evidence_raw` pre-gating + `relations`) → the two estimators → `pooling.pool_type_fit` (P-7: prefer supersession once it clears `min_n`, else corroboration; empirical-Bayes shrinkage toward the class prior; min-n gate forces `pooled_prior` method label for sparse types) → `write_back.write_back_fit` (shadow/write-gated; a `pooled_prior` fit is left NULL, never written) → `backfill.backfill_relations_for_type` (explicit `UPDATE relations SET decay_alpha=…, confidence_stale=true WHERE canonical_type=…` — REQUIRED because `relations.decay_alpha` denormalizes only on upsert; a relation with no new evidence would otherwise never pick up a freshly-fitted alpha) → `evaluation.evaluate_fit_vs_prior` (held-out check, honest `insufficient_data` verdict below `min_n`) → `observability.record_decay_fit_metrics`/`record_type_counts` (7 Prometheus gauges, `application/metrics.py`).
- **Scope guard (P-3)**: every extraction function asserts `semantic_mode == 'TEMPORAL_CLAIM'` via `relation_type_registry`, raising `RelationStateNotFittableError` for any `RELATION_STATE` type. 5 pre-existing call sites (`relation.py:150`, `entity_consumer.py:340`, `entity_enrichment_adapter.py:416`, `fundamentals_refresh.py:1070/1111`) hardcode `decay_alpha` for `RELATION_STATE` types — out of scope, verified by a static scope-guard test, not touched.
- Full run is not yet wired into a scheduled worker/CLI entrypoint end-to-end (each stage is independently tested and composable — `run_fitter.run_shadow_fit` currently only exercises the shadow/raw-fit half of the pipeline); wiring pooling→write-back→backfill→metrics into one scheduled job is follow-up work.
