## Confidence Formula
4-step bounded: support (temporal-weighted avg of source_weights) → corroboration gain (distinct sources, cap 0.20) → contradiction penalty (top-3 decayed links, cap 0.60) → clamp to [0,1].
- RELATION_STATE: uses decay_alpha from decay_class_config
- TEMPORAL_CLAIM: always uses 0.02310 (30-day half-life)

## Critical Rules
- **NEVER** run Alembic against `intelligence_db` (`ALEMBIC_ENABLED=false` → RuntimeError guard in session factory)
- `partition_key` in `relations` and `relation_evidence_raw` is `GENERATED ALWAYS AS STORED` — **NEVER include in INSERT**
- `entity.dirtied.v1` is a compacted topic — direct produce (bypasses outbox), key = entity_id
- Advisory lock (`pg_advisory_xact_lock`) on triple hash before relation upsert
- Contradiction requires: same subject, opposite polarity, both non-neutral, within 90-day window

## Pitfalls
- Relations table hash-partitioned ×8 on `subject_entity_id`; upsert needs advisory lock
- `partition_key` STORED column — omit from all INSERTs
- Single evidence piece: decay cancels in support formula (support = source_weight regardless of alpha)
- To test alpha differentiation, use ≥2 evidence pieces at different ages
- `make_interval(days => :param)` for parameterized INTERVAL in PostgreSQL (avoids f-string injection)
- `readyz` returns 503 when DB is not configured (test client); integration tests needed for 200 path
- DLQ admin auth: empty `KNOWLEDGE_GRAPH_ADMIN_TOKEN` disables all DLQ access (not enables it)
- `list_for_entity` / `list_filtered` use static parameterized SQL with `IS NULL OR ...` pattern — no f-strings
