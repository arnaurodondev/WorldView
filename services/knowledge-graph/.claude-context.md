## Confidence Formula
4-step bounded: support (temporal-weighted avg of source_weights) ‚Üí corroboration gain (distinct qualifying sources, cap 0.20) ‚Üí contradiction penalty (top-3 decayed links, cap 0.60) ‚Üí clamp [0,1].
- RELATION_STATE: uses `decay_alpha` from `decay_class_config` row
- TEMPORAL_CLAIM: always uses `0.02310` (30-day half-life)

## Provisional Enrichment Backoff (DEF-033)
```
next_retry_at = now() + min(base √ó 2^retry_count, max) minutes
```
Default: base=2, max=1440 (24h cap). `claim_batch` SELECT filters `next_retry_at IS NULL OR next_retry_at <= now()`.

## Key Pitfalls
- **`partition_key` STORED** ‚Äî never in INSERT; DB raises error
- **`relation_evidence_raw` has NO `relation_id`** ‚Äî JOIN on the triple
- **`relation_evidence_raw` has NO `canonicalized_evidence_text`** ‚Äî only `relation_evidence` (processed table) has it
- **AGE Cypher injection** ‚Äî entity IDs as `$params`; edge labels only exception (validate against `_VALID_EDGE_LABELS` whitelist)
- **AGE `LOAD 'age'` per connection** ‚Äî handled by AGE adapter; do NOT use read-only sessions for Cypher
- **AGE O(n¬≥) at depth=3 on hub entities** ‚Äî 5s timeout enforced; use pre-computed `path_insights` for scale
- **`entity.dirtied.v1` post-commit ordering** ‚Äî accumulate `entity_ids_to_dirty`, produce AFTER commit
- **`ensure_rows_exist()` entity type check** ‚Äî `financial_instrument` ‚Üí 3 rows; all others ‚Üí 2 rows (`definition` + `narrative` only)
- **Ollama model dimension** ‚Äî MUST use `bge-large:latest` (1024-dim); `nomic-embed-text` (768-dim) raises FatalError
- **GLOBAL temporal events** ‚Äî link to sector/industry entities ONLY; never per-company rows
- **EODHD API fields that do NOT exist**: `General.Officers`, `Holders.Institutions`, `Financials.Revenue_Segment` ‚Äî use Insider Transactions API instead
- **`create_or_get()` partial index BP-459** ‚Äî unique index on `lower(canonical_name)` has `WHERE entity_type != 'financial_instrument'`; phantom duplicates possible for provisional entities with name variations
- **`EmbeddingClientProtocol` is duck-typed locally** ‚Äî no ml-clients runtime dep in knowledge-graph; Python version boundary
- **`SummaryWorker` ARCH-003 3-phase** ‚Äî Phase 1+2 on read replica, Phase 3 (LLM) holds no session, Phase 4 opens fresh write session per row
- **`summary_worker_force_regen_batch_size=0` default** ‚Äî only set to positive value for one tick after prompt template upgrades
- **`_NOISE_BLOCKLIST` location** ‚Äî `infrastructure/workers/provisional_enrichment.py` module-level frozenset
- **`provisional_entity_queue.status` has 5 values**: `pending/processing/resolved/failed/noise` (CHECK constraint migration 0020)
- **`_DETERMINISTIC_CREATED_AT_FALLBACK` in `graph_write.py`** ‚Äî claims use a fixed `created_at = 2024-01-01` as a sentinel to satisfy the `(claim_id, created_at)` PRIMARY KEY on the RANGE-partitioned `claims` table. This means all claims land in `claims_2024_01`. **Data loss risk**: if the retention script drops partitions older than 24 months, `claims_2024_01` will be eligible for DROP in May 2026. Also, the 90-day contradiction detection window (`created_at >= now() - 90 days`) silently excludes all sentinel-dated claims. Fix: thread `published_at` from the enriched article event through to `_insert_claim()` and use it as `created_at`.

## PLAN-0093 Pitfalls (added 2026-05-24)
- **PLAN-0093 B-1 AGE label bootstrap** ‚Äî every relation type written to AGE must exist as a `label` first or `CREATE (a)-[:NEW_LABEL]->(b)` silently no-ops (BP-544). The bootstrap list lives in `infrastructure/age/age_bootstrap.py` and is replayed on every container start via the lifespan hook; missing labels surface as 0-edge graphs with no error. When adding a new relation type, append to the bootstrap list AND ship a migration that runs `SELECT create_elabel('worldview_kg', '<TYPE>')` so existing volumes catch up.
- **PLAN-0093 B-2 sentinel UUID stability** ‚Äî migration 0044 seeds 5 sentinel entities (`unknown_entity`, `unresolved_entity`, plus three reserved slots) with hard-coded UUIDv7 IDs. Migration 0045 adds a `CHECK` constraint on `canonical_entities` that pins those 5 IDs. FIX-4 added the sync test `tests/integration/test_sentinel_uuid_sync.py` that fails fast if the migration constants drift from `domain/constants/sentinels.py`. Never change a sentinel UUID ‚Äî extend with a NEW slot.
- **PLAN-0093 B-2 DEFERRABLE FK on outbox** ‚Äî `outbox_events.aggregate_id ‚Üí canonical_entities.entity_id` is declared `DEFERRABLE INITIALLY DEFERRED` so the same transaction can `INSERT canonical_entities` and `INSERT outbox_events` referencing it. Never change to `IMMEDIATE`; the dispatcher relies on this for batch writes from `ProvisionalEnrichmentWorker`.
- **PLAN-0093 B-3 evidence `claim_id`/`chunk_id` NOT NULL** ‚Äî `relation_evidence` and `relation_evidence_raw` enforce NOT NULL on both at the application layer (the partitioned `claims` and `chunks` tables prevent DB-level FKs). Any writer that omits either column drops the row at the next bulk-load; never default to empty string or `'unknown'`. Validation lives in `application/writers/evidence_writer.py:_validate_required_keys`.
- **PLAN-0093 B-4 `entity_mentions.tenant_id` NOT NULL** ‚Äî migration 0046 made `entity_mentions.tenant_id` NOT NULL with no NULL fallback. The previous `COALESCE(tenant_id, '00000000-0000-0000-0000-000000000000')` pattern was a cross-tenant leak vector (S6 mentions written under the system tenant became visible to every tenant's RAG retrieval). Use the producing tenant's id verbatim; reject rows that arrive without one.
- **PLAN-0093 D-1 atomic `claim_for_enrichment` (BP-548)** ‚Äî the SELECT ‚Ä¶ FOR UPDATE SKIP LOCKED + UPDATE pair must run inside the same transaction; the in-memory counter increment must always be paired with the UPDATE write. The pre-BP-548 pattern incremented `enrichment_attempts` in Python memory and skipped the UPDATE when the row was already locked elsewhere, producing silent attempt loss and infinite retry loops. The fixed pattern uses `RETURNING enrichment_attempts` so the worker uses the DB's authoritative value.
- **PLAN-0096 W3 AGE label bootstrap session-cache (BP-574)** ‚Äî `_bootstrap_age_labels` MUST call `await self._invalidate_session_connection(session)` after its `session.commit()`. Without it, PostgreSQL's plpgsql engine caches the pre-bootstrap AGE label catalog on the physical connection; the subsequent phase MERGE statements then silently no-op (the 0/14,822 TemporalEvent silent-drop bug). Reconciliation script for any existing backlog: `scripts/reconcile_age_temporal_events.py` (idempotent, `--dry-run` supported).
- **PLAN-0093 D-3 SummaryWorker ordering** ‚Äî the catchup query uses `ORDER BY last_summary_attempt_at NULLS FIRST, dirtied_at ASC` so brand-new entities (NULL attempt) get summarised before stale ones, and the per-row counter resets on a successful write but persists across worker restarts. Rows with `last_summary_attempt_count > 3` are skipped without retry until a manual reset; do not "fix" stuck entities by clearing the counter alone ‚Äî investigate the LLM failure first, then run `UPDATE ‚Ä¶ SET last_summary_attempt_count = 0 WHERE entity_id IN (‚Ä¶)` only after the root cause is patched.
