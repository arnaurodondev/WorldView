# Market Data Service (S3) — Agent Context

**Port**: 8003 | **DB**: `market_data_db` (TimescaleDB) | **Status**: Mature

## Mission
Materialize OHLCV bars, quotes, and fundamentals from claim-check pointers. Serve query APIs for charts and financial data. Manage instrument lifecycle events.

## Runtime Topology (8 processes — IMPORTANT)
The API server and the consumers are SEPARATE processes (separate docker-compose `command:` entries):
- **API server** (`app.py`) — serves all routes + runs 4 daemon loops via `asyncio.create_task`: `_screen_fields_refresh_loop` (6h), `_computed_metrics_refresh_loop`, `insider_rollup_loop` (daily), `_intelligence_rollup_loop` (daily). Consumers do NOT run in the API lifespan.
- **10 Kafka consumers** — each its own `infrastructure/messaging/consumers/<name>_consumer_main.py` entrypoint: ohlcv, quotes, fundamentals, intraday_resampling, insider_transactions, prediction_market, and (PLAN-0056 A3) prediction_history, prediction_event, prediction_trade, prediction_oi.
- **Outbox dispatcher** — `infrastructure/messaging/outbox/dispatcher_main.py` (publishes instrument lifecycle events).

## Domain Entities
Security, Instrument, OhlcvBar, Quote, Fundamentals (18 sections), FundamentalMetric, PredictionMarket, PredictionMarketSnapshot, PredictionMarketPrice, PredictionMarketTrade, PredictionMarketOI, PredictionEvent (last 4 = PLAN-0056 A2 frozen dataclasses)

## Architecture Pattern (QA-013, Wave 3)
All read operations are wrapped in explicit use case classes under `application/use_cases/`:
- `query_instruments.py` — GetInstrumentByIdUseCase, GetInstrumentBySymbolUseCase, SearchInstrumentsUseCase
- `lookup_instrument.py` — InstrumentLookupUseCase (unified lookup: id > isin > symbol, optional extra_info Security fetch) [PLAN-0073 Wave B]
- `on_demand_profile.py` — OnDemandProfileUseCase (DB-first → EODHD fallback → persist; SSRF-guarded) [PLAN-0073 Wave B]
- `query_securities.py` — GetSecurityUseCase, ListSecuritiesUseCase
- `query_quotes.py` — GetQuoteUseCase, GetQuotesBatchUseCase
- `query_ohlcv.py` — GetOHLCVBarsUseCase, GetOHLCVBulkUseCase, GetAvailableTimeframesUseCase, GetOHLCVRangeUseCase

- `query_fundamental_metrics.py` — GetFundamentalMetricsTimeseriesUseCase, ScreenInstrumentsUseCase, GetAvailableFundamentalMetricsUseCase

- `query_quote_stats.py` — intraday-stats / multi-period returns / price-levels use cases (Quote tab)
- `get_sector_returns.py`, `get_period_movers.py` — `/market/*` aggregations (two-tier prev LEFT JOIN, NULLS LAST)
- `get_fundamentals_history.py` — GetFundamentalsHistoryUseCase (multi-period; explicit `period_type` per BP-540/577)
- `query_fundamentals.py` / `query_fundamentals_snapshot.py` / `query_fundamentals_metrics.py` — section + snapshot + metric reads
- `get_top_by_market_cap.py` — internal top-N for market-ingestion FundamentalsRefreshWorker (PLAN-0100)

- `get_ohlcv_bars_flexible.py` / `get_or_derive_ohlcv.py` / `derive_ohlcv.py` — OHLCV bars (flex id|symbol; derive-on-miss)
- `sync_intelligence_rollup.py` / `rollup_insider_90d.py` — daily background-loop rollups (intelligence + insider 90d)

**API routers must never import from `infrastructure/`** (enforced by import guard IG-LAYER-002). Use cases receive an already-entered UoW from `get_uow` — do NOT use `async with self._uow:` inside use case methods.

### Read/Write UoW Split (R27)
Query use cases receive a `ReadOnlyUnitOfWork` (not `UnitOfWork`) via `get_read_uow` / `ReadUoWDep` in `api/dependencies.py`. `SqlAlchemyReadOnlyUnitOfWork` opens only the read session. Only mutation use cases use `UoWDep`.

### Port Interface Pattern (N-002)
`QuoteCachePort` ABC is defined in `application/ports/cache.py`. Routers depend on `QuoteCachePort` (not `QuoteCache`) injected via `get_quote_cache()` in `api/dependencies.py`. `QuoteCache` extends `QuoteCachePort` and is only referenced in the dependency factory. This keeps the API layer decoupled from infrastructure details.

### Metrics Auth (M-004)
`GET /metrics` requires `X-Internal-Token` header (validated via `InternalAuthDep` / `verify_internal_token` in `api/dependencies.py`). Uses `hmac.compare_digest` for timing-safe comparison. Returns 401 if token missing or invalid.

## Pitfalls
- Fundamentals path param is `instrument_id` (instrument UUID), not security ID — parameter was renamed from `security_id` in Wave E2-2
- Literal routes registered before path-param routes to avoid matching conflicts
- `fundamental_metrics` is derived projection; source of truth is 18 section tables
- Caching: quotes cached 5s via Valkey
- Quote.bid/ask/last/volume are nullable (`Decimal | None`, `int | None`) — NULL means "no data", 0 means "zero activity" (D-004)
- Consumer dedup uses atomic `create_if_not_exists()` — INSERT … ON CONFLICT DO NOTHING … RETURNING (BP-035). `is_duplicate()` always returns False; `mark_processed()` is a no-op.
- `event_type`/`schema_version` in domain events are ClassVar — not dataclass instance fields. Use `type(event).event_type` or `event.event_type`; `dataclasses.asdict()` excludes them so the outbox dispatcher adds them manually.
- Instrument search uses `_escape_like()` before building ILIKE pattern — escapes `%`, `_`, `\` to prevent wildcard injection (M-002)
- `get_uow` yields an already-entered UoW (via `async with ... as uow: yield uow`) — use cases must NOT call `async with self._uow:` again
- `is_partial=True` implies `is_derived=True` — domain entity `__post_init__` enforces this invariant; never set `is_partial=True` without `is_derived=True`
- Use `find_by_instrument_timeframe_datetime_range` (datetime args) for intraday period queries; `find_by_instrument_timeframe_range` (date args) is for daily/weekly ranges only
- `query_fundamentals` for **`income_statement`** has NO repo-layer `period_type` default — callers MUST pass `period_type=PeriodType.QUARTERLY` explicitly. `GetFundamentalsHistoryUseCase` already does this (PLAN-0095 T-W1-02); any new caller must follow the same pattern or risk silent QUARTERLY+ANNUAL row mixing (BP-540 / BP-543).
- `query_fundamentals` for **`balance_sheet`** and **`cash_flow`** defaults to `PeriodType.QUARTERLY` at the repo layer when the caller passes `None` (PLAN-0096 T-W1-01, **BP-546**). To query ANNUAL rows for these sections, pass `period_type=PeriodType.ANNUAL` explicitly — the override path still works.
- **Tape endpoint two-tier fallback (PLAN-0103 W7 / BP-628)**: `_resolve_one` returns `session="prior_close"` (NOT `"unavailable"`) whenever intraday is missing but `ohlcv_bars(1d)` has a recent close; `"unavailable"` is reserved for true gaps. Any new caller of the endpoint MUST handle the new `prior_close` enum value. Prom counter `tape_symbol_data_source_total{symbol,source}` exposes the tier per request — alert on `source="unavailable"` rather than absence of data.
- `instruments.last_fundamentals_ingest_at` is bumped by `FundamentalsConsumer` inside the same UoW on every successful section materialisation (PLAN-0096 T-W1-02, BP-545). Use it for stale-ticker queries instead of joining against the section tables. The repo method ends in an explicit `await self._session.flush()` (PLAN-0101, **BP-610**) because the same UoW also runs `_upsert_fundamentals_snapshot` inside a try/except that swallows exceptions — without the flush the touch UPDATE would buffer behind a later op and silently drop on commit (live ran 0/629 non-NULL until this fix). Every new repo method that targets an observational/freshness column MUST follow the same pattern.
- `GetFundamentalsHistoryUseCase` returns every period dict with an explicit `period_type: "QUARTERLY"` field (PLAN-0097 T-W1-01, **BP-577**). The HIGHLIGHTS section is fetched without a `period_type` filter (it is TTM-only by EODHD contract) but only TTM-safe scalar fields (`PERatio`, `MarketCapitalization`) are extracted from it — never revenue/net_income. Any new field plumbed through the use case from HIGHLIGHTS MUST be flow-safe (snapshot ratio) or it will re-open the $26.4B annualized-leak class.
- Fundamentals refresh is driven by **market-ingestion's `FundamentalsRefreshWorker`** (PLAN-0099 W2, spawned by its scheduler). That worker calls market-data's internal `GET /internal/v1/instruments/top-by-market-cap` to pick the refresh universe, then re-polls EODHD → emits `market.dataset.fetched` → market-data's `FundamentalsConsumer` materializes. (Historical note BP-578: this worker did not exist before PLAN-0099; `scripts/refresh_fundamentals.py` was the manual stopgap.)
- Migration 022 runs `ANALYZE` on the 18 fundamentals tables post-019 composite indexes (PLAN-0097 T-W3-01, **BP-581**). Wrapped in `op.get_context().autocommit_block()` because ANALYZE cannot run inside a transaction. Any future composite-index migration on a query-hot table MUST ship a paired ANALYZE; the planner does not pick a fresh index until `pg_statistic` refreshes.
- `ScreenFieldMetadata.field_type` must be one of `{'numeric','text','date'}` — the DB CHECK constraint `ck_screen_field_metadata_field_type` rejects anything else (incl. `'boolean'`). Boolean-shaped fields (`has_fundamentals`, `has_ohlcv`) are stored as `numeric` (0/1) per PLAN-0098 W3, **BP-585**. The `'date'` value was added by migration 031 (WL-5c QA finding #2) so calendar fields (`next_earnings_date`, `next_dividend_date`) carry a UI-correct type instead of the prior `numeric`+`unit='date'` workaround. Adding a new static screen field with `field_type='boolean'` (or any other value) will make the periodic refresh fail every ~60s with `CheckViolationError`.
- `POST /v1/fundamentals/batch` runs two explicit `asyncio.gather` phases — phase 1 resolves every ticker → instrument_id concurrently, phase 2 fetches history for every resolved instrument concurrently (PLAN-0097 T-W3-02). The per-ticker `reason` field is restricted to four typed codes (`invalid_ticker`, `upstream_timeout`, `upstream_404`, `upstream_error`) — full exception detail goes to structlog only (PLAN-0097 T-W3-04, **BP-582**). Never re-introduce `str(exc)` into the response body.
- `POST /v1/fundamentals/batch` binds fetch outcomes to their originating ticker **structurally** via `dict(zip(pending_tickers, fetch_results, strict=True))` — NEVER iterate `fetch_results` positionally inside the assembly loop (PLAN-0099 W1 T-W1-01, **BP-592**). The prior `next(fetch_iter)` pattern silently desynced row→ticker mappings whenever a resolution failed mid-list (NVDA Q3 FY2025 got AMD's $10.3B in chat-eval Q4). Rule of thumb for any per-item batch endpoint: bind fan-out outcomes by key, never by iterator position.
- **Lock-step requirement for L-3 computed-metric screen fields** (PLAN-0089 W L-3): the 8 rows in `alembic/versions/029_seed_l3_computed_metrics_fields.py::_L3_FIELDS` MUST be byte-identical to the L-3 block in `app.py::_get_static_screen_fields()`. The 6-hour `_screen_fields_refresh_loop` re-upserts the in-memory list and will silently overwrite any divergent migration value. `test_l3_migration_lockstep.py` enforces this at CI. When adding a new field to either source, mirror it in both, run the lock-step test, and bump the canonical name in this pitfall entry.
- **Wave L-4b** (PLAN-0089): `insider_transactions` is a NEW per-row table populated by `InsiderTransactionsConsumer` from the dedicated EODHD `/insider-transactions` endpoint — distinct from `insider_transactions_snapshot` (which stores an embedded summary from the Fundamentals payload, FIX-F7). The two are intentionally disjoint. Natural-key UNIQUE `(instrument_id, filer_name, transaction_date, transaction_type, shares)` allows idempotent ON CONFLICT DO NOTHING; EODHD does not expose a stable transaction id.
- **Wave L-4b** screen-field lock-step: the `insider_net_buy_90d` row MUST stay byte-identical to migration 030's seed (label `"INSIDER 90D"`, `field_type="numeric"`, `unit="currency_compact"`) — divergence causes the 6h refresh loop to silently overwrite the migration's row. Verified by `tests/unit/test_l4b_migration_lockstep.py`.
- **Wave L-4b** internal endpoint `GET /internal/v1/instruments/ohlcv-covered` returns every `has_ohlcv=TRUE` instrument paginated by symbol ASC. Used by market-ingestion's `InsiderUniverseLoader` to expand the insider polling universe beyond the seed 3 tickers. Same JWT auth model as `top-by-market-cap`.
- **PLAN-0102 W3 — `/internal/v1/market/tape`**: futures/pre-market tape snapshot used by the rag-chat morning brief. `session` field is one of `pre-mkt`/`open`/`after-hours`/`closed`/`unavailable`. `session="open"` MEANS THE MARKET IS OPEN — UI must check this before labelling a value as "pre-market"; otherwise an `open` price becomes a misleading "pre-mkt" badge. Endpoint **never 500s** — per-symbol DB / lookup failures degrade to `session="unavailable"` with `premkt_price=null`. Symbol list is capped at 20 to protect the DB. Session boundaries are UTC-only (no DST adjustment) — fine for the brief's hour-bucket needs, never for a trading signal. The endpoint reads existing `ohlcv_bars` (1d + 5m) + `quotes`, no new ingestion path.
- **PLAN-0102 W3 — `/internal/v1/calendar/earnings`**: forward-looking earnings calendar reading the `earnings_calendar` table (NOT the S7 temporal_events route used by S9's public `/v1/fundamentals/earnings-calendar`). `consensus_rev_usd` and `period` always return `null` — EODHD does not surface those fields on the calendar endpoint; the schema keeps them present so the JSON contract is stable. `when` is the BMO/AMC/DMH shorthand translated from EODHD's verbose `BeforeMarket`/`AfterMarket`/`DuringMarket` string. Fail-open on DB exception (logged + empty `events: []`, never 500). Range cap is 90 days. When the calendar appears empty in dev, check whether the L-5b sync worker has run for the requested window — the table is unpopulated until then by design.
- **`query_screen` projects only DB-present snap-fields** (PLAN-0103 W16, **BP-635**): `fundamental_metrics_query.py::_resolve_available_snap_fields(session)` introspects `information_schema.columns` once per process and the screener projects / filters / sorts on the INTERSECTION of `_SNAP_FIELDS` with the live table schema. NEVER add an unconditional `getattr(snap, "<new_col>")` projection without going through this guard — the live `market_data_db` is allowed to lag the ORM (DDL-only migrations are not run on every release), and an ORM-projected-but-DB-absent column produces `UndefinedColumnError → HTTP 500 → rag-chat BP-623 honest-refusal`. When you add a new snapshot column: (1) add it to `_SNAP_FIELDS`, (2) the introspection guard handles it automatically (no extra code), (3) backfill via a new alembic migration so the column lands in production. Unit tests in `tests/unit/test_screener_snap_field_introspection.py` cover the guard; the autouse `_prefill_snap_field_cache` fixture in `tests/conftest.py` keeps legacy captured-session tests asserting against the full projection.
- **Screener POST branch unions `_KEY_METRICS`** (2026-06-10 frontend-audit gap #1): `query_screen`'s filtered branch enriches every page row with the module-level `_KEY_METRICS` display set (mkt cap, P/E, daily_return, revenue_ttm, …, dist_from_52w_*) via `_fetch_page_extras` — one page-bounded `DISTINCT ON (instrument_id, metric)` query. Before this, applying ANY filter blanked all non-filtered display columns. `_fetch_page_extras` also adds `volume` (latest 1d bar) and `high_52w`/`low_52w` (latest `technicals_snapshots` JSONB `52WeekHigh`/`52WeekLow`) to BOTH branches. All three lookups are fail-open (WARNING `screen_page_extras_failed`) — display extras must never 500 a working screener page. When adding a new always-on display metric, add it to `_KEY_METRICS` (single source for both branches).
- **PriceSnapshotResolver day-change on all paths** (2026-06-10 frontend-audit gap #6): FRESH_QUOTE / BULK_QUOTE / INTRADAY_5M / INTRADAY_1H now pass `prev_close=_prior_session_close(bars, price_timestamp)` into `_build` — previously only DAILY_CLOSE did, so `price_change(_pct)` was `null` on every live resolution and S9 coerced it to `+0.00%` for the whole dashboard. `_prior_session_close` compares UTC **dates** (not raw timestamps) so today's derived/partial 1d bar is never used as its own "previous close". `null` change still means "prior close unknown" (single-session history) — never emit a fake 0.

- **Screener default ORDER BY** (2026-06-12, chat-eval root cause #5): `query_screen` now resolves a default sort when the caller passes `sort_by=None` — the PRIMARY FILTER METRIC descending if a metric filter is present, else `market_capitalization` descending. When adding a new default-eligible sort target, ensure it is reachable via the page-query branch (snap col → snap join; otherwise treated as a `fundamental_metrics` metric).
- **Screener default-sort page subquery MUST be `DISTINCT ON`, NOT `GROUP BY` (2026-06-12, post-fix Theme B regression)**: the first cut of the default-sort page selection (commit `2d71ba1ae`) resolved "latest value per instrument for the sort metric" with an UN-SCOPED `SELECT instrument_id, MAX(as_of_date) FROM fundamental_metrics WHERE metric = :m GROUP BY instrument_id` + self-JOIN, executed BEFORE the LIMIT. Because the page IDs are not yet known (this subquery *selects* them), the aggregate scanned the entire metric partition → cold-cache nested-loop → 8 s `statement_timeout` → 504 → `screen_universe` transport_error. The fix rewrites it to a single `DISTINCT ON (instrument_id) ... WHERE metric = :m ORDER BY instrument_id, as_of_date DESC` scan (no aggregate, no self-JOIN); the outer query then `ORDER BY value DESC NULLS LAST LIMIT N` over the deduplicated latest-per-instrument set. Correctness is identical (top-5 by market cap stays GOOGL/AVGO/META). Backed by covering index `ix_fundamental_metrics_metric_instr_date_val` = `(metric, instrument_id, as_of_date DESC) INCLUDE (value_numeric)` (**migration 038**, paired ANALYZE per BP-581) so the dedup scan is index-only. NB for test fixtures: under the non-PG default dialect `DISTINCT ON` renders as plain `SELECT DISTINCT`, colliding with the `_fetch_page_extras` enrichment DISTINCT — disambiguate by the `… AS metric` label the enrichment arms carry (page-sort/per-filter subqueries project `value_numeric`, never a `metric` column) and compile against the PostgreSQL dialect when asserting the `DISTINCT ON` shape.

- **Screener key-metric ENRICHMENT must fan out per-metric, NEVER a double `= ANY` (2026-07-06, NEW-6 R3)**: the R1 audit found the metric-FILTER path (`sector=Technology` + `market_cap`, the chat "large-cap technology" screen) STILL 504'd (~15.6 s cold / ~8.2 s warm) after the R1/R2 `DISTINCT ON` fixes above. - **Period-movers / sector-returns: two-tier `prev` LEFT JOIN** (2026-06-12, chat-eval root cause C): `get_period_movers` and `get_sector_period_returns` previously used an INNER `JOIN LATERAL` for the prior-close bar, requiring a daily bar at-or-before `latest.bar_date - N days`. For `1D` (N=1) almost every instrument qualified; for `1W`/`1M` any instrument whose daily history was shorter than the window was silently DROPPED, collapsing weekly/monthly movers to a 1-item placeholder. `prev` is now a `LEFT JOIN LATERAL` with a two-tier pick: tier-1 = newest bar at-or-before the horizon, tier-2 (fallback) = OLDEST bar strictly before `latest`. Instruments with only one daily bar yield `prev IS NULL` → NULL return → sorted last via NULLS LAST. `1D`/`1W`/`1M` are all valid; truly unsupported periods (e.g. `7D`) still raise `ValueError` → HTTP 400 (never silent-empty). Tests assert the SQL shape in `tests/unit/test_ohlcv_repo_market_aggregations.py`.
- **Large-universe payloads can hit per-message timeout** (PLAN-0102 W6, **BP-617**): `FundamentalsConsumer` processes Russell-1000-scale sweeps with 600+ sections per message; the library-wide 45 s watchdog default was dead-lettering ~1 msg per 5 min. ## Outbox / Event Emission Pattern (QA-016, Wave 5)

Consumers emit `InstrumentCreated` and `InstrumentUpdated` by writing **directly to `uow.outbox_events`** within the same DB transaction as domain writes. This is the canonical pattern for atomic event emission:

```python
from market_data.infrastructure.messaging.outbox.dispatcher import EVENT_TOPIC_MAP, event_to_outbox_payload

event = InstrumentCreated(instrument_id=..., symbol=..., exchange=...)
await uow.outbox_events.create(
    event_type=event.event_type,
    topic=EVENT_TOPIC_MAP[event.event_type],
    payload=event_to_outbox_payload(event),
)
```

Key points:
- **Do NOT use `uow.collect_event()`** for cross-service events — requires `outbox_notifier` to be wired (BP-058); not guaranteed in app.py
- **`EVENT_TOPIC_MAP`** routes each event type to its own dedicated topic — never to a shared/legacy topic (BP-059)
- **`event_to_outbox_payload()`** adds `entity_id = instrument_id` (M-017; enforced platform-wide per [ADR-F-16](../../docs/architecture/decisions/ADR-F-16-instrument-entity-id-unification.md)) and converts `tuple → list` for Avro arrays
- **Consumers must emit `InstrumentUpdated`** when `update_flags()` is called — not just `InstrumentCreated` (BP-061)
- **`InstrumentCreated.schema_version = 2`** since Wave 5: has optional `name`, `isin`, `instrument_type` fields
- **Canonical Avro schemas** in `infra/kafka/schemas/` (not service-local `infrastructure/messaging/schemas/`)

## Intraday Resampling (PLAN-0040 Waves B-1..B-3)
- `IntradayResamplingWorker(BaseKafkaConsumer)` on group `market-data-intraday-resampling`, topic `market.dataset.fetched`
- Filters: only `dataset_type=ohlcv, timeframe=1m` events are processed; all others ACK'd immediately
- Calls `ResampledOHLCVUseCase` for each target timeframe in `_RESAMPLE_TARGETS = ["5m","15m","30m","1h","4h","1d"]`
- `_floor_to_period(bar_dt, period_seconds)`: `epoch_seconds = int(bar_dt.timestamp()); (epoch_seconds // period_seconds) * period_seconds`
- Partial-bar invariant: every resampled bar in an incomplete period has `is_partial=True, is_derived=True`
- `OHLCVBar.is_partial: bool = False` with `__post_init__` asserting `if self.is_partial and not self.is_derived: raise ValueError`
