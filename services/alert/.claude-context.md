# S10 Alert Service ‚Äî Agent Context

**Port**: 8010 | **DB**: `alert_db` (OWNED, Alembic enabled) | **Env prefix**: `ALERT_`

## Mission
Consume intelligence + watchlist events, resolve affected users via S1 REST API, dedup alerts per entity+type+window, push to WebSocket, store for offline clients.

## Domain Entities
- `Alert` ‚Äî core alert with dedup_key (sha256, UNIQUE)
- `PendingAlert` ‚Äî per-user unacknowledged alert
- `AlertDelivery` ‚Äî delivery tracking per user/channel
- `AlertSubscription` ‚Äî user‚Üíentity subscription
- `OutboxEvent` / `DeadLetterEntry` ‚Äî outbox pattern
- `EmailPreference` ‚Äî per-user digest schedule (day/hour/timezone)
- `AlertRule` (PLAN-0113) ‚Äî user standing rule; `last_state` JSONB = edge-trigger memory; KG_CONNECTION needs 2 distinct nodes, else `entity_id`
- `SeverityThresholds` / `EvalResult` ‚Äî VOs (severity classification, rule-eval outcome)

## Enums
- `AlertSeverity`: LOW (`"low"`), MEDIUM (`"medium"`), HIGH (`"high"`), CRITICAL (`"critical"`) ‚Äî **query params must be lowercase** (StrEnum)
- `AlertType`: SIGNAL (`"SIGNAL"`), GRAPH_CHANGE (`"GRAPH_CHANGE"`), CONTRADICTION (`"CONTRADICTION"`), USER_RULE (`"user_rule"` ‚Äî added PLAN-0082 Wave B for LLM `create_alert` tool), PREDICTION (`"prediction"` ‚Äî added PLAN-0056 Wave D3 for prediction-market fanout alerts). Stored as VARCHAR(100), NO CHECK (BP-007, no PG enum ‚Üí no alerts-table migration)
- `RuleType` (PLAN-0113): PRICE_CROSS, NEWS_COUNT, NEWS_MOMENTUM, KG_CONNECTION, FUNDAMENTAL_CROSS, PREDICTION (PLAN-0056 Wave D3) ‚Äî stored VARCHAR(50)+CHECK (`ck_alert_rules_rule_type`, widened by migration 0011); one evaluator each in `application/rules/registry.py`. NOTE: prediction alerts fire via the watchlist fanout regardless of any rule; the PREDICTION rule type is a user-configurable score/polarity/trigger toggle only
- `OutboxStatus`: pending, dispatched, failed
- `DeliveryChannel`: websocket
- `DeliveryStatus`: pending, delivered
- `DLQStatus`: failed, resolved

## Key Patterns
- **Dedup**: `sha256(entity_id:alert_type:window_bucket)` ‚Äî AD-9, no source_event_id
- **S1 Client**: Best-effort ‚Äî 503/connection error ‚Üí empty list, never raises
- **Watchlist Cache**: Valkey cache-aside, key `s10:v1:watchlist:by_entity:{eid}`, TTL=300s
- **Outbox**: Transactional outbox for `alert.delivered.v1`; outbox stores pre-serialized Avro bytes
- **AlertFanoutUseCase**: single transaction (alert+pending+outbox), WS push AFTER commit
- **IntelligenceConsumer**: resolves topic from `event_type` field (fallback: `X-Source-Topic` header)
- **ConnectionManager**: in-memory dict; single-replica constraint; replaces stale connections on send failure
- **WebSocket polling**: route uses `pubsub.get_message(ignore_subscribe_messages=True, timeout=30.0)` ‚Äî on None (timeout) sends `{"type":"ping"}` to detect stale connections; avoids indefinitely blocked `listen()` coroutines
- **Backfill suppression**: signal/graph ‚Üí suppress all backfill; contradiction ‚Üí suppress if backfill AND age > 30 days

## AVRO-FILE-ONLY Rule
All Avro schemas MUST be stored in `infra/kafka/schemas/*.avsc` and loaded with `fastavro.schema.load_schema(<path>)`.
Never define Avro schemas as inline Python dicts ‚Äî they will drift from the canonical `.avsc` file.
See BP-119.

## Rule Engine (PLAN-0113)
- `alert-rule-poller` process wakes every `ALERT_ALERT_RULE_POLL_TICK_SECONDS` (60s); per-type cadence (`AlertRule.is_due`) throttles which rules are read each tick.
- Edge-triggered via `last_state` JSONB (e.g. `was_above`, `last_count`, `connected`); changing a rule's `condition` re-arms (`last_state=NULL`).
- Master switch `ALERT_ALERT_RULE_POLLER_ENABLED` (default true) ‚Üí instant rollback; per-user cap `ALERT_ALERT_RULE_MAX_PER_USER` (200, 429 on exceed).
- Metrics: `s10_rule_poller_runs_total{outcome}`, `s10_rule_poller_last_success_timestamp_seconds` (watchdog), `s10_rule_poller_due_rules`. Prometheus rules: `infra/prometheus/rules/alert_rule_poller.yml`.

## REST API
> `user_id`/`tenant_id` are taken from the RS256 internal JWT (InternalJWTMiddleware), **never** from a query param.
- `POST /api/v1/alerts` ‚Äî create user-initiated alert (alert_type=USER_RULE); 201 / 409 dup
- `GET /api/v1/alerts/pending?limit=50&offset=0[&min_severity=low|medium|high|critical]` ‚Äî paginated unacked; ReadDbSessionDep (R27)
- `DELETE /api/v1/alerts/{alert_id}/ack` ‚Äî acknowledge per-user pending row (200 ok / 404)
- `PATCH /api/v1/alerts/{alert_id}/acknowledge` ‚Äî tenant-level ack (idempotent)
- `PATCH /api/v1/alerts/{alert_id}/snooze` ‚Äî body `{"until": <ISO>}`, `now < until <= now+30d`
- `GET /api/v1/alerts/history?severity=&entity_id=&from=&to=&status=active|acknowledged|snoozed|all&limit=&offset=` ‚Äî tenant history
- `WebSocket /api/v1/alerts/stream?token=<ws-jwt>` ‚Äî real-time push; 30s ws-token from S9, scope `alerts:stream`, aud `worldview-internal`
- **Alert rules (PLAN-0113)**: `POST /api/v1/alert-rules` (201/400/409/422/429), `GET /api/v1/alert-rules` (filters enabled,rule_type), `GET|PATCH|DELETE /api/v1/alert-rules/{rule_id}`
- `GET /internal/v1/instruments/{instrument_id}/active-alert-flag` ‚Äî screener sync (PLAN-0089 L-5a)
- `GET /internal/v1/users/{user_id}/alerts/pending` ‚Äî service-caller variant; requires role==system AND service_name in `_SERVICE_BRIEF_ALLOWED` else 403 (PLAN-0094)
- `GET|PUT /api/v1/email/preferences` ‚Äî email digest prefs
- `POST /admin/email/digest/trigger` ‚Äî manual digest trigger (X-Admin-Token), 202
- `GET /healthz` ‚Äî liveness
- `GET /readyz` ‚Äî readiness (checks alert_db, Kafka, Valkey, S1)
- `GET /metrics` ‚Äî Prometheus (s10_ prefix)
- `GET /admin/dlq` ¬∑ `GET /admin/dlq/{dlq_id}` ¬∑ `POST /admin/dlq/{dlq_id}/resolve` ‚Äî DLQ admin (X-Admin-Token required)

## Pitfalls
- `pytest-asyncio>=0.23.4` required (0.23.0 crashes with asyncio_mode=auto + __init__.py)
- `validation_alias` on Settings needs `populate_by_name=True` for kwargs in tests
- S1 may not be running ‚Äî always test with graceful degradation (best-effort contract)
- Watchlist reverse-index cached in Valkey (eventually consistent)
- Dedup window per entity+alert_type (300s default). PLAN-0056 QA: `Alert.compute_dedup_key` takes an optional `discriminator`; `AlertFanoutUseCase` threads `market_id:trigger` for `AlertType.PREDICTION` ONLY (so two distinct prediction markets on the same entity in one window each raise their own alert). All other alert types pass `discriminator=None` ‚Üí key identical to the historical per-entity+type key (their collapse is intended, unchanged). Repeated moves on the SAME market still collapse and are rate-limited by the S7 `PredictionRuleEvaluator` 1h per-market cooldown.
- Pending alerts retained 7 days (`ALERT_PENDING_ALERT_TTL_DAYS`)
- WebSocket push must be post-commit (never inside the DB transaction)
- `ConnectionManager._connections` is in-memory only ‚Äî ‚â•2 replicas need Valkey Pub/Sub (`ValkeyNotificationPublisher` is the production path)
- Outbox dispatcher uses raw `confluent_kafka.Producer` (no schema registry; bytes pre-serialized by fanout)
- `fastavro.parse_schema` returns an opaque type; use `# type: ignore[assignment]` for mypy
- `ack` endpoint returns 200 (not 204) ‚Äî FastAPI 0.111 requires explicit `Response` for 204, return dict instead
- `create_app()` does NOT run lifespan ‚Äî wire `app.state.*` manually in tests (bypass lifespan for speed)
- `AlertFanoutUseCase` takes `notification_publisher: INotificationPublisher` (not `connection_manager`) ‚Äî tests must use `notification_publisher=` kwarg
- `severity` query param must be lowercase (`?min_severity=high` not `HIGH`) ‚Äî `AlertSeverity` is a StrEnum with lowercase values
- Snooze body field is `until` NOT `snooze_until`
- `ALERT_JTI_REPLAY_CHECK_ENABLED=false` by default ‚Äî S10 receives forwarded user JWTs from S8 (rag-chat briefing calls) that have already been consumed by InternalJWTMiddleware on S8; re-checking here causes 401
- `ALERT_S1_INTERNAL_JWT`, `ALERT_S7_INTERNAL_JWT`, `ALERT_S8_INTERNAL_JWT` must be pre-signed long-lived RS256 service JWTs; obtain via `POST /internal/v1/service-token` on S9 or generate offline
- Integration tests use testcontainers for Postgres + fakeredis for Valkey + pytest-httpserver for S1
- `alert_type` is VARCHAR(100) (NOT 30) and `alert_rules.rule_type` VARCHAR(50)+CHECK ‚Äî never PG enums (BP-007), new types add zero DDL
- Rule `condition` is a discriminated-union VO validated at the API boundary; `rule_type` is immutable on PATCH (only `condition` re-validates against existing type)
- KG_CONNECTION rules require `node_a_entity_id != node_b_entity_id` (422); all other types require `entity_id`
- Rule poller is edge-triggered via `last_state` ‚Äî duplicate fires are suppressed by state, not the dedup window; changing `condition` clears `last_state`
