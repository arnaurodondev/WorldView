# S10 Alert Service — Agent Context

**Port**: 8010 | **DB**: `alert_db` (OWNED, Alembic enabled) | **Env prefix**: `ALERT_`

## Mission
Consume intelligence + watchlist events, resolve affected users via S1 REST API, dedup alerts per entity+type+window, push to WebSocket, store for offline clients.

## Domain Entities
- `Alert` — core alert with dedup_key (sha256, UNIQUE)
- `PendingAlert` — per-user unacknowledged alert
- `AlertDelivery` — delivery tracking per user/channel
- `AlertSubscription` — user→entity subscription
- `OutboxEvent` / `DeadLetterEntry` — outbox pattern

## Enums
- `AlertSeverity`: LOW, MEDIUM, HIGH, CRITICAL (derived from market_impact_score via SeverityThresholds VO)
- `AlertType`: SIGNAL, GRAPH_CHANGE, CONTRADICTION
- `OutboxStatus`: pending, dispatched, failed
- `DeliveryChannel`: websocket (EMAIL planned — extend enum + add EmailNotificationPublisher adapter + alert-email-dispatcher process)
- `DeliveryStatus`: pending, delivered

## Key Patterns
- **Dedup**: `sha256(entity_id:alert_type:window_bucket)` — AD-9, no source_event_id
- **S1 Client**: Best-effort — 503/connection error → empty list, never raises
- **Watchlist Cache**: Valkey cache-aside, key `s10:v1:watchlist:by_entity:{eid}`, TTL=300s
- **Outbox**: Transactional outbox for `alert.delivered.v1`; outbox stores pre-serialized Avro bytes
- **AlertFanoutUseCase**: single transaction (alert+pending+outbox), WS push AFTER commit
- **IntelligenceConsumer**: resolves topic from `event_type` field (fallback: `X-Source-Topic` header)
- **ConnectionManager**: in-memory dict; single-replica constraint; replaces stale connections on send failure
- **WebSocket polling**: route uses `pubsub.get_message(ignore_subscribe_messages=True, timeout=30.0)` — on None (timeout) sends `{"type":"ping"}` to detect stale connections; avoids indefinitely blocked `listen()` coroutines
- **Backfill suppression**: signal/graph → suppress all backfill; contradiction → suppress if backfill AND age > 30 days

## AVRO-FILE-ONLY Rule
All Avro schemas MUST be stored in `infra/kafka/schemas/*.avsc` and loaded with `fastavro.schema.load_schema(<path>)`.
Never define Avro schemas as inline Python dicts — they will drift from the canonical `.avsc` file.
See BP-119.

## Pitfalls
- `pytest-asyncio>=0.23.4` required (0.23.0 crashes with asyncio_mode=auto + __init__.py)
- `validation_alias` on Settings needs `populate_by_name=True` for kwargs in tests
- S1 may not be running — always test with graceful degradation
- Deployment gate: Cannot deploy until S1 provides 4 internal endpoints
- Watchlist reverse-index cached in Valkey (eventually consistent)
- Dedup window per entity+alert_type (300s default)
- Pending alerts retained 7 days
- WebSocket push must be post-commit (never inside the DB transaction)
- `ConnectionManager._connections` is in-memory only — ≥2 replicas need Redis Pub/Sub
- Outbox dispatcher uses raw `confluent_kafka.Producer` (no schema registry; bytes pre-serialized by fanout)
- `fastavro.parse_schema` returns an opaque type; use `# type: ignore[assignment]` for mypy
- Import guard baseline: `watchlist_cache.py:17:IG-MSG-002` is a known E-1 TYPE_CHECKING-only violation
- `ack` endpoint returns 200 (not 204) — FastAPI 0.111 requires explicit `Response` for 204, return dict instead
- `create_app()` does NOT run lifespan — wire `app.state.*` manually in tests (bypass lifespan for speed)
- `AlertFanoutUseCase` takes `notification_publisher: INotificationPublisher` (not `connection_manager`) — tests must use `notification_publisher=` kwarg
- `WatchlistCache(valkey_client, ...)` needs `# type: ignore[arg-type]` since ValkeyClient ≠ Redis type
- Integration tests use testcontainers for Postgres + fakeredis for Valkey + pytest-httpserver for S1
