# Alert Service (S10) — Agent Context

**Port**: 8010 | **DB**: `alert_db` | **Status**: Compliant

## Mission
Consume intelligence + watchlist events, resolve affected users via S1 REST API, dedup alerts per user+entity+type, push to WebSocket, store for offline clients.

## Domain Entities
Alert, PendingAlert, AlertDedup

## Kafka Topics
| Direction | Topic | Notes |
|-----------|-------|-------|
| Produces | `alert.delivered.v1` | Alert successfully delivered |
| Consumes | `nlp.signal.detected.v1` | Financial signals from S6 |
| Consumes | `graph.state.changed.v1` | Graph changes from S7 |
| Consumes | `intelligence.contradiction.v1` | Contradictions from S7 |
| Consumes | `portfolio.watchlist.updated.v1` | Watchlist changes from S1 |

## Key Tables
alerts, pending_alerts, alert_dedup, outbox_events, idempotency

## API Endpoints (3)
WebSocket stream, pending alerts, acknowledge

## Dependencies
Requires S1 internal endpoints:
- Watchlist lookup
- Batch entity lookup
- Internal token auth (`INTERNAL_SERVICE_TOKEN` header)

## Pitfalls
- Deployment gate: Cannot deploy until S1 provides 4 internal endpoints
- Watchlist reverse-index cached in Valkey (eventually consistent)
- Dedup window per user+entity+alert_type (300s default)
- Pending alerts retained 7 days
