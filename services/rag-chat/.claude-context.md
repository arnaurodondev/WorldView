# RAG / Chat Service (S8) — Agent Context

**Port**: 8008 | **DB**: `rag_db` (owned, Alembic enabled, head `0010`) | **Status**: tool-use chat pipeline + **28-tool catalog (`capability_manifest.yaml` v7)** + SSE streaming + chat/thread endpoints + morning/instrument briefings + brief pre-generation scheduler (second process) + per-source circuit breaker + optional citation cron + S9 gateway routes

## Mission
Tool-use chat pipeline: load history → LLM first turn with the 27-tool schema → execute tool calls → inject results → LLM second turn streams answer. SSE events: `thinking`, `status`, `tool_call`, `tool_result`, `token`, `citations`, `contradictions`, `suggestions`, `metadata`. Conversation persistence in `rag_db`. Also owns morning/instrument briefings (on-demand routes + a separate APScheduler pre-gen process).

## Key Invariants
- `fusion_score = score × recency_score × trust_weight` — deviation > 1e-9 is a programming error
- `recency_score = exp(-0.005 × days_old)` where `days_old = (now() - published_at).days`; `published_at is None` → `recency_score = 0.5`
- Trust weights: `sec_10k/10q → 0.95`, `earnings_data → 0.95`, `eodhd_news → 0.70`, `finnhub_news → 0.65`, `other → 0.50`
- Input max length: 2000 chars after HTML strip; empty after strip → EMPTY error

## Entity Resolution Routing (BP-485)
- `entity_resolution` phase calls **S6 (NLP pipeline, port 8006)** via `S6Client.resolve_entities()` → `POST /api/v1/entities/resolve`
- KG (port 8007) has only `GET /entities/resolve` (fuzzy alias lookup used by ToolExecutor directly, NOT by the entity_resolution phase)
- NLP pipeline resolver: 5-stage cascade; API process runs only stages 1-3 (DB queries, no ML); stages 4/5 (GLiNER, ANN) are injected in workers only → entity resolve always responds in <200ms container-to-container
- If NLP pipeline goes unhealthy (BP-485), entity resolve times out after `RAG_CHAT_UPSTREAM_TIMEOUT_SECONDS` (default 5.0) → `UpstreamTransportError` → empty entity list → LLM produces ungrounded answer (no visible error to user)
- Restart fix: `docker-compose up -d rag-chat` (if container in Created state) + `docker-compose restart nlp-pipeline` (if unhealthy)

## Pipeline (Tool-Use Loop — PLAN-0067, replaces 13-step classical pipeline)

**IMPORTANT**: The live `ChatOrchestratorUseCase` uses the tool-use loop as its ONLY chat path. There is NO feature flag (no `TOOL_USE_ENABLED`). The legacy classical-pipeline intent-classifier stack (`IntentClassifier`, `RetrievalPlanBuilder`, `IntentClassifierPort`) has been RETIRED (see note below); `ParallelRetrievalOrchestrator` still exists and backs the eval-only `/v1/internal/retrieve` path. Do NOT wire any of these into the chat orchestrator.

```
1. Input validation (bleach strip → PII → injection heuristic)
2. Cache check + rate limit
3. Load history → release UoW (UoW must be released before long I/O)
4. emit_thinking(stage)
5. LLM first turn: chat_with_tools(messages, tool_schemas)
6. For each tool_call in LLM response:
     emit_tool_call(tool_name, input_summary, status="running")
     ToolExecutorFactory.execute(tool_name, tool_input) → result
     emit_tool_result(tool_name, status="ok"|"error"|"empty", item_count)
     inject result into context
7. All-tools-failed guard (if ALL tools returned error/empty, skip second turn)
8. LLM second turn: stream_chat(messages_with_results) → emit token events
9. Output processing (strip <think>/<reasoning> tags, citation injection)
10. Re-acquire UoW → persist thread + message to rag_db
```

### Critical implementation pitfalls (post-QA fixes)
- **ToolExecutorFactory wiring**: `ToolExecutorFactory` MUST be instantiated in `app.py` lifespan and stored on `app.state`; it is NOT auto-wired. Forgetting this causes `AttributeError` at runtime on first chat request.
- **ToolUseBlock field name**: Use `.id` (not `.tool_use_id`) when constructing `ToolCallProvenance` from a `ToolUseBlock`. The field is named `id` in the dataclass definition.
- **Cypher injection guard**: `traverse_graph` handler validates the `relationship_type` parameter against `_ALLOWED_CYPHER_REL_TYPES` frozenset in `tool_executor.py`. NEVER pass raw Cypher strings from the LLM directly to S7 — always validate against the allowlist. `RAG_CHAT_CYPHER_ENABLED` config flag has been REMOVED; Cypher traversal is always available but always guarded by the allowlist.
- **`LlmChatProvider` Protocol**: LLM adapters that support tool-use must implement both `chat_with_tools` and `stream_chat`. Adapters that only implement `stream_chat` satisfy `LlmStreamProvider` only — they cannot be used for the tool-use first turn.
- **UoW release before tool loop**: The unit-of-work MUST be released (committed/rolled back) before entering the tool-call loop. Tool calls are potentially slow (up to `UPSTREAM_TIMEOUT_SECONDS` each). Holding the UoW open across tool I/O causes connection pool exhaustion under load. Re-acquire a new UoW for the final persist step.
- **All-tools-failed guard**: If every tool call returns `status="error"` or `status="empty"`, the orchestrator MUST NOT invoke the second LLM turn with zero context — it short-circuits to a fallback answer. Skipping this guard causes confabulation.
- **`llm_usage_log` new columns**: Migration `0006_extend_llm_usage_log_tool_tracking.py` adds `prompt_cache_read_tokens`, `prompt_cache_creation_tokens`, `tool_calls_count`, `tool_names ARRAY(TEXT)`. Follow BP-427: use raw SQL `ALTER TABLE … ADD COLUMN IF NOT EXISTS …` in migrations. Migration `0010` (PLAN-0117 W2) adds `cost_source VARCHAR(16)` + `user_id UUID` (both nullable).
- **Cost metering (PLAN-0117 W4)**: NEVER hardcode `estimated_cost_usd=0.0` for a paid model. The leaf LLM cost is recorded ONCE at the DeepInfra adapter boundary (`_record_cost` → `CostRecorder.record`), which now resolves the §2.2 priority via **`ml_clients.resolve_cost`** (provider `usage.estimated_cost` → local → pricematrix) and stamps `cost_source` + threads `user_id`. **OQ-3 (leaf vs aggregate, resolved live 2026-07-01)**: the `capability='chat_with_tools'` rows are an AGGREGATE WRAPPER logged by `provider_chain` around the SAME round-trip the adapter records as the `tool_loop_iter` LEAF — the wrapper stays at `$0` with `cost_source='aggregate'` to avoid double counting (each real request costed exactly once). `tool_loop_iter`/`synthesis` are the priced leaves; `chat_with_tools` is the $0 aggregate. The W5 silent-zero guard MUST exempt `cost_source='aggregate'`.
- **Thread/user attribution — resolve the id EARLY, not at persist time (2026-07-23 fix)**: `ChatOrchestratorUseCase.execute_streaming` resolves `effective_thread_id = request.thread_id or _turn_id` ONCE at the top of the turn and threads it through `_execute_streaming_inner` as an explicit parameter — every LLM cost-attribution call site (planning `chat_with_tools`, synthesis/degraded-reprompt/grounding-rewrite `stream_chat`) AND the final `persist_chat` call use THIS value, never `request.thread_id` directly. Before this fix, `request.thread_id` stayed `None` for the whole turn on every NEW conversation (the client has no thread_id yet), while the persist step independently minted a SECOND, different UUID at the very end — so every `llm_usage_log` row for a first-turn conversation had `chat_thread_id=NULL` (100% of prod rows over 7 days, `docs/audits/2026-07-23-three-vertical-prod-investigation.md`). If you add a new LLM call site in the orchestrator, use `effective_thread_id`, NOT `request.thread_id`, for cost attribution. The raw `request.thread_id` (possibly `None`) is still correct and MUST be left alone for `check_cache`/`write_completion_cache`/`load_history` — those intentionally key off the client-supplied value (eval-harness cache-key determinism). ## SSE Event Types (stream endpoint)
- `thinking` — `{stage: str}` — emitted before first LLM call; shows pulsing indicator in UI
- `status` — pipeline phase updates (`loading_context`, `entity_resolution`, `query_expansion`)
- `tool_call` — `{type, tool, label, input, status: "running"}` — before each tool executes
- `tool_result` — `{type, tool, status: "ok"|"error"|"empty"|"transport_error", item_count, duration_ms?, result_preview?}` — after each tool completes. `duration_ms` is server-measured wall-clock (frontend prefers it over client-side timing); `result_preview` is a bounded list (≤3 of `{id, title}`, built by `SSEEmitter.build_result_preview`) attached on non-transport-error results
- `suggestions` — JSON array of exactly 3 follow-up question strings, emitted after `contradictions` and before `metadata`. Derived deterministically (NO extra LLM call) from resolved entities + executed tool names — see `application/services/suggestions.py`. Toggle: `RAG_CHAT_SUGGESTIONS_ENABLED` (default true, read per-call from env). Frontend prefers server suggestions over its client-templated ones
- `token` — LLM output token
- `citations` — JSON array of Citation objects
- `contradictions` — JSON array of flagged contradiction pairs
- `metadata` — retrieval stats (items_retrieved, reranked_count, latency_ms)
- `error` — error details (closes stream)
- `pending_action` — emitted when a write-action tool proposes an action awaiting user confirmation (PLAN-0082)
- `action_executed` — emitted when a confirmed action is executed successfully (PLAN-0082)
- `action_rejected` — emitted when a user rejects an action proposal or execution fails (PLAN-0082)

## Process Topology (R22 — PLAN-0015-D Wave D-3 + PLAN-0084 A-1 + PLAN-0094 W2)

## Critical Rules
- **R25**: API routes only import use case classes — never `infrastructure/` directly
- **R27**: Threads/messages reads use `ReadOnlyUnitOfWork`; writes use `UnitOfWork`
- **R26**: No auto-commit in `UnitOfWork.__aexit__` — explicit `await uow.commit()` in every use case
- **R22**: No `asyncio.create_task()` in app lifespan — ASGI only
- **Never** store raw LLM output in DB — always strip `<think>` / `<reasoning>` blocks first
- **Never** interpolate user input into prompt templates without XML-wrapping
- Rate limit enforced per `(tenant_id, user_id)` via Valkey sliding window
- All upstream service calls (S6, S7, S3, S1) use `httpx.AsyncClient` with timeouts
- Thread ownership: every thread read/write verifies `thread.tenant_id == request_tenant_id`

- All `valkey` constructor parameters must be typed as `ValkeyClient` (from `messaging.valkey.client`), never `aioredis.Redis` or `redis.asyncio.Redis` — see BP-103b. New `ValkeyClient` methods must be staged in the same commit as the callers to avoid pre-commit hook `attr-defined` failures.

Attempt 1 runs as before; if it raises BEFORE yielding any token AND the error string matches a transient signature (`"429"` / `"too many requests"` / `"all llm providers failed"` / `"timeout"` / `"503"` / `"502"` / `"504"`), the wrapper waits 0.75 s and retries exactly once. Mid-stream failures (any token already yielded) MUST bypass the retry (raised up to the existing FIX-LIVE-V partial-content recovery) — retrying mid-stream would emit duplicate tokens. Non-transient errors (`ValueError`, validation, etc.) short-circuit to one attempt — doubling latency on bugs that will never recover is the wrong tradeoff. Worst-case extra latency ≈ 1.5 s, bounded inside benchmark budgets. The retry is logged as `tool_use_second_turn_transient_retry` (warning level) so operators can see upstream rate-limit pressure even when users never see a failure. NEVER swallow the synthesis error silently — the log line is the only signal that the chain has a single-provider-burn risk. Regression gates: `tests/unit/use_cases/test_chat_orchestrator_fallback.py::TestPartialStreamRecovery::test_stream_chat_transient_429_retries_once_and_recovers` + `::test_stream_chat_non_transient_error_does_not_retry`.

When you add a new `_handle_*` method to ANY handler, add it to that handler's `dispatch` dict in `execute()` (do NOT introduce a new `if tool_name == …: return await self._handle_X(**args)` branch; that bypasses the filter). Regression gate: `tests/unit/application/pipeline/handlers/test_kwargs_filter.py`.
