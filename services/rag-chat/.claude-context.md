# RAG / Chat Service (S8) — Agent Context

**Port**: 8008 | **DB**: `rag_db` (owned, Alembic enabled) | **Status**: PLAN-0080 Wave A COMPLETE — **PLAN-0067 W11 (Full Tool Catalog) COMPLETE** — tool-use chat pipeline + **14-tool catalog (v2 manifest)** + SSE streaming + chat endpoints + S9 gateway routes + per-source circuit breaker (PLAN-0031 D-1)

## Mission
Tool-use chat pipeline: load history → LLM first turn with 14-tool schema → execute tool calls → inject results → LLM second turn streams answer. SSE events: `thinking`, `tool_call`, `tool_result`, `token`, `citations`. Conversation persistence in `rag_db`.

## Key Invariants
- `fusion_score = score × recency_score × trust_weight` — deviation > 1e-9 is a programming error
- `recency_score = exp(-0.005 × days_old)` where `days_old = (now() - published_at).days`; `published_at is None` → `recency_score = 0.5`
- Trust weights: `sec_10k/10q → 0.95`, `earnings_data → 0.95`, `eodhd_news → 0.70`, `finnhub_news → 0.65`, `other → 0.50`
- Input max length: 2000 chars after HTML strip; empty after strip → EMPTY error

## Pipeline (Tool-Use Loop — PLAN-0067, replaces 13-step classical pipeline)

**IMPORTANT**: The classical 13-step pipeline is GONE. Tool-use is the ONLY chat path. There is NO feature flag (no `TOOL_USE_ENABLED`). Do NOT recreate the deleted components listed below.

```
1. Input validation (bleach strip → PII → injection heuristic)
2. Cache check + rate limit
3. Load history → release UoW (UoW must be released before long I/O)
4. emit_thinking(stage)
5. LLM first turn: chat_with_tools(messages, tool_schemas)
6. For each tool_call in LLM response:
     emit_tool_call(tool_name, input_summary, status="running")
     ToolExecutorFactory.execute(tool_name, tool_input) → result
     emit_tool_result(tool_name, status="ok"|"error"|"empty", item_count)
     inject result into context
7. All-tools-failed guard (if ALL tools returned error/empty, skip second turn)
8. LLM second turn: stream_chat(messages_with_results) → emit token events
9. Output processing (strip <think>/<reasoning> tags, citation injection)
10. Re-acquire UoW → persist thread + message to rag_db
```

### Critical implementation pitfalls (post-QA fixes)
- **ToolExecutorFactory wiring**: `ToolExecutorFactory` MUST be instantiated in `app.py` lifespan and stored on `app.state`; it is NOT auto-wired. Forgetting this causes `AttributeError` at runtime on first chat request.
- **ToolUseBlock field name**: Use `.id` (not `.tool_use_id`) when constructing `ToolCallProvenance` from a `ToolUseBlock`. The field is named `id` in the dataclass definition.
- **Cypher injection guard**: `traverse_graph` handler validates the `relationship_type` parameter against `_ALLOWED_CYPHER_REL_TYPES` frozenset in `tool_executor.py`. NEVER pass raw Cypher strings from the LLM directly to S7 — always validate against the allowlist. `RAG_CHAT_CYPHER_ENABLED` config flag has been REMOVED; Cypher traversal is always available but always guarded by the allowlist.
- **`LlmChatProvider` Protocol**: LLM adapters that support tool-use must implement both `chat_with_tools` and `stream_chat`. Adapters that only implement `stream_chat` satisfy `LlmStreamProvider` only — they cannot be used for the tool-use first turn.
- **UoW release before tool loop**: The unit-of-work MUST be released (committed/rolled back) before entering the tool-call loop. Tool calls are potentially slow (up to `UPSTREAM_TIMEOUT_SECONDS` each). Holding the UoW open across tool I/O causes connection pool exhaustion under load. Re-acquire a new UoW for the final persist step.
- **All-tools-failed guard**: If every tool call returns `status="error"` or `status="empty"`, the orchestrator MUST NOT invoke the second LLM turn with zero context — it short-circuits to a fallback answer. Skipping this guard causes confabulation.
- **`llm_usage_log` new columns**: Migration `0006_extend_llm_usage_log_tool_tracking.py` adds `prompt_cache_read_tokens`, `prompt_cache_creation_tokens`, `tool_calls_count`, `tool_names ARRAY(TEXT)`. Follow BP-427: use raw SQL `ALTER TABLE … ADD COLUMN IF NOT EXISTS …` in migrations.
- **PLAN-0093 E-1 tool-call dedup**: `_tool_result_cache` in `chat_orchestrator._execute_streaming_inner` is keyed by `(tool_name, frozenset((str(k), repr(v))))` — `repr(v)` is mandatory so list/dict inputs hash stably. The cache lives for the duration of one call to `_execute_streaming_inner`; the turn boundary clears it implicitly. Never lift the cache to instance scope — it would cross-pollinate between users.
- **PLAN-0093 E-2 numeric grounding rewrite, not silent strip**: when `NumericGroundingValidator.validate()` returns `passed=False`, `_run_grounding_validation` issues ONE additional `stream_chat` re-prompt that lists the unsupported numbers and asks the LLM to rewrite. Only if the rewrite ALSO fails do we append the "⚠ Some numbers could not be verified" banner. Never strip the failing numbers in code — the user must either see verified numbers or be told they aren't verified.
- **PLAN-0093 E-3 name resolution ambiguity**: `_resolve_entity_by_name` in `application/pipeline/handlers/intelligence.py` always returns `candidates[0]` from `S7.resolve_entity_by_name(name, limit=3)`. The S7 alias-search endpoint is responsible for ranking by similarity; rag-chat does not enforce an explicit ambiguity threshold. When the alias index returns nothing, the helper returns `None` and logs `tool_entity_unresolved` so the orchestrator's all-tools-failed guard or E-4 fallback can act.
- **PLAN-0093 E-4 multi-tool fallback chain**: `_FALLBACK_MAP` in `chat_orchestrator.py` is a small, explicit whitelist: `search_documents → get_entity_intelligence`, `get_contradictions → search_claims` (`polarity=negative` injected), `get_economic_calendar → get_temporal_events`, `search_claims → search_documents`. Fallback runs on iteration 0 only when ALL primary tools returned empty AND no action is pending. Each fallback consumes the same per-turn budget (`cumulative_tool_latency` accumulates inside the surrounding loop). Adding a new entry should be deliberate — paired with an audit log entry and a regression test.
- **PLAN-0093 E-5 citation scrubber**: `_scrub_orphan_citations(text, max_index=len(reranked))` runs in `chat_orchestrator.py:774`, post-LLM and pre-delivery. Markers `[N1]..[N<max_index>]` survive; `[N<max_index+1>]+` are deleted in place (text retained, marker stripped). Pair with `_scrub_unseen_refs` which redacts `entity:UUID`/`article:UUID` references not in the live `seen_item_ids` set harvested from tool results across all iterations.
- See `docs/services/rag-chat.md` §PLAN-0093 E-Wave Hardening for full architecture, file pointers, and the audit cross-reference table (F-CHAT-AGENT-001..005).
- **PLAN-0095 W3 tool descriptions**: `get_entity_graph`, `traverse_graph`, `get_entity_paths`, `get_entity_intelligence`, `compare_entities` each carry explicit "DO NOT use for…" anti-pattern clauses redirecting the model to the correct sibling by name. Peer / biographical questions MUST route to `get_entity_intelligence` (preferred over `get_entity_graph + search_documents`). When editing any of these descriptions, keep the cross-references symmetric — break one tool's anti-pattern list and the regression `test_tool_descriptions_contain_anti_patterns` (in `tests/unit/application/pipeline/test_tool_registry_builder.py`) will fail.
- **PLAN-0095 W3 intent map**: `_TOOL_TO_INTENT` in `application/services/intent_inference.py` now maps `get_entity_intelligence`, `search_entity_relations`, `get_entity_narrative` → `QueryIntent.RELATIONSHIP` so bundle/narrative tool selections trigger the relationship-specialised second-turn prompt addendum instead of the GENERAL fallback.
- **PLAN-0095 W3 `RAG_COMPLETION_CACHE_DISABLED` env**: set to `"true"` (case- and whitespace-tolerant) to short-circuit `ChatPipeline.check_cache` and force a cold-path execution. Intended for chat-eval sessions (auto-set by `tests/validation/chat_eval/conftest.py:_disable_completion_cache`). Production leaves it unset. Read per-call inside `check_cache()`, so the toggle takes effect without a service restart.
- **PLAN-0095 W2 T-W2-03 cache-before-classifier order**: `_execute_streaming_inner` in `chat_orchestrator.py` runs `check_cache()` BEFORE `validate_input()`. Cache HIT short-circuits the LLM injection classifier (5-8 s on DeepInfra). A poisoned cache entry was classifier-validated on its first write, so re-classifying every read is defensive duplication, not a real gate. Never invert this order without a paired update to the regression `test_chat_pipeline_cache_hit_skips_classifier` in `tests/unit/use_cases/test_chat_orchestrator_tool_use.py`.
- **PLAN-0095 W2 T-W2-02 `get_fundamentals_history_batch` tool**: the registry now has 23 tools (was 22). When comparing or screening MULTIPLE tickers, the LLM MUST prefer this over N×`get_fundamentals_history` — calling the singular variant in a fan-out is 5-10x slower because each turn is a separate agent round-trip. The tool description carries the exact "5-10x slower" string; do not weaken it.
- **PLAN-0097 T-W3-03 reciprocal anti-pattern callouts**: `get_fundamentals_history` description includes a hard "Do NOT call this in a loop — use `get_fundamentals_history_batch` instead" warning, and the batch tool leads with a strict imperative directive (`**Use this tool — NOT get_fundamentals_history — when the user mentions 2 or more tickers, OR …**`). Both clauses are pinned by `test_get_fundamentals_history_description_warns_against_loop` and `test_get_fundamentals_history_batch_description_is_strict_directive` in `tests/unit/application/pipeline/test_tool_registry_builder.py` — never soften either or the iter-9 multi-ticker misroute regression will recur.
- **PLAN-0100 W1 BP-604 / BP-605 entity-drift guards (`chat_orchestrator.py`)**: every iter ≥ 1 tool call is screened by `_validate_fallback_tool_call()` — entity-typed input fields (`entity_id/entity_ids/entity_name/entity_names/entity_ticker/entity_tickers`, pinned in `_ENTITY_TYPED_FIELDS`) whose values are NOT in (question entities + prior-turn entity inputs) are rejected as STRUCTURED `role="tool"` error messages so the LLM self-corrects without crashing the loop (never raised). Iter-0 is exempt — the first batch IS the surfacing event. After synthesis but before numeric-grounding validation, `_check_entity_grounding()` refuses with a fixed "I cannot find information…" string when ZERO retrieved-item entities overlap the question set; the refusal REPLACES `full_text` and sets `grounded=False` so it is never cached. Any new tool whose input carries an entity identifier MUST register the field name in `_ENTITY_TYPED_FIELDS` or the guard will silently miss it. Any new retrieval source MUST populate `RetrievedItem.entity_id` OR `citation_meta.entity_name`. Q2 MSTR canary regression: `tests/unit/use_cases/test_chat_orchestrator_entity_drift_guards.py`.
- **PLAN-0099 W1-T03 chat phase timings**: every `execute_streaming` run wraps its major pipeline phases (`check_cache`, `validate_input`, `load_history`, `entity_resolution`, `llm_tool_planning`, `tool_execution`, `llm_synthesis_streaming`, `grounding_validation`, `persist_and_cache`) using `rag_chat.application.observability.phase_timings.phase(...)` and emits a `chat_phase_timings_ms` structlog event at the end. The same dict is attached to the terminal SSE `done` event as `data.phase_timings_ms` so the chat-eval harness can scrape it from artifact frames. Agent-loop phases (`llm_tool_planning`, `tool_execution`) sum across iterations. Never bypass `PhaseTimings.record(...)`/`async with phase(...)` for a wrapped block — direct `time.monotonic()` math defeats the exception-still-records guarantee and creates silent drops.

## LLM Provider Fallback Chain
DeepInfra (primary, if `RAG_CHAT_DEEPINFRA_API_KEY` set) → OpenRouter (if `RAG_CHAT_OPENROUTER_API_KEY` set) → Ollama (emergency fallback)
Negative cache: `rag:v1:neg:{provider_name}` in Valkey, TTL 60s. A failed provider is skipped for 60s.
All providers unavailable → `ProviderUnavailableError` → 503 to client.

**Classification model**: `meta-llama/Meta-Llama-3.1-8B-Instruct-Turbo` via DeepInfra (`RAG_CHAT_DEEPINFRA_CLASSIFICATION_MODEL`); Ollama `qwen3:0.6b` fallback

**Completion model (OpenRouter fallback)**: `deepseek/deepseek-r1-distill-qwen-32b` (`RAG_CHAT_OPENROUTER_COMPLETION_MODEL`)
**Completion model (Ollama emergency)**: `deepseek-r1:32b` (`RAG_CHAT_OLLAMA_COMPLETION_MODEL`)
**Embedding model**: Jina AI embeddings-v3 (1024-dim) via `RAG_CHAT_JINA_API_KEY` when set; falls back to S6/Ollama bge-large (7-13s CPU) when absent
**Reranker**: Cohere Rerank v2 via `RAG_CHAT_COHERE_API_KEY` when set; falls back to fusion_score sort when absent

- `citations` — JSON array of Citation objects
- `contradictions` — JSON array of flagged contradiction pairs
- `metadata` — retrieval stats (items_retrieved, reranked_count, latency_ms)
- `error` — error details (closes stream)
- `pending_action` — emitted when a write-action tool proposes an action awaiting user confirmation (PLAN-0082)
- `action_executed` — emitted when a confirmed action is executed successfully (PLAN-0082)
- `action_rejected` — emitted when a user rejects an action proposal or execution fails (PLAN-0082)

## Critical Rules
- **R25**: API routes only import use case classes — never `infrastructure/` directly
- **R27**: Threads/messages reads use `ReadOnlyUnitOfWork`; writes use `UnitOfWork`
- **R26**: No auto-commit in `UnitOfWork.__aexit__` — explicit `await uow.commit()` in every use case
- **R22**: No `asyncio.create_task()` in app lifespan — ASGI only
- **Never** store raw LLM output in DB — always strip `<think>` / `<reasoning>` blocks first
- **Never** interpolate user input into prompt templates without XML-wrapping
- Rate limit enforced per `(tenant_id, user_id)` via Valkey sliding window
- All upstream service calls (S6, S7, S3, S1) use `httpx.AsyncClient` with timeouts
- Thread ownership: every thread read/write verifies `thread.tenant_id == request_tenant_id`

## Pitfalls
- `bleach` required as a dependency for HTML stripping in `InputValidator` — must be in pyproject.toml
- All `valkey` constructor parameters must be typed as `ValkeyClient` (from `messaging.valkey.client`), never `aioredis.Redis` or `redis.asyncio.Redis` — see BP-103b. New `ValkeyClient` methods must be staged in the same commit as the callers to avoid pre-commit hook `attr-defined` failures.
- `asyncio.gather` in parallel retrieval: use `return_exceptions=True` to prevent one failing source from aborting all retrieval
- SSE stream: if client disconnects, `asyncio.CancelledError` is raised — must clean up any open `httpx` connections
- `fusion_score` must be computed atomically — do not pass partial `RetrievedItem` without all 3 factor fields set
- Negative provider cache key format: `rag:v1:neg:{provider_name}` (not just `rag:neg:`)
- BGE reranker input format: `(query, passage)` pairs — NOT `(passage, query)` (model is asymmetric)
- `thread.tenant_id` ownership check MUST happen in the use case, not just in the route — defense in depth
- Contradiction detection: only inject warning block when S7 returns `is_contradicted=true` on a relation
- SSE per-chunk streaming (PLAN-0099 W1 / BP-595): the orchestrator's "LLM answered directly" branch slices `direct_text` via `_chunk_text_for_streaming()` (default 8 words/chunk) and yields one `emit_delta()` frame per chunk. `emit_delta` is a wire-compatible alias of `emit_token` (same `event: token`). DO NOT replace this loop with `emit_token(direct_text)` — that regression takes TPS back to ~0.087 tok/s and breaks the chat-eval TTFT gate. The concatenation of all chunks must equal `direct_text` exactly because grounding validation reads it back from the captured stream.
- `RAG_CHAT_CYPHER_ENABLED` is GONE — do not add it back; `traverse_graph` is always guarded by `_ALLOWED_CYPHER_REL_TYPES` frozenset in `tool_executor.py`
- `ToolExecutorFactory` must be wired in `app.py` lifespan — it is NOT auto-instantiated; forgetting causes `AttributeError` at first chat request
- `ToolUseBlock.id` (not `.tool_use_id`) — use `.id` when building `ToolCallProvenance`
- Morning brief uses lastgood fallback — never overwrite `briefing:morning:lastgood:*` on per-user failure (PLAN-0094 W2 contract).
- Benign relationship-discovery queries between named entities ("How is X connected to Y?", "Show me the relationship paths between A and B") MUST classify as SAFE — the classifier system prompt carries an explicit SAFE exemplar for this shape (PLAN-0097 W2 T-W2-01, BP-579). The regression gate is `tests/unit/security/test_llm_injection_classifier_benign_relationships.py`; bump `CLASSIFIER_PROMPT_VERSION` if you change `_SYSTEM_PROMPT` so downstream caches invalidate.
- `DEBUG_SKIP_CLASSIFIER=true` short-circuits `LLMInjectionClassifier.classify()` to return False — ONLY when `APP_ENV != "production"` (security gate, PLAN-0097 W2 T-W2-04). The chat-eval conftest sets both env-vars so eval runs do not pay the DeepInfra latency or flake on the L2 model's non-determinism.
- Morning-brief context gathering is observable via `brief_context_availability_score` (Histogram), `brief_upstream_latency_ms{source}` + `brief_upstream_status{source,outcome}` (Counter), and `brief_cache_outcome{cache_name,outcome}` (Counter) — PLAN-0099 Wave A. The `BriefingContext.context_availability_score` field carries the same value into downstream Wave B refusal-on-low-context (BP-599 / BP-600).
- Brief context truncation limits are env-overridable (PLAN-0099 Wave B): `RAG_CHAT_BRIEF_NEWS_LIMIT` (default 12, was 8), `_EVENTS_LIMIT` (default 10, was 6), `_ALERTS_LIMIT` (default 8, was 5). Bare-name forms (`BRIEF_NEWS_LIMIT` etc.) are also accepted. `_dedupe_news()` in `brief_context_formatter.py` runs BEFORE truncation — never bypass it on a code path that adds another news source, or syndicated copies will silently re-consume slots. `RAG_CHAT_BRIEF_MIN_CONTEXT_SCORE` (default 0.3) triggers refusal-on-low-context: scores below threshold skip the LLM and return a "Limited data available today" lead; set to 0.0 to disable.
- **PLAN-0100 W2 T-W2-01 pre-tool status badge**: `chat_orchestrator.py` emits ONE `emit_status(f"Loading {tool_summary}…")` event right after iteration-0's LLM picks tools and BEFORE the `tool_call` loop. This is the FIRST user-visible activity on tool-using questions; the frontend renders it as a badge above `ToolCallTray` (`MessageTurn.tsx` `initialStatus` prop) AND the chat-eval harness counts it toward TTFT via `_CONTENT_EVENT_KINDS` (`tests/validation/chat_eval/harness.py`). Removing or relocating this emit without also updating those two consumers will silently regress TTFT-p95 (pre-fix p95 = 69.7s because synthesis came after the tool loop). Regression gate: `tests/unit/use_cases/test_chat_orchestrator_tool_loop.py::test_orchestrator_emits_aggregate_status_badge_before_tool_calls`.
- **PLAN-0101 W3 `llm_synthesis_streaming` phase label is a public contract**: the chat-eval `tps_streaming` metric and its aggregate gate (≥ 20 tok/s p50) extract this exact key from `done.data.phase_timings_ms` (`tests/validation/chat_eval/harness.py::_compute_tps_streaming`). The `phases.record("llm_synthesis_streaming", …)` calls in `chat_orchestrator.py` (success + failure branches around the second-turn `stream_chat`) MUST keep this string verbatim. Renaming silently collapses the metric to NaN and disables the gate. See `docs/audits/2026-05-28-plan-0101-tps-metric-redesign.md`.
- `AgenticBriefGenerator` (`application/use_cases/agentic_brief_generator.py`) is EXPERIMENTAL and OFF by default (`RAG_CHAT_BRIEF_AGENTIC_ENABLED=false`). PLAN-0099 Wave C scaffolded the iterative tool-use loop for A/B comparison against the standard `execute_public_morning` single-turn path. The route layer branches in `api/routes/public_briefings.py`; ALL failures (exception / `budget_exhausted` / `empty_response`) fall back to the standard generator and increment `brief_agentic_fallback_total{reason}`. Do NOT enable in production without an explicit eval pass.

## Citation-Accuracy Cron (PLAN-0063 W5-5 T-W5-5-02 + PLAN-0084 A-1)
- **`application/use_cases/score_citation_accuracy.py`**: `ScoreCitationAccuracyUseCase` + `iter_cited_claims(msg)` + `LLMJudgePort` Protocol

  - `_next_sunday_03_utc()` handles same-Sunday edge case correctly
- **Wire-up** (`app.py`): `_wire_citation_cron()` function wires cron in lifespan step 8; gated by `RAG_CHAT_CITATION_CRON_ENABLED`; done-callback logs CRITICAL on crash (BP-268)

## Context Manager Notes (Wave A-3)
- TTL constants (`_CHUNK_TTL=4h`, `_SUMMARY_TTL=24h`) are defined in `context_manager.py` (application layer) — NOT imported from `valkey_chunk_cache.py`. Architecture boundary: application must not import from infrastructure at runtime.
