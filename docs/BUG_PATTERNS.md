## Parallel-session reverts — full taxonomy

PLAN-0107 D-5. Detail and recovery guide for BP-590 (parallel-session worktree corruption).

### Pattern 1: Bulk-commit
**Detection**: unexpected commit author + large file count.
**Recovery**: `git revert <sha>`; re-stage your work.
**Prevention**: `bash scripts/worktree_lock.sh acquire` before editing.

### Pattern 2: Branch-rewind
**Detection**: `git reflog` shows HEAD jump backward (e.g. `reset: moving to origin/main`).
**Recovery**: `bash scripts/orphan_commit_check.sh` to find orphaned SHAs; `git cherry-pick <sha>` to re-apply.
**Prevention**: lockfile + post-cherry-pick watchdog.

### Pattern 3: Selective file revert
**Detection**: `git log -p <file>` shows your change present then absent.
**Recovery**: re-apply diff; `git cherry-pick` from a worktree branch if available.
**Prevention**: lockfile blocks during edit window.

### Pattern 4: YAML/JSON revert
Same as #3 but on config files; often missed because tests still pass.
**Detection**: explicit diff against expected state in CI.

### BP-725: naive mean-of-observed-lifetimes underestimates half-life under right-censoring (2026-07-14)
**Symptom**: fitting a decay/survival rate from "how long until X happened" data by averaging only the durations that were ACTUALLY observed to terminate silently biases the rate estimate high (half-life estimate biased low) whenever a meaningful fraction of the population hasn't terminated yet by the observation cutoff.
**Root cause**: excluding still-alive (right-censored) observations from the average keeps only the SHORT, already-terminated lifetimes — the long-lived population is systematically underrepresented because it hasn't finished yet. This isn't a rounding error; it's a structural selection bias that gets worse as the censoring fraction grows.
**Fix (PLAN-0123/PRD-0120, `knowledge_graph.application.analytics.decay_fitting.supersession_estimator.fit_supersession`)**: use the standard right-censored exponential MLE — `lambda_hat = (observed terminal events) / (total exposure time, censored + uncensored)` — which correctly credits every still-alive observation with its full exposure duration instead of dropping it. `test_naive_mean_lifetime_underestimates_halflife` demonstrates the naive estimator is measurably worse on the same synthetic data with a known true rate.
**Prevention**: any time-to-event / survival-style rate estimation MUST use a censoring-aware estimator (Kaplan-Meier, exponential MLE with exposure, Cox, etc.) the moment some observations haven't terminated by the cutoff. A silent `mean(observed_durations)` is a correctness bug, not a simplification — flag it in review whenever "average lifetime/duration" appears near data that can still be "in progress".

### BP-726: inter-arrival-gap MLE estimates mention/event RATE, not the decay RATE a `exp(-alpha*age)` model consumes (2026-07-14)
**Symptom**: fitting a repeated-mention/repeated-event process's "decay" by taking the mean gap between consecutive observed mentions and treating `1/mean_gap` as the decay parameter produces a number that looks plausible but is measuring the wrong quantity — two processes with identical mean inter-arrival gaps can have wildly different TRUE relevance decay (e.g. "mentioned every 3 days forever" vs. "mentioned every 3 days for a month then never again").
**Root cause**: inter-arrival gaps are only defined BETWEEN observed events — they say nothing about the long silent tail after the last observed mention, which is exactly where decay signal lives. A steady process and a process that quietly stopped can produce statistically indistinguishable observed gaps while having opposite true half-lives.
**Fix (PLAN-0123/PRD-0120 review finding SS-1, `knowledge_graph.application.analytics.decay_fitting.nhpp_estimator.fit_nhpp`)**: model the mention timestamps as a non-homogeneous Poisson process with intensity `lambda(t) = lambda0 * exp(-alpha*t)` and fit `(lambda0, alpha)` by MLE over the FULL observation window (including the silent tail) — this directly estimates the decay parameter the confidence model consumes. `test_nhpp_rejects_inter_arrival_proxy` constructs two mention patterns with identical inter-arrival statistics but different true decay and shows the NHPP fit distinguishes them while the naive proxy cannot.
**Prevention**: when a PRD or spec proposes fitting a decay/relevance-half-life parameter from a repeated-mention process, check whether the proposed estimator uses ONLY observed inter-event gaps (proxy for RATE, wrong) or the full observation window including post-last-mention silence (correct, estimates DECAY). The two are easy to conflate because both look like "exponential MLE on the data."

**Symptom**: after the DeepInfra spend cap was hit, DeepInfra returned HTTP 402 `Payment Required`. 693 article extractions silently empty-committed (logged as a normal `deep_extraction.complete` with 0 events/0 claims/0 relations) and 2,383 chunk/section embeddings were abandoned in `embedding_pending` (`retry_count >= max_retries`, skipped by `claim_batch` forever → indefinitely unsearchable). Both losses were invisible: no error surfaced, and raising the cap did NOT self-heal the stuck work.
**Root cause**: the ml-clients adapters mapped EVERY non-429 4xx to `FatalError` (terminal). `deepinfra_embedding.py` did `if status >= 500: RetryableError; else: FatalError`; `deepseek_extraction._classify_transient` did the same. So HTTP 402 (a purely BILLING refusal that clears the instant the operator raises the cap) was treated as an unrecoverable bad request. In extraction, `deep_extraction.run_deep_extraction_block` catches `RetryableError` (→ re-queue) but its generic `except Exception` empty-commits — a `FatalError` fell into that branch and became a fake empty result. In embedding, `EmbeddingRetryWorker` catches `FatalError` → `mark_abandoned` immediately. Even a bounded transient retry would not self-heal a cap-down lasting longer than the ~2 h backoff schedule.
**Fix (`fix/deepinfra-402-reraise`)**: split 4xx into three buckets in `ml_clients.errors` (`is_billing_status` {401,402,403}, `is_transient_status` {408,409,425,429,≥500}) and add `ProviderBillingError(RetryableError)`. Both adapters now raise `ProviderBillingError` on 401/402/403, `RateLimitError` on 429, `RetryableError` on other transient, `FatalError` only on genuine bad input (400/404/413/422 — e.g. the 513>512 BGE token overflow). Because it is a `RetryableError`, extraction's existing `except RetryableError` re-queues the doc (Kafka redelivery) instead of empty-committing. The embedding retry worker gets a dedicated `except ProviderBillingError` branch that backs off at a fixed cadence WITHOUT incrementing `retry_count` (`mark_failure(increment_retry=False)`) so a cap-down never abandons the row — it self-heals the moment the cap is raised. Rows abandoned before the fix are recovered idempotently via `EmbeddingPendingRepository.requeue_abandoned` / `python -m nlp_pipeline.workers.requeue_abandoned_embeddings` (reset `retry_count=0, next_retry_at=now, created_at=now`, throttled, dry-run by default). Recovery on 2026-07-18: cap confirmed raised (HTTP 200 embeddings flowing, zero 402 in 2 h), 2,383 rows re-queued and drained.
**Observability + escalation (review follow-up)**: because a billing deferral does NOT consume the retry budget, a PERMANENT auth failure (revoked/misconfigured key returning 401/403) would loop forever, invisibly (structlog line only). Added: (1) `s6_embedding_retry_billing_deferred_total` counter on every deferral (alertable on sustained cap/auth pressure) and `s6_embedding_retry_abandoned_total{reason}` on every permanent abandon (`fatal_4xx` | `billing_auth_persistent` | `retry_exhausted`); (2) a max-age escalation — once a row has been unembeddable longer than `embedding_billing_defer_max_age_s` (default 24 h, since first enqueue), a further billing/auth refusal ABANDONS it with the `billing_auth_persistent` signal + metric so a genuinely stuck key surfaces instead of hiding. A real 402 spend-cap self-heals well before that ceiling; `requeue_abandoned` resets `created_at` so re-queued rows get a fresh age clock. **Prevention**: classify provider HTTP failures by RECOVERABILITY, not just the 4xx/5xx boundary. A billing/quota/auth refusal is transient-from-the-app's-POV (an operator action clears it) — it must be retryable, and any bounded retry budget must NOT be consumed on it or a long outage dead-ends all in-flight work. Never let a generic `except Exception` substitute an empty-but-successful result for a failed external call; distinguish "provider said nothing found" from "provider refused to answer". Any queue with a max-retries-then-abandon policy needs an idempotent re-queue path for operator recovery.

### BP-733: GLiNER native-process memory OOM via two independent mechanisms — unbounded per-forward-pass activation size and glibc arena fragmentation (2026-07-21/22)
**Symptom**: the GLiNER NER pod OOMKilled repeatedly under sustained load — one incident showed ~1 OOMKill/day under a backfill burst of long sections even with the batch-count cap (`GLINER_MAX_BATCH`) in place; a separate, worse incident showed ~2.7 OOMKills/hour (33 restarts/12h, `dmesg` showing `pt_main_thread` killed at anon-RSS ~13GB against a 16Gi cap) with a flat ~2.4Gi working set — i.e. the process was fragmenting, not leaking or spiking.
**Root cause**: two distinct memory-growth mechanisms share the same "OOMKilled crashloop under load" symptom and must not be conflated. (1) **Activation-size scaling**: `batch_predict_entities` pads every text in a batch to the LONGEST text's length, and GLiNER's span-enumeration tensors scale as `batch_size × padded_seq_len × max_span_width` — the count-only `GLINER_MAX_BATCH` cap bounds how many texts go in a batch but not how LONG they are, so a batch of long sections can still spike peak RSS past the container limit. (2) **glibc arena fragmentation**: GLiNER's multi-threaded CPU inference uses glibc's default per-thread malloc-arena behavior; freed tensor blocks accumulate across many small forward passes in per-arena free lists that are never returned to the OS, so anon-RSS ratchets upward over time even without any single large batch (leak-LIKE behavior with no actual Python-level leak).
**Fix**: (1) `5b7cf8b2c` adds `GLINER_MAX_BATCH_CHARS`, an adaptive budget on the padded-activation proxy (`batch_size × longest-text-chars`) — the collector shrinks a batch when its texts are long and keeps it full when they're short, bounding peak memory without sacrificing throughput on normal-length sections. (2) `8e9f5c6f6` bounds the fragmentation ramp via `MALLOC_ARENA_MAX=2` baked into the Docker image (glibc reads it pre-init, so it must be an image `ENV`, not set from Python — requires a rebuild), a single-thread inference executor (one OS thread = one arena), `malloc_trim(0)` after every forward pass to return freed pages to the OS, a `GLINER_MAX_INPUT_CHARS` hard cap, `torch.inference_mode()` to avoid autograd retention, and an optional worker-recycle backstop.
**Prevention**: this is native process memory management, not a DB planner/pooling issue (distinct from the BP-730/BP-732 Postgres-side OOM cluster) — an on-call triaging a GLiNER crashloop should check `dmesg`/RSS trend shape first: a flat-working-set ratchet under load points to arena fragmentation (`MALLOC_ARENA_MAX`, single-thread executor, `malloc_trim`), while a spiky peak correlated with a burst of long inputs points to activation-size scaling (an adaptive char/token budget, not just a batch-count cap). Any CPU-bound multi-threaded native-tensor inference service should set `MALLOC_ARENA_MAX` in its image by default rather than discovering the fragmentation ramp under a load incident.

### BP-739: Polymarket Gamma pagination — "short page ⇒ last page" heuristic breaks when the provider silently caps page size below the requested limit; duplicated client boilerplate meant the same wrong assumption was coded twice and the fix had to be applied twice (2026-07-16)
**Symptom**: the Polymarket Gamma `/markets` and `/events` offset-paginated clients each stopped after roughly their first page — the market universe stayed at ~101 rows and long-tail-but-active event groups (2028 Presidential, GTA VI, WC 2026) were never fetched, leaving their member markets with `event_id=NULL` — even after a first fix (`e1745828b`) had already replaced a broken `next_cursor`-based loop (Gamma never returns `next_cursor`; it's plain offset/limit over a bare JSON array) with proper offset advancing.
**Root cause**: the first fix's termination condition — "stop when a page returns fewer than `limit` items" — is itself unsafe: live verification (2026-07-16) showed Gamma silently CAPS its page size at ~100 rows regardless of the requested `limit` (`limit=500` → 100 rows returned). So with a `page_size=500` default, the very FIRST page already looks "short" (100 < 500) and the loop concluded "done" after ~100 rows — reproducing the exact ~101-market bug the first fix set out to solve. This wrong assumption ("provider honors the requested page size, so a short page means end-of-data") was never verified against LIVE behavior, only against the provider's documentation; and because `polymarket/client.py` and `polymarket_gamma_events/client.py` are near byte-for-byte duplicated (`_cursor_to_offset`, `_extract_items`, offset-advance/termination logic all copy-pasted, not shared), the SAME wrong assumption was independently coded in both files and had to be fixed identically in both.
**Fix (`4b094c53e`)**: advance the synthetic pagination cursor by the ACTUAL returned row count, not the requested limit (`next_offset = str(offset + len(rows)) if rows else None`), and terminate ONLY on a genuinely EMPTY page — robust to any server-side page-size cap. `page_size` default lowered 500→100 to match the real cap (an honest per-cycle bound). Both fixes (`e1745828b`, `4b094c53e`) were applied to both duplicated clients. Verification was informal: the commit message cites a manual read-only "Live smoke" check as the only way the wrong page-size contract was caught — no automated live-contract or fixture-replay test exercises the real page-size behavior, and the unit test suite (`test_polymarket_client.py`) mocks pages that always honor the requested limit, i.e. it encodes the ASSUMED (wrong) contract rather than the corrected live one.
**Prevention**: (1) never terminate offset/cursor pagination on "fewer rows than requested" alone — terminate ONLY on an empty page; a provider is free to cap page size below your requested limit without saying so. (2) any new paginated external-API client must LIVE-verify (not just doc-verify) the page-size contract before shipping, and a regression test should assert the client advances correctly when the live response is shorter than the requested limit but still non-empty. (3) as of 2026-07-23, `polymarket_data_trades/client.py` (line ~94, `has_more = len(trades) >= limit`) still carries the EXACT disproven `has_more`-on-requested-limit heuristic against the Data-API `/trades` endpoint and has NOT been verified/fixed — treat it as an open instance of this bug pending live verification. (4) extract one shared pagination/cursor helper for the Polymarket client family (`polymarket`, `polymarket_gamma_events`, `polymarket_clob`, `polymarket_data_trades`, `polymarket_data_oi`) instead of copy-pasting cursor logic per endpoint — a proven fix (like this one) should propagate to all siblings automatically instead of needing to be independently rediscovered per client.
