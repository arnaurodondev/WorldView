# PLAN-0101 — BP-610 `last_fundamentals_ingest_at` Touch UPDATE Silent Drop

**Date**: 2026-05-28
**Plan**: PLAN-0101
**Service**: market-data
**Status**: FIXED (commit `ca9e4dc7`)
**BP**: BP-610 (Data Pipeline, ORM)

---

## Pain Point

`instruments.last_fundamentals_ingest_at` stayed **NULL for ALL 629
instruments** platform-wide, even though `FundamentalsConsumer.process_message`
was correctly awaiting:

```python
await uow.instruments.touch_fundamentals_ingest_at(instrument_id, ingested_at)
```

at `services/market-data/src/market_data/infrastructure/messaging/consumers/fundamentals_consumer.py:589`.

Every other write in the same UoW persisted as expected: the per-section
fundamentals rows landed, the `instrument_fundamentals_snapshot` row was
upserted, structlog reported both `materialized` and `snapshot_upserted`
on every cycle. Yet the freshness column never advanced. PLAN-0100 W4 had
just flipped `FUNDAMENTALS_REFRESH_ENABLED` from `False` to `True`, so the
worker started fanning out — but the operability column it depended on
for "what is stale?" decisions remained zeroed out, masking the rollout's
actual coverage.

---

## Root Cause

Two compounding problems:

1. **Docker image staleness.** The running container's installed
   `fundamentals_consumer.py` (`/app/.venv/.../fundamentals_consumer.py`)
   had **zero references** to `touch_fundamentals_ingest_at`. The PLAN-0096
   T-W1-02 wiring (commit `8450666b`) had landed in source but the
   `market-data-fundamentals-consumer` image had not been rebuilt since.
   This was the immediate cause of the live NULLs.

2. **Latent buffer-and-drop in the repo method.** Even after the rebuild,
   the SQLAlchemy Core `update()` statement inside
   `PgInstrumentRepository.touch_fundamentals_ingest_at` was being
   executed against the async session but **not flushed**. SQLAlchemy
   buffers Core statements until the next flush/commit. The consumer
   wrapped its downstream best-effort writes in `try/except`; if any of
   those raised after the touch UPDATE was issued, the implicit rollback
   would silently discard the buffered UPDATE while still committing the
   ones that flushed earlier. Every other repo in the codebase
   (`content-store`, `alert`, `rag-chat`, `content-ingestion`) flushes
   inside the repo write — this one did not.

Investigation eliminated:
- Migration 021 (the column-add) applied. ✓
- No PG triggers on `instruments`. ✓
- Zero `failed_tasks` / dead-letter rows. ✓
- `instruments.updated_at` ran **before** the consumer's `materialized`
  log timestamp on AMD/MSFT/NVDA — proof that the touch UPDATE never hit
  the database (otherwise SQLAlchemy's Core `update()` would have
  re-fired `updated_at`). ✓

## Fix Sketch

Single-line defensive addition inside the repo method:

```python
# services/market-data/src/market_data/infrastructure/db/repositories/instrument_repo.py
async def touch_fundamentals_ingest_at(self, instrument_id: UUID, ingested_at: datetime) -> None:
    stmt = (
        update(InstrumentModel)
        .where(InstrumentModel.id == instrument_id)
        .values(last_fundamentals_ingest_at=ingested_at)
    )
    await self._session.execute(stmt)
    await self._session.flush()  # ← NEW: guarantee the UPDATE reaches the txn
```

This matches the repo-wide convention; the consumer-layer commit boundary
is no longer the sole guarantor that the freshness write persists. Any
later try/except-swallowed exception in the same UoW can no longer mask
the touch.

**Image rebuild step** (required for live activation):

```bash
docker compose build market-data
docker compose up -d --no-deps --force-recreate market-data-fundamentals-consumer
```

## Detection / Prevention Notes

**Detection pattern** for "observational" tracking columns: any column
that should monotonically advance every cycle and is written through a
single repo call wrapped in a UoW that contains a best-effort `try/except`
block downstream. Cross-check by joining the column against the operation
log timestamp — if the column is NULL on rows that successfully ran the
surrounding writes, the buffer-and-drop pattern is in play.

**Prevention rule** (added to `services/market-data/.claude-context.md`):
every repo write method that targets a tracking/freshness column MUST
end in `await self._session.flush()`. Callers (consumers, workers, use
cases) must not be expected to know that the repo's write is unflushed.
