## Issues — Detail

### Issue F-001 (Data Platform): Legacy `source_id` keys in cache blobs no longer normalized on read

**Severity**: MINOR
**Confidence**: MEDIUM
**Flagged by**: Data Platform Engineer
**File**: `services/rag-chat/src/rag_chat/api/schemas.py:185-188`, `services/rag-chat/src/rag_chat/api/routes/public_briefings.py:111`

**Summary**: Pre-refactor, Pydantic's `populate_by_name=True` on `BriefCitation` meant a cached blob containing `{"source_id": "x", ...}` would be re-validated on `model_validate_json` and re-emitted with `document_id`. Post-refactor, `sections` is `list[dict[str, Any]]` and the field_validator does NOT recurse into bullets/citations — legacy `source_id` keys pass through verbatim to the wire response.

**Root Cause**: The migration moved `populate_by_name` semantics into `BriefCitation.from_dict`, but the cache-read path uses `model_validate_json` (Pydantic) which goes through `_coerce_sections_to_dicts` (dict-passthrough), never invoking `from_dict`.

**Impact**:
- Immediate: zero (no code currently writes `source_id` to cache — that field name disappeared at PLAN-0062-W4)
- Blast radius: bounded to cache TTL (`_CACHE_TTL`, minutes-to-hours)
- Data risk: zero
- User impact: zero in practice; theoretical only if a pre-W4 service writes a cache entry that survives long enough to be read by post-W4 code

**Verification (Distributed Systems agent)**: cache key prefix `briefing:morning:v2:` / `briefing:instrument:v2:` from PLAN-0062-W4 partitions away pre-W4 blobs. NEW code never writes `source_id`. OLD-pre-W4 code wrote to non-`:v2:` keys. Therefore in-flight `:v2:` blobs are guaranteed to use `document_id` only.

**Resolution**: ACCEPT as-is. No cache-key bump required.

### Issue F-002 (QA/Test): `BriefSection.title max_length=120` lost test coverage

**Severity**: MINOR
**Confidence**: HIGH
**Flagged by**: QA/Test Engineer
**File**: `services/rag-chat/src/rag_chat/domain/brief.py:175` / `tests/unit/api/test_schemas_brief.py`

**Summary**: The Pydantic→dataclass migration ported `Field(..., max_length=120)` to a `__post_init__` `ValueError` raise, but the test file dropped the constraint test — `BriefSection.title` length had ZERO assertions before this QA pass.

**Root Cause**: `Field`-driven constraints were tested implicitly via `pytest.raises(ValidationError)`. When the test file was migrated to `pytest.raises(ValueError)`, the `title` constraint case was missing from the original Pydantic test set and so was not propagated.

**Fix Applied**: Added 2 tests:
- `test_section_title_at_exactly_120_chars_accepted` — boundary acceptance
- `test_section_title_above_120_chars_rejected` — boundary rejection (`ValueError`)

Plus 1 bonus boundary test for `BriefSection.bullets`:
- `test_bullets_at_exactly_8_accepted` — pins the upper boundary against off-by-one tightening

**Verification**: 38/38 targeted tests PASS; 724/724 full suite PASS.

### Issue F-007 (QA/Test): Cache round-trip used leaf-by-leaf assertions, not deep-equality

**Severity**: MINOR
**Confidence**: MEDIUM
**Flagged by**: QA/Test Engineer
**File**: `services/rag-chat/tests/contract/test_brief_contract.py:308-311`

**Summary**: The cache JSON round-trip test (`test_public_briefing_response_json_round_trip_preserves_nested_brief_data`, added by the implementation fix subagent) walked individual nested keys with positional indexing (`cits[0]["document_id"] == "doc-1"`) but never asserted full structural equality. A future `to_dict()` change that silently *adds* a key would not be caught.

**Fix Applied**: Appended a deep-equality assertion `rehydrated.sections == [section.to_dict()]` so any structural drift trips the test.

**Verification**: targeted test passes after the strengthening; if `to_dict()` ever emits an extra key without updating the test, the assertion will now flag it.
