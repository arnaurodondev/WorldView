# QA Report — PLAN-0053 Iteration 1

**Date**: 2026-04-30
**Scope**: PLAN-0053 (8 waves, ~50 tasks across 7 squashed merge commits)
**Reviewer**: strict QA validator (final quality gate)
**Verdict**: **NEEDS_FIXES**

The implementation team shipped an enormous amount of high-quality work in 7
parallel-worktree commits. Test suites all pass at the claimed numbers (849
frontend Vitest, 417 alert, 644 portfolio, 548 market-data, 587 content-ingestion,
203 market-ingestion). TypeScript is clean. Architecture compliance (R25/R27)
is preserved on the new endpoints. Empty / loading / error states are present
on most new widgets.

However three findings block a clean ship:

1. **BLOCKING** — market-data migration `012_add_instruments_name_trgm_index.py`
   uses `CREATE INDEX CONCURRENTLY` inside Alembic's default transactional
   wrapper. Postgres rejects `CONCURRENTLY` inside any transaction, so the
   migration WILL fail on every deploy. This breaks Wave B's name-search
   fix end-to-end and prevents migration `013` (Wave C source columns) from
   running at all.

2. **CRITICAL** — alert migration `0008_backfill_alert_titles.py`'s WHERE
   clause does not match the strings the OLD code actually emitted. Old
   `_compose_alert_title()` produced `"Graph Change alert"`, `"Contradiction
   alert"`, `"Signal alert"` (lowercase 'a' in 'alert'), but the migration
   tries to match `"Graph Change Alert"`, `"Contradiction Alert"`, `"Signal
   Alert"` (capital 'A'). The regex catches only `"Graph Change alert"`. Net
   result: every legacy `CONTRADICTION` and SIGNAL-fallback row keeps its
   broken title.

3. **CRITICAL** — `ScreenshotCapture` (Wave G) captures the screenshot but
   discards it — the data URL is never sent to the backend. Per the comment
   in the file the modal sends only `screenshot_data_uri_present: true` in a
   metadata blob. Users believe they're attaching a screenshot to bug
   reports; they are not. This is a silent feature shortfall, not a typed
   error.

Plus 9 MAJOR / MINOR / NIT findings below.

## Findings

### F-001 (BLOCKING)
**Task**: T-B-2-01 (S3 instrument name search)
**File**: `services/market-data/alembic/versions/012_add_instruments_name_trgm_index.py:47`, `services/market-data/alembic/env.py:38-41`
**Issue**: The migration uses `CREATE INDEX CONCURRENTLY`, which Postgres explicitly forbids inside any transaction block (`ERROR: CREATE INDEX CONCURRENTLY cannot run inside a transaction block`). Yet `alembic/env.py` wraps every migration in `context.begin_transaction()`. The migration's own comment (line 46) acknowledges this risk — "we also disable autocommit for this migration in env.py if needed" — but no such code was added.

The deploy path will fail with:
```
psycopg2.errors.ActiveSqlTransaction: CREATE INDEX CONCURRENTLY cannot run
inside a transaction block
```

This breaks the entire Wave B fix (name-search relies on this index for the <100ms p95 acceptance criterion) AND chains a failure to migration 013 (Wave C source columns), since 013 declares `down_revision = "012"`.

**Evidence**:
```python
# services/market-data/alembic/env.py:38-41
def do_run_migrations(connection):
    context.configure(connection=connection, target_metadata=target_metadata)
    with context.begin_transaction():     # <— wraps EVERY migration
        context.run_migrations()
```
```python
# 012_add_instruments_name_trgm_index.py:47
op.execute(
    "CREATE INDEX CONCURRENTLY IF NOT EXISTS ix_instruments_name_trgm ..."
)
```

**Fix**: Either
(a) drop `CONCURRENTLY` and accept the brief table lock (the column is small, ILIKE is rare, lock is sub-second on a fresh dev DB), or
(b) declare the migration as autocommit by setting `transaction_per_migration = True` + per-revision `op.get_context().autocommit_block()` wrapper, or
(c) gate `CONCURRENTLY` behind a custom Alembic env-level check that detects this revision id and skips `begin_transaction()`.

For a thesis-grade demo, option (a) is simplest and safe.

### F-002 (CRITICAL)
**Task**: T-A-1-07 (alert title backfill)
**File**: `services/alert/alembic/versions/0008_backfill_alert_titles.py:78-83`
**Issue**: The WHERE clause does not match the string the OLD `_compose_alert_title()` actually produced. Verified by inspecting the pre-Wave-A code:
```python
# pre-Wave-A fallback (now removed):
raw = str(alert_type).replace("_", " ").strip()
return f"{raw.title()} alert" if raw else "Alert"
```
Python evaluation:
```
'GRAPH_CHANGE'  → 'GRAPH CHANGE' → 'Graph Change' → "Graph Change alert"
'CONTRADICTION' → 'CONTRADICTION' → 'Contradiction' → "Contradiction alert"
'SIGNAL'        → 'SIGNAL' → 'Signal' → "Signal alert"
```
All end with lowercase `' alert'`, not capitalised `' Alert'`.

The migration matches:
```sql
WHERE title IS NULL
   OR title = 'Graph Change Alert'   -- never produced by old code
   OR title = 'Contradiction Alert'  -- never produced
   OR title = 'Signal Alert'         -- never produced
   OR title ~ '^[A-Z][a-z]+ [A-Z][a-z]+ alert$'  -- only catches 2-word titles
```

Verified empirically:
```
'Graph Change alert'   → regex match ✓ (so this row DOES get fixed)
'Contradiction alert'  → regex match ✗ (only ONE word before "alert")
'Signal alert'         → regex match ✗
```

**Net behaviour**: The migration backfills exactly one of the three categories of bad titles (graph-change). Every `CONTRADICTION alert` and SIGNAL-fallback `Signal alert` row stays broken — defeating T-A-1-07's stated purpose and leaving the user-visible "Contradiction Alert" complaint partially unresolved on existing data.

**Fix**: Replace the literal CHECK strings (the `'Xxx Alert'` ones) with the real lowercase-`a` ones:
```sql
WHERE title IS NULL
   OR title IN ('Graph Change alert', 'Contradiction alert', 'Signal alert')
   OR title ~ '^[A-Z][a-z]+( [A-Z][a-z]+)? alert$'   -- 1-or-2 capitalised words + ' alert'
```
Also add an integration test under `services/alert/tests/integration/test_migrations.py` (called out in the plan but never created) seeding rows with each of the three legacy strings and asserting the backfill rewrites them.

### F-003 (CRITICAL)
**Task**: T-G-7-03 (ScreenshotCapture component)
**File**: `apps/worldview-web/components/feedback/ScreenshotCapture.tsx:14-23` (comment), `apps/worldview-web/components/feedback/FeedbackModal.tsx:196-205`
**Issue**: The ScreenshotCapture component runs `html2canvas`, generates a base64 data URI, displays it in a preview, and forwards it to the parent via `onCapture(dataUrl)`. **But the parent (FeedbackModal) discards it.** The modal sends only a metadata flag (`screenshot_data_uri_present: true`) in the submission payload — the actual image bytes never reach S9 / S1.

This is intentional per the comment ("BACKEND currently rejects non-https URLs") but the user has no idea — they spent 5 seconds capturing + maybe blurring the screenshot, see the preview, expect it to attach to their bug report. It does not. Cause-and-effect breakage between user mental model and product behaviour is the worst class of UX bug — it silently destroys trust.

**Evidence**:
```ts
// FeedbackModal.tsx ~line 196-205 — what actually gets sent
const meta = {
  ...
  screenshot_data_uri_present: screenshotDataUrl !== null,
  // screenshot_url is NOT set
};
```

The plan specified (T-G-7-03):
> ScreenshotCapture.tsx (html2canvas + blur tool + S3 upload)

The S3-upload half was never implemented; instead a comment was left.

**Fix**: One of these three options before claiming Wave G done:
1. Implement the presigned-URL upload path (preferred — backend already has S3 upload routes).
2. Compress the data URI and POST it inline as a base64 field on the submission (acceptable for the thesis demo; ~150 KB max).
3. Hide the ScreenshotCapture surface entirely until the upload route ships, replacing the buttons with a "Screenshot upload coming soon" disclosure. **At minimum surface this disclaimer in the UI today** — silent shortfall is unacceptable.

### F-007 (MAJOR)
**Task**: T-B-2-04 (CashManagementCard)
**File**: `apps/worldview-web/components/portfolio/CashManagementCard.tsx:91-92`
**Issue**: Operator-precedence trap. The line:
```ts
const total = data.invested + data.cash || 1;
```
parses as `(data.invested + data.cash) || 1`. If `data.cash` is `undefined` or `null` (which the backend type does not explicitly forbid — `Exposure.cash` is typed `number` but a partial response or older S1 build might omit it), then `data.invested + undefined === NaN`, and `NaN || 1 === 1`. Then:
```ts
const cashPct = (data.cash / total) * 100;  // (undefined / 1) * 100 === NaN
```
Renders `NaN%` to the user.

**Fix**:
```ts
const cash = data.cash ?? 0;
const invested = data.invested ?? 0;
const total = (cash + invested) || 1;
const cashPct = (cash / total) * 100;
```

## Verdict rationale

The work shipped is broad, well-commented (heavy WHY-style comments throughout — the team clearly absorbed the user feedback on comment density), test-backed (3,248 tests pass), and architecturally sound. **Most of the audit's 36 findings are now closed.** That is the headline.

However, two of the three flagship Wave A acceptance gates have a code-level defect that prevents them from delivering their stated value to existing users:

1. The migration that promises to clean up legacy alert titles will silently no-op on most rows (F-002). New alerts get good titles; legacy alerts stay broken. The user complaint that started this fix is only half-resolved.

2. The pg_trgm index that promises sub-100ms `?q=apple` will fail to apply at all in any environment running Alembic against a real Postgres (F-001). The fallback is a sequential scan; the SQL itself works (the `name.ilike()` clause is correct), so the feature is functionally correct but performance-poor — and the index migration is a deploy footgun waiting for the next CI cycle.

Plus a third silent shortfall — the Wave G screenshot capture is theatre; no bytes leave the browser (F-003).

These three together make the iteration NEEDS_FIXES rather than SHIP. They are all small fixes (one SQL pattern fix, one Alembic transactional-block disable, one decision about how to handle screenshot bytes). I expect a clean SHIP after one more iteration that closes F-001 / F-002 / F-003 and addresses F-005 / F-007 / F-008.

The remaining MAJOR/MINOR/NIT findings are polish and can land in a single follow-up commit alongside the three CRITICAL fixes.

**Recommended next action**: implement F-001, F-002, F-003 fixes; add the missing migration integration test from F-006; close F-005 by either extending the backend or replacing the chart; close F-007 with the nullish-coalescing fix; clean up F-008 lint. Re-run this audit to confirm SHIP.
