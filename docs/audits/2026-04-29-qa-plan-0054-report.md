## MAJOR Issues (outstanding — acceptable for current phase)

### M-001: Synthetic monitor has no retry on Pushgateway push failure
- **File**: `infra/synthetic/synthetic_monitor.py:95-99`
- **Issue**: If Pushgateway is unavailable, the error is logged but probes run without their results being recorded. Prometheus would see no data point, potentially causing false `SyntheticProbeDown` alerts.
- **Severity decision**: Acceptable for dev/thesis environment. Pushgateway and synthetic monitor run on the same Docker network; sustained unavailability would also cause the SLO recording rules to fail visibly.
- **Suggested follow-up**: Add exponential backoff with 3 retries, or a startup health check against Pushgateway.

### M-002: ML adapter metric recording exceptions not guarded
- **File**: `libs/ml-clients/src/ml_clients/adapters/*.py` (11 files)
- **Issue**: If `self._metrics` is set but a Prometheus registry operation raises (e.g., `labels().observe()`), the exception propagates to the adapter caller rather than degrading gracefully.
- **Severity decision**: In practice this cannot occur without a catastrophic registry failure. Acceptable given metrics=None default.
- **Suggested follow-up**: Wrap metric recording in try/except in `MLMetrics`-calling code.

## MINOR Issues (outstanding)

### N-001: SLO p95 recording rule lacks endpoint breakdown
- **File**: `infra/prometheus/rules/slo-recording-rules.yml:17-22`
- The `slo:market_data:p95_latency_5m` rule aggregates across all endpoints. A slow internal endpoint could mask a fast external one.
- **Suggested follow-up**: Add a `by (handler)` dimension to the p95 recording rule.

### N-002: Re-export shims have no deprecation warning
- **Files**: `services/rag-chat/src/rag_chat/infrastructure/metrics/prometheus.py`, `services/market-ingestion/src/market_ingestion/infrastructure/metrics/eodhd.py`
- Future contributors may add new imports at the old infrastructure path without realizing they should use application.metrics instead.
- **Suggested follow-up**: Add a Python `DeprecationWarning` or a `warnings.warn()` call on import.

### N-003: Alertmanager `${SLACK_WEBHOOK_URL}` expansion not verified in CI
- **File**: `infra/alertmanager/alertmanager.yml:43`
- Alertmanager v0.25+ supports native `${VAR}` expansion, which should work. But there is no CI smoke test verifying the config parses successfully with the env var.
- **Suggested follow-up**: Add `alertmanager --config.check` in a CI job or Makefile target.
