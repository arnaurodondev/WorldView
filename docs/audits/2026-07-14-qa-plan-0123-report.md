## Executive Summary

PLAN-0123 (PRD-0120) implements an offline decay-rate fitter for the knowledge-graph service plus a 5-column schema migration. Five specialist agents (QA/test, security, data-platform, distributed-systems, architecture) reviewed the 35 changed files independently. No BLOCKING or CRITICAL findings. The statistical core is genuinely sound — all three prior-review methodology concerns (SS-1 identification error, SS-3 competing risks, P-4 censoring) were independently re-verified as correctly fixed in code, not just claimed in docs, and `confidence.py` has zero diff across all 4 waves as required.

The most important cross-agent signal (independently flagged by 4 of 5 agents) is that **the write path is not wired end-to-end**: `pool_type_fit`, `write_back_fit`, and `backfill_relations_for_type` have no caller outside tests. This was already disclosed as a known gap in the plan's own Wave 4 notes, but the review surfaced two consequences worth fixing now regardless of when orchestration lands: (1) every shadow-report `DecayFit` was being stamped with a single hardcoded MEDIUM prior instead of each type's real class — fixed in this pass; (2) the FR-5 per-entity normalization helper is never invoked by the fit path — left as a follow-up decision since it requires new extraction logic. 6 concrete bugs were fixed directly (wrong prior, missing metric, unindexed-scan-adjacent idempotency gap, dead code, false CLI-invocation claim in a docstring, 2 missing test cases); one schema-level performance item (no index on `relations.canonical_type`) and two design decisions (orchestration wiring, FR-5 wiring) are left for the user.

All 137 knowledge-graph unit tests related to this plan pass (1872 total suite, 0 regressions), ruff/mypy/import-guards/architecture tests are clean, and the 5 new/changed SQL statements were verified via a live PREPARE pass against the actual deployed `intelligence_db` (non-destructive — PREPARE+DEALLOCATE only, no execution) rather than mocks alone.

## Issues — Full Investigation

## Issue F-001: Decay-fitter write path (pooling → write-back → backfill → metrics) has no orchestrator

### Summary
`pool_type_fit`, `write_back_fit`, `backfill_relations_for_type`, and `record_decay_fit_metrics`/`record_type_counts` are fully implemented and unit-tested in isolation, but nothing in `src/` calls them — `run_fitter.run_shadow_fit` only runs the Wave-2 read/estimate half of the pipeline. No fitted alpha can currently reach `relation_type_registry` or `relations` through any code path that actually runs.

### Severity / Confidence
**Severity**: MAJOR (per-agent) — elevated to a **Decision Needed** item rather than auto-fixed, because closing it means designing a transaction boundary and a locking strategy, not applying a patch.
**Confidence**: HIGH
**Flagged by**: qa-agent-1 (QA/Test, F-002), qa-agent-4 (Distributed Systems, F-001/F-002/F-003), qa-agent-5 (Architecture, F-001) — 3 of 5 agents independently, a 4th (data-platform) noted it as a side-finding.

### Root Cause Analysis
- **What**: `services/knowledge-graph/src/knowledge_graph/application/analytics/decay_fitting/run_fitter.py:74-119` (`run_shadow_fit`) calls only `extract_mention_series`/`extract_supersession_lifetimes` + `fit_nhpp`/`fit_supersession`. `pool_type_fit`, `write_back_fit`, `backfill_relations_for_type` are never imported by `run_fitter.py` or any other non-test file.
- **Why**: This was a deliberate wave-boundary design choice — Wave 2 built the shadow/estimation half, Wave 3 built the pooling/write-back/backfill half as independently-testable units, and no wave was scoped to build the orchestrator that chains them (the plan's Wave 4 notes explicitly flag this as a "known gap").
- **When**: Always — this isn't a conditional bug, it's an absent code path.
- **Where**: Application layer, `decay_fitting` package — a missing composition function, not a layering violation (each existing piece is correctly layered).
- **History**: Newly introduced by this plan (PLAN-0123); not a regression of prior behavior since the feature didn't exist before.

### Evidence
```
$ grep -rn "write_back_fit\|backfill_relations_for_type\|pool_type_fit" services/knowledge-graph/src/
# (only definitions inside the decay_fitting package itself; zero calls from run_fitter.py or elsewhere)
```

### Impact
- **Immediate**: No fitted alpha can be produced end-to-end today; the feature is "substrate + components," not a running pipeline.
- **Blast radius**: None to existing behavior — `confidence.py` is untouched and the Wave-1 COALESCE is a pure no-op until something writes a non-NULL `decay_alpha`, which nothing currently does.
- **Data risk**: None — no write path is reachable, so no risk of a bad write.
- **User impact**: None currently visible; this only matters once/if the orchestrator is built and scheduled.

### Solution Options

#### Option A: Build the orchestrator + transaction/locking now
**Description**: Add a composed function (e.g. `run_full_fit(session, target_types)`) that, per type, resolves the prior, runs both estimators, pools, write-backs, backfills, and records metrics — all inside one transaction with a `pg_advisory_xact_lock` keyed on the type, per qa-agent-4's F-002/F-003 recommendation. Add a CLI entrypoint and wire it into a scheduled job.
**Changes required**:
- [ ] `decay_fitting/run_fitter.py` — new `run_full_fit` composing all stages
- [ ] Advisory lock + single-transaction wrapper around write_back + backfill
- [ ] CLI/`__main__` entrypoint + scheduler wiring (new worker or APScheduler job)
- [ ] New integration test proving the composed flow against a real DB
- [ ] Docs: `.claude-context.md` "not yet wired" note removed once true
**Benefits**: PRD-0120 becomes actually operable; class priors get replaced by real fits.
**Drawbacks**: New scope beyond the reviewed diff; needs its own QA pass (scheduling cadence, failure/retry semantics, monitoring).
**Effort**: Medium-High
**Risk**: Medium (first real write path touching `relation_type_registry`/`relations` in production)

#### Option B: Ship as-is; document and schedule as explicit follow-up
**Description**: Keep the current state — every building block implemented, tested, and safe (nothing runs, nothing writes). Track the orchestrator as a new, explicitly-scoped follow-up plan.
**Changes required**:
- [ ] `docs/plans/TRACKING.md` — add a follow-up plan stub referencing this gap
- [ ] No code changes
**Benefits**: Zero additional risk introduced in this session; keeps the already-large PLAN-0123 diff reviewable.
**Drawbacks**: PRD-0120 remains not-actually-operable until the follow-up lands.
**Effort**: Low
**Risk**: Low

### Recommended Option
**Option B** for this session (already reflected in the plan's own Wave 4 self-assessment); Option A is the correct next step but is a genuinely separate, non-trivial scope decision that should go through its own `/plan` + `/implement` cycle rather than being bolted on inside a QA-fix pass.

### Verification Steps
- [ ] When Option A is undertaken: a new integration test drives `run_full_fit` end-to-end against a seeded test DB and asserts `relation_type_registry.decay_alpha` and `relations.decay_alpha` converge for a synthetic type.
- [ ] Confirm `confidence.py` remains zero-diff.

## Issue F-002: `run_fitter.py` stamped every `DecayFit` with a hardcoded MEDIUM prior regardless of the type's real decay class

### Summary
`_fit_one_type` passed a single module constant (`_DEFAULT_PRIOR_ALPHA`, the MEDIUM class value) as `prior_alpha` for all ~14 target types, even though 9 of them are NOT MEDIUM (FAST, DURABLE, EPHEMERAL, PERMANENT, SLOW). This mis-reports the prior in every shadow report and would mis-calibrate pooling shrinkage for any non-MEDIUM sparse type once the write path is wired (Issue F-001).

### Severity / Confidence
**Severity**: MAJOR
**Confidence**: HIGH
**Flagged by**: qa-agent-5 (Architecture, F-003)

### Root Cause Analysis
- **What**: `run_fitter.py:169` and `:196` (pre-fix) both read `prior_alpha=_DEFAULT_PRIOR_ALPHA`, a constant fixed to `_CLASS_PRIOR_ALPHAS["MEDIUM"]` (0.011552).
- **Why**: Wave 2 built the report/fit path before Wave 3's pooling existed; the "real prior per type" resolution was deferred with an explicit code comment ("until Wave 3 resolves the type's real decay_class") — but Wave 3 built `pool_type_fit` (which *accepts* a `prior_alpha` parameter) without anyone updating the *caller* to supply the correct per-type value, so the deferred TODO was never actually closed.
- **When**: Always, for every type whose seeded `decay_class != MEDIUM` — verified live against `relation_type_registry`: 9 of the 14 target types (`analyst_rating`=FAST, `price_target`=FAST, `downgraded_by`=FAST, `sentiment_signal`=EPHEMERAL, `credit_rating`=DURABLE, `corporate_action`=DURABLE, `divested_from`=PERMANENT, `filed_lawsuit_against`=SLOW, plus 5 correctly MEDIUM).
- **Where**: Application layer, `run_fitter.py`, the per-type fit-assembly step.
- **History**: Newly introduced by Wave 2; not closed by Wave 3 as the code comment implied it would be.

### Evidence
```sql
-- verified live 2026-07-14 against deployed intelligence_db:
 analyst_rating        | FAST
 corporate_action      | DURABLE
 credit_rating         | DURABLE
 divested_from         | PERMANENT
 downgraded_by         | FAST
 filed_lawsuit_against | SLOW
 price_target          | FAST
 sentiment_signal      | EPHEMERAL
 -- (5 others correctly MEDIUM)
```
- **File**: `services/knowledge-graph/src/knowledge_graph/application/analytics/decay_fitting/run_fitter.py:169,196` (pre-fix)

### Impact
- **Immediate**: Every shadow-report `DecayFit.prior_alpha` for 9/14 types was wrong (e.g. `sentiment_signal`'s true prior is 0.231049, not 0.011552 — nearly 20x off).
- **Blast radius**: `evaluation.evaluate_fit_vs_prior` and `pooling.pool_type_fit` both take `prior_alpha` as an argument from the caller — if the write path is later wired (F-001) without this fix, sparse types would shrink toward the wrong class value.
- **Data risk**: None yet (write path unreachable), but this would have silently mis-calibrated 9 of 14 types the moment F-001 is closed.
- **User impact**: None yet; would have surfaced as systematically wrong fitted alphas for FAST/DURABLE/EPHEMERAL/PERMANENT/SLOW types.

### Fix Applied
Added `_TYPE_DECAY_CLASS: dict[str, str]` (verified live against the registry) and `_type_prior_alpha(canonical_type)`; both `DecayFit(...)` construction sites in `_fit_one_type` now call `_type_prior_alpha(canonical_type)` instead of the single constant.

### Verification Steps
- [x] `.venv/bin/pytest tests/unit/application/analytics/decay_fitting/ -q` — all pass, no regression
- [x] Manually traced: `sentiment_signal` now resolves `prior_alpha=0.231049` (EPHEMERAL), `credit_rating` resolves `0.000950` (DURABLE)

## Issue F-004: `backfill.py` re-writes and re-flips `confidence_stale` on every re-run even when nothing changed (MINOR)

**Severity**: LOW / **Confidence**: HIGH
**File**: `services/knowledge-graph/src/knowledge_graph/application/analytics/decay_fitting/backfill.py`
**Issue**: The UPDATE had no guard against re-applying an unchanged `decay_alpha`, so a repeated run (or a re-fit landing on the same value) would needlessly rewrite every row of the type and force a full confidence recompute.

**Auto-fixable**: YES — applied.
