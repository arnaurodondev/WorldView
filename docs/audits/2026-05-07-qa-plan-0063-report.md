#### Evidence

### Issue F-X01: Circuit breaker has no HALF_OPEN gating (stampede on cooldown expiry)

**Severity**: CRITICAL
**Confidence**: HIGH
**Flagged by**: Distributed Systems (F-X01); cross-references F-X05 (record_success race)

#### Summary
`SourceCircuitBreaker.is_open()` returns `False` for **every** concurrent request once the `state` key TTL expires. Every in-flight retrieval then probes the (still-failing) downstream simultaneously, re-records failures, re-trips the breaker, and amplifies load on the broken source by N-fold per cooldown cycle.

#### Root Cause Analysis
- **What**: `services/rag-chat/src/rag_chat/application/pipeline/circuit_breaker.py:131-140` — `is_open()` reads only `state` key; no probe-slot reservation.
- **Why**: The class docstring acknowledges "HALF_OPEN = implicit (key expired, failures ZSET still present)" — but implicit half-open without per-probe gating is a stampede waiting to happen.
- **When**: At every cooldown boundary while downstream remains unhealthy.
- **Where**: rag-chat retrieval orchestrator, every per-source breaker (`chunk`, `entity_kg`, `financial`, `portfolio`, etc.).
- **History**: Net-new in W5-3 / W5-5 hardening (the BP-403 Lua-script fix on `record_failure` correctly closed the intra-key race; the cross-key/probe-gating gap was not addressed).

#### Evidence
```python
async def is_open(self) -> bool:
    state = await self._valkey.get(self._state_key)
    return state == "open"   # → False once TTL expires; no probe gating
```

#### Impact
- **Immediate**: 1× cooldown-expiry → N × concurrent retry storm on the recovering source.
- **Blast radius**: Worst case S6/S7 (the most-used downstreams) — every chat request stalls together; gauge `rag_circuit_breaker_open{source=}` flaps.
- **Data risk**: None.
- **User impact**: All concurrent chat sessions experience the same elevated latency / 5xx burst at every cooldown boundary; if downstream fails again, breaker re-trips immediately.

#### Solution Options

##### Option A: SETNX probe slot (recommended)
**Description**: When `state` key is absent, attempt `SET rag:cb:{src}:probe "1" EX 5 NX`. Only the request that wins the SETNX runs the actual call; everyone else returns `True` (open) and short-circuits. On success, `record_success()` clears both keys; on failure, `record_failure()` re-trips.
**Changes required**:
- [ ] `services/rag-chat/src/rag_chat/application/pipeline/circuit_breaker.py` — modify `is_open()` (or introduce `try_probe()`) to do the SETNX dance using the existing `valkey_client.set_nx()` helper.
- [ ] Update tests in `services/rag-chat/tests/unit/application/test_circuit_breaker.py` — assert: only one of N concurrent calls wins the probe; the rest see `is_open() == True`.
- [ ] Lower `cool_down_seconds` default from 3600 → 120 (paired with F-X04).
**Benefits**: textbook half-open behaviour; bounds blast radius regardless of cooldown choice.
**Drawbacks**: introduces a 5s probe TTL window where the breaker is "owned" by one request; correct but adds one Valkey round-trip on the recovery path.
**Effort**: Low (~30 LOC + tests). **Risk**: Low.

##### Option B: Probabilistic admission
Keep is_open() as-is but admit only `1/N` of requests during the cooldown window. Simpler but worse — leaks a bounded fraction of stampede, doesn't bound it absolutely.

#### Recommended
**Option A** — pair with F-X04 (lower cooldown) so a transient hiccup does not lock chat for an hour.

#### Verification Steps
- [ ] Concurrency test: spawn 50 tasks calling `is_open()` after the state key expires; assert exactly one observes `False` and 49 observe `True`.
- [ ] Recovery test: simulate downstream success after the probe slot lands; assert breaker transitions OPEN→CLOSED in one round-trip.

### Issue F-X12 / F-D03 / F-X11: D-004 dual-DB commit gap silently amplifies duplicates

**Severity**: CRITICAL
**Confidence**: HIGH
**Flagged by**: Distributed Systems (F-X11, F-X12), Data Platform (F-D03)

#### Summary
`article_consumer.is_duplicate(event_id)` returns hard-coded `False` and `mark_processed(event_id)` is a no-op; idempotency is "via DB-level constraints". Chunks have deterministic PKs (ON CONFLICT DO NOTHING), but mentions, embeddings, and the outbound `nlp.article.enriched.v1` event use `common.ids.new_uuid7()`. On Kafka re-delivery, the same article creates duplicate mentions + embeddings + enriched events with different IDs — sailing past every PK conflict. Worse: if the intel-db commit fails inside `_run_pipeline`, the `routing_decisions` row is committed but intel rows are not — and the early-skip path (`routing_decisions.exists()` at line 247) means the intel side is **never retried**.

#### Evidence
```python
# article_consumer.py:796-800
async def is_duplicate(self, event_id: str) -> bool:
    return False  # At-least-once; idempotency via DB-level constraints
async def mark_processed(self, event_id: str) -> None:
    pass
```
- BP-124 (existing pattern) addressed a related gap for entity replay; the same shape exists for full article replay.

#### Impact
- **Immediate**: Re-delivery on rebalance amplifies every downstream metric (mention counts, embedding rows, llm_usage_log) by N×.
- **Blast radius**: KG receives duplicate enriched events → potential duplicate canonical entities / relations (mitigated by KG dedup but adds work).
- **Data risk**: medium — duplicate mentions skew downstream relevance scoring; duplicate embeddings consume DeepInfra budget.
- **User impact**: indirect — degraded ranking from duplicate signal.

#### Solution Options

##### Option A: `processed_events` table with `INSERT ... ON CONFLICT DO NOTHING RETURNING` (recommended)
**Description**: Add a `processed_events` table in `nlp_db` with `event_id` PK; insert at the start of `_run_pipeline` in the same transaction as the rest of the writes. On conflict, abort early. Wire `BaseKafkaConsumer` to use it via the existing `is_duplicate`/`mark_processed` hooks.
**Changes required**:
- [ ] `services/nlp-pipeline/alembic/versions/0018_add_processed_events.py` (new migration).
- [ ] `services/nlp-pipeline/src/nlp_pipeline/infrastructure/messaging/consumers/article_consumer.py` — implement `is_duplicate()` properly + INSERT into `processed_events` in the pipeline UoW.
- [ ] Regression test: re-deliver same `event_id` twice; assert row counts identical after both runs.
**Effort**: Medium (~50 LOC + migration). **Risk**: Low.

##### Option B: Deterministic IDs (`uuid5` from `event_id`)
Make all generated IDs in `_run_pipeline` derived from `(event_id, kind, index)` via uuid5 — same pattern PLAN-0076 Wave A used for `temporal_events` (BP-397). Re-delivery then idempotently overwrites/skips on PK conflict.
**Effort**: Higher (~80 LOC). **Risk**: Medium (changes ID contracts; downstream consumers depending on uuid7 monotonicity may need review).

#### Recommended
**Option A** — simpler, more transparent, matches BaseKafkaConsumer idiom. File as a tracked sub-plan task (touches migration + consumer + tests; is its own ~half-day of work).

#### Verification

- [ ] Force an intel-commit failure (Faulty UoW); confirm `routing_decisions` is NOT committed (transaction rolls back) so the next delivery re-attempts both DBs.

### Issue F-X04: 1-hour circuit-breaker cooldown + no fallback path (CRITICAL)

**Severity**: CRITICAL
**Confidence**: HIGH
**Flagged by**: Distributed Systems (F-X04)

#### Summary
`SourceCircuitBreaker.cool_down_seconds: int = 3600` default. On 3 failures within 120s, the chunk breaker stays OPEN for an hour. There is no fallback retrieval path when the chunk breaker is open — the orchestrator returns `items=[]` and chat is grounded in nothing.

#### Solution
Two-part fix:
1. Lower `cool_down_seconds` default from 3600 → 120 (matches the failure-window semantics).
2. When chunk breaker is open, fall back to lexical-only via S6 (the lexical leg uses a different DB index family — FTS, not HNSW — so it is a viable redundant path).
3. Emit a Prometheus gauge `rag_circuit_breaker_open{source=}` so an alert can fire when chunk is open >5 min.

Pair fix with F-X01 (probe gating) so transient hiccups do not catastrophically lock chat.

#### Recommended
Low effort; pair with F-X01 in the same CB hardening commit.
