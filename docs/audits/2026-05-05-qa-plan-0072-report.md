### F-DS-002: Phase 3 batch-session rollback silently discards all prior rows

**Severity**: CRITICAL
**Confidence**: HIGH
**Flagged by**: Distributed Systems

**Root Cause Analysis**
- **What**: `ProvisionalEnrichmentWorker.run()` opens a single `AsyncSession` in Phase 3 (`provisional_enrichment.py:322`) and iterates over all `enrichment_results` (up to 50 rows) with no per-row commit. When `_persist_enrichment` or the queue UPDATE fails for row N, `session.rollback()` is called at line 369 — this rolls back all prior successful writes in the same session (rows 0 through N-1).
- **Why**: The session was designed as a single-transaction batch. The comment at line 367–368 says "rollback aborted transaction before retry UPDATE so SQLAlchemy does not raise InvalidRequestError" — which is correct, but the rollback scope is the entire session, not just the failing row's work.
- **When**: Any time a single row fails during Phase 3 extraction persist — DB constraint violation, unique-key conflict, connection interruption.
- **Where**: `services/knowledge-graph/src/knowledge_graph/infrastructure/workers/provisional_enrichment.py:369`
- **History**: Not a known BP; introduced with the Phase-2/Phase-3 session split in PLAN-0072.

**Evidence**
```python
# lines 322-373
async with self._sf() as session:          # single session for entire batch
    for (queue_id, ...) in enrichment_results:
        try:
            entity_id = await self._persist_enrichment(session, ...)
            await session.execute(UPDATE status='resolved' ...)
        except Exception:
            await session.rollback()        # ← rolls back ALL prior rows
            await self._apply_retry(session, queue_id, retry_count)
    await session.commit()                  # ← only commits rows after last rollback
```

**Impact**
- Rows successfully enriched before the failing row lose all DB writes and remain in `status='processing'`
- They are recovered only by the stale sweep (30-minute window) — up to 30 minutes of data loss
- Entity IDs that were computed and added to `entity_ids_to_dirty` still get Kafka events posted despite the writes being rolled back

**Solution Options**

#### Option A: Per-row commit inside the loop
- **Description**: Call `await session.commit()` after each successful `_persist_enrichment` + UPDATE, before moving to the next row. Use a separate `try/except` that only catches the rollback for the failing row without affecting prior committed rows.
- **Changes required**:
  - [ ] `provisional_enrichment.py:358` — add `await session.commit()` after successful UPDATE
  - [ ] Move the `entity_ids_to_dirty.append(entity_id)` AFTER the per-row commit
  - [ ] Change rollback to only affect the current (uncommitted) row — no change needed since prior rows are already committed
- **Benefits**: Matches the SummaryWorker pattern (commits inside loop); isolated failure per row
- **Drawbacks**: Multiple round-trips to DB per batch instead of one; slightly higher overhead
- **Effort**: Low | **Risk**: Low

#### Option B: SAVEPOINT (begin_nested) per row
- **Description**: Use `async with session.begin_nested():` for each row. A savepoint rollback only undoes the current row's work without affecting prior rows in the outer transaction.
- **Effort**: Low | **Risk**: Low

**Recommended Option**: **Option A** — simpler, consistent with SummaryWorker per-relation commit pattern.

**Verification Steps**
- [ ] `python -m pytest services/knowledge-graph/tests/unit/infrastructure/workers/test_provisional_enrichment.py -v`
- [ ] Add a test where row[2] fails and assert rows [0,1] were committed

### F-DS-001: SummaryWorker holds DB session open across LLM I/O

**Severity**: CRITICAL
**Confidence**: HIGH
**Flagged by**: Distributed Systems

**Root Cause Analysis**
- **What**: `SummaryWorker.run()` opens a single `AsyncSession` at line 65 that spans the entire 20-relation batch, including per-relation LLM calls at line 151 (`_generate_summary`). Each LLM call (DeepInfra) takes 5–30 s. For a full batch of 20, the session is held open for up to 600 s.
- **Why**: The session was not split into fetch/write phases — the original design held the session for simplicity.
- **When**: Any time the SummaryWorker runs with a populated stale-relations queue.
- **Where**: `services/knowledge-graph/src/knowledge_graph/infrastructure/workers/summary.py:65`

**Evidence**
```python
async with self._sf() as session:     # line 65 — held for entire batch
    stale_relations = await rel_repo.fetch_stale_summary(20)    # line 70
    for rel in stale_relations:           # lines 83–169
        evidence_rows = await ev_repo.get_all_for_relation(...)  # DB read
        summary_text = await self._generate_summary(...)         # LLM call (5–30s)
        await session.commit()            # committed per-relation, but session stays open
```

**Impact**
- Connection pool exhaustion: if `_concurrency=5` workers each hold a session across 5 LLM calls, up to 100 s of connection hold time per cycle
- `fetch_stale_summary` may use `SELECT FOR UPDATE SKIP LOCKED` — those row locks are released at first per-relation commit, mitigating the worst case, but the DB connection remains occupied

**Solution**
Apply the three-phase ARCH-003 pattern from `ProvisionalEnrichmentWorker`:
- Phase 1: open session → fetch stale relations → read evidence → close session
- Phase 2: call `_generate_summary` for each relation (no session)
- Phase 3: open new session → write summaries + mark updated → commit per-relation → close session

**Verification Steps**
- [ ] No session open during any `_generate_summary` call
- [ ] All existing `TestSummaryWorker*` tests still pass

### F-DATA-002: No standalone index on `relations.relation_id` — partition fanout on every graph call

**Severity**: CRITICAL
**Confidence**: HIGH
**Flagged by**: Data Platform

**Root Cause Analysis**
`relations` is HASH-partitioned ×8 on `subject_entity_id`. The new `get_evidence_snippets_batch` CTE (T-72-2-01) filters `WHERE r.relation_id = ANY(:relation_ids)` — but there is no standalone index on `relations.relation_id` (only partition-local PKs on `(relation_id, subject_entity_id)`). PostgreSQL must scan all 8 partitions for every graph API call. On a graph with 50+ relations, this is 400 partition scans per API request.

**Evidence**: Migration 0001 creates indexes `uidx_relations_triple`, `idx_relations_subject` — none on `relation_id` alone. The new CTE in `relation_evidence.py:334` performs a JOIN `ON rer.relation_id = r.relation_id`.

**Solution**: New migration adding `CREATE INDEX CONCURRENTLY idx_relations_relation_id ON relations (relation_id)` — can be created as revision 0022 (currently unused).

**Verification Steps**
- [ ] `EXPLAIN (ANALYZE, BUFFERS)` on `GET /api/v1/entities/{id}/graph` confirms index scan used

### F-ARCH-011: Noise batch UPDATE is N+1 per-row loop (plan acceptance criterion violation)

**Severity**: MAJOR
**Confidence**: HIGH
**Flagged by**: Architecture + QA/Test
**File**: `services/knowledge-graph/src/knowledge_graph/infrastructure/workers/provisional_enrichment.py:275-285`

**Issue**: The noise UPDATE iterates each noise ID individually, issuing one `UPDATE` per row. The plan's T-72-1-01 acceptance criteria explicitly specified a **single batch UPDATE** using `WHERE queue_id = ANY(:ids)`.

**Evidence**
```python
for noise_id in all_noise_ids:              # N+1 writes
    await session.execute(text("""
UPDATE provisional_entity_queue SET status = 'noise'
WHERE queue_id = CAST(:qid AS uuid) AND status = 'processing'"""), {"qid": str(noise_id)})
```

**Fix**:
```python
await session.execute(text("""
UPDATE provisional_entity_queue SET status = 'noise', resolved_at = now()
WHERE queue_id = ANY(CAST(:ids AS uuid[]))
  AND status = 'processing'"""), {"ids": [str(nid) for nid in all_noise_ids]})
```

### F-DATA-003 / F-DS-008 (MERGED): Missing status guard on resolved UPDATE

**Severity**: MAJOR
**Confidence**: HIGH (2 agents)
**File**: `services/knowledge-graph/src/knowledge_graph/infrastructure/workers/provisional_enrichment.py:347-353`

**Issue**: The `status='resolved'` UPDATE has no `AND status='processing'` guard. All other status transitions in this file include this guard (noise UPDATE uses `AND status='processing'`; processing SET uses `AND status='pending'`). Without the guard, a crash-recovery race where the sweep resets a row to `'pending'` and a second worker picks it up can result in the first worker overwriting the second worker's `'processing'` state with `'resolved'`.

**Fix**: Change line 350 to add `AND status = 'processing'` to the WHERE clause.

### F-DATA-005 / F-DS-005 (MERGED): Sequential Layer 2 noise classification blocks worker for 100–500 s

**Severity**: MAJOR
**Confidence**: HIGH (2 agents)
**File**: `services/knowledge-graph/src/knowledge_graph/infrastructure/workers/provisional_enrichment.py:507-514`

**Issue**: `_run_noise_filters` iterates Layer 2 candidates serially. With `batch_limit=50` and `noise_classifier_timeout_s=10.0`, worst-case blocking time is 500 s — exceeding the 5-minute worker cycle. Phase 2 already uses `asyncio.gather` + semaphore; Phase 1.5 must match.

**Fix**: Wrap Layer 2 classification with `asyncio.gather`:
```python
sem = asyncio.Semaphore(self._concurrency)
async def _classify_one(row):
    async with sem:
        return row, await self._layer2_classify(row[1])
results = await asyncio.gather(*[_classify_one(r) for r in layer2_candidates])
```

### F-DS-003: Non-atomic dedup race → duplicate canonical entities

**Severity**: MAJOR
**Confidence**: HIGH
**File**: `services/knowledge-graph/src/knowledge_graph/infrastructure/workers/provisional_enrichment_core.py:246-259`

**Issue**: `persist_enrichment` does a read-then-write (`find_exact` → `entity_repo.create()`) without atomicity. Two concurrent workers (ProvisionalQueuedConsumer + polling worker) both processing mentions of the same entity name see `find_exact → None` and both INSERT — `canonical_entities` has no UNIQUE constraint on `canonical_name`, so both succeed creating duplicates (previously documented as BP-384 partially, now more precise).

**Requires Decision**: Whether to add a `UNIQUE INDEX ON canonical_entities(lower(canonical_name))` migration (safer, prevents duplicates at DB level) or use an advisory lock (application-level, no schema change).

### F-SEC-001: Structurally fragile dynamic WHERE clause in list_filtered

**Severity**: MAJOR
**Confidence**: HIGH
**File**: `services/knowledge-graph/src/knowledge_graph/infrastructure/intelligence_db/repositories/relation.py:397-413`

**Issue**: `list_filtered()` builds `where_sql` by concatenating clause strings and interpolates via `text(f"... WHERE {where_sql} ...")`. Safe today (all clauses are hardcoded), but one contributor error away from SQL injection if a user-supplied value is added to a clause template. `semantic_mode` and `canonical_type` arrive from API query params with no allowlist validation.

**Fix**: Use the static parameterized pattern with `CAST(:param AS text) IS NULL OR col = :param`. Also add `max_length` or `Literal` type to both query params in `routes.py`.
