## BLOCKING (must fix before /implement)

### B-1 — Wave W5-4-02 mis-models the watchlist signal: tickers vs entity_ids

**Where**: PLAN-0063 §7 W5-4-02 (lines ~1114–1182)

**Plan claim**:
- Valkey key is `s6:routing:watchlist:tickers` storing **ticker strings** (AAPL, MSFT, …).
- The set "is not populated" (verified by `grep -rn "s6:routing:watchlist" services/nlp-pipeline/src/` returning 0 hits).
- Solution: 5-minute cron that fetches `GET /internal/v1/portfolios/watchlist-tickers` (a flat ticker list) and rebuilds the Valkey set.

**Reality (verified from source)**:
- The actual Valkey key is `nlp:v1:watched_entities` (see `services/nlp-pipeline/src/nlp_pipeline/infrastructure/valkey/watchlist_cache.py:31`). The default is overridable via `Settings.valkey_watchlist_key`.
- The set stores **entity UUIDs**, not tickers (`add_entity(entity_id: UUID)` does `SADD self._key, str(entity_id)`).
- The set IS already populated event-driven: `services/nlp-pipeline/src/nlp_pipeline/infrastructure/messaging/consumers/watchlist_consumer.py:107` consumes `portfolio.watchlist.updated.v1` and calls `await self._cache.add_entity(entity_id)`.
- The Avro schema `infra/kafka/schemas/portfolio.watchlist.updated.v1.avsc` already carries `entity_id` and `entity_ids_affected` — there is no ticker field on the wire.
- Block 5 routing (`services/nlp-pipeline/src/nlp_pipeline/application/blocks/routing.py:79`) takes `watched_entity_ids: frozenset[UUID]` and uses `m.resolved_entity_id in watched_entity_ids`. It has nothing to do with tickers.
- The article consumer (`article_consumer.py:319`) hydrates this via `await self._watchlist.get_all_watched()`.

**Why this is BLOCKING**: implementing the plan as written would (a) create a parallel Valkey key that the consumer never reads, (b) duplicate an existing Kafka-driven sync mechanism with a polling cron, (c) fail to feed the routing signal because tickers cannot be matched against `resolved_entity_id` (UUID). The result is "successfully shipped, zero behavioural change."

If the empirical observation behind this task is "watchlist signal is currently 0 in production", the actual root cause must be diagnosed (likely: the watchlist consumer is not running, OR the Kafka topic has no events because portfolio-side outbox is silent, OR resolved_entity_id is `None` on most mentions). A 5-minute cron does not address any of those.

**Required fix before /implement**:
1. Drop T-W5-4-02's "5-min cron + ticker endpoint" architecture entirely, OR
2. Re-scope T-W5-4-02 to "diagnose & repair the existing entity-id-keyed watchlist consumer pipeline" (likely the right answer).
3. Update §2 codebase-state-verification table accordingly.
4. Remove from §9.4 the `WATCHLIST_SYNC_INTERVAL_SECONDS` and `WATCHLIST_SYNC_PORTFOLIO_URL` settings — neither should exist if the existing consumer is the chosen path.

### B-2 — Wave W5-4-03 proposes externalising signals that are already externalised

**Where**: PLAN-0063 §7 W5-4-03 (lines ~1186–1247)

**Plan claim**: "Two hardcoded `0.5` constants in the routing-score computation are replaced with rule-based dictionaries keyed on `source_type`." Adds `_SOURCE_RELIABILITY` and `_DOC_TYPE_SIGNAL` maps with sensible per-source values (sec_filing=0.95, eodhd_news=0.65, etc.).

**Reality (verified from `services/nlp-pipeline/src/nlp_pipeline/application/blocks/routing.py`)**:
- `source_reliability` is **already externalised** — it is supplied via the `source_trust_weight: float` parameter on `compute_routing_score()` (line 156: `"source_reliability": source_trust_weight`). This value is sourced from `intelligence_db.source_trust_weights` (per the existing PRD note in PLAN-0058 Wave E-3).
- `document_type` is **already externalised** — uses the `DOCUMENT_TYPE_SIGNAL` dict at routing.py:43–55 with sensible per-source values (sec_8k=0.95, sec_10k=0.90, eodhd_news=0.55, …) and a `_DEFAULT_DOCUMENT_TYPE_SIGNAL=0.50`.
- The only `0.5` literals near routing logic are the *default* fallbacks for unknown source_types — which is correct behaviour.
- Plan's proposed default of `default=0.50` for source_reliability would **regress** behaviour: today the value comes from a per-source DB-driven trust weight; replacing it with an in-code dict drops support for the database-driven source-trust-weights table.

**Why this is BLOCKING**: implementing this task would (a) duplicate existing constants in a new location, (b) shadow the database-driven `source_trust_weights` table (existing PLAN-0057 / PLAN-0058 substrate), (c) silently weaken the routing signal because the new in-code dict is shorter than the existing combined keys.

**Required fix before /implement**:
1. Drop T-W5-4-03 entirely, OR
2. Re-scope to "audit `source_trust_weights` table population — confirm rows for sec_8k/eodhd_news/etc. exist; if not, add Alembic data migration to seed them." This is the actual gap.
3. Reflect in the Codebase State Verification table — the row claiming `0.5` constants need replacement is wrong.

### B-3 — Cross-plan conflict with PLAN-0064 (W6 FTS): tsquery parser and index name divergence

**Where**: PLAN-0063 §0 + §5 W5-2-01/02 vs PLAN-0064 §3 + §4 + §6 (multiple sections)

**Conflict**:
- **Tsquery parser**: PLAN-0063 uses `websearch_to_tsquery('english', :q)` for both index lookups and ranking (line 454, 564, 568, 569, 593). PLAN-0064 uses `plainto_tsquery('english', :q)` (lines 106, 179, 194, 395, 400, 456, 473) — and explicitly justifies the choice on injection-safety grounds in §9 of the PRD.
- **GIN index name**: PLAN-0063 W5-2-01 creates `ix_chunks_tsv_gin`. PLAN-0064 line 548 explicitly asserts the index is named `idx_chunks_tsv_gin` (`Bitmap Index Scan on idx_chunks_tsv_gin`).
- **PLAN-0063's own §0 says they "should reuse the Postgres-side helper `websearch_to_tsquery('english', :q)` rather than diverge into different parsers."** The diverging parsers in PLAN-0064 invalidate that coordination claim — and PLAN-0064 was authored on the same day (2026-05-03), so the divergence is current and unresolved.
- **PLAN-0063's §0 also asserts**: "the W6 plan is expected to add separate `documents.tsv` indexes." But PLAN-0064 AD-W6-3 explicitly **rejects** adding `documents.tsv`. So the claim of "no conflict on column names" is also based on a wrong premise.

**Why this is BLOCKING**: PLAN-0064 Wave 3 will run `EXPLAIN ANALYZE` and assert `Bitmap Index Scan on idx_chunks_tsv_gin`. PLAN-0063's migration creates `ix_chunks_tsv_gin`. PLAN-0064's integration test will fail. Additionally, mixing `websearch_to_tsquery` (W5 internal RAG path) and `plainto_tsquery` (W6 public search path) on the same index produces different match semantics for the same input — debugging "why does the same query return different results in chat vs in /search" will be a forever-task.

**Required fix before /implement**:
1. Decide ownership: which plan owns the index name? (Recommend `idx_chunks_tsv_gin` per Postgres convention.)
2. Decide a single tsquery helper for the GIN index. Recommendation: `websearch_to_tsquery` for both — it auto-escapes (so PLAN-0064's injection rationale still holds) and supports operator syntax (so PLAN-0063's `'iPhone -Android'` test case works).
3. Update PLAN-0063 §0 coordination paragraph and PLAN-0064 §3 + §6 + §10 (line 548 EXPLAIN assertion) to match.
4. Remove PLAN-0063's claim that W6 will add `documents.tsv` — it explicitly will not.

### B-5 — Cross-tenant aggregation in W5-4-02 collides with W8 RLS within MVP window

**Where**: PLAN-0063 W5-4-02 vs PRD-0034 §3 FR-T2-2

**Conflict**:
- PLAN-0063 W5-4-02 plans a flat-list-of-tickers endpoint that intentionally aggregates **across all tenants** ("flat de-duplicated list across **all tenants**. This is intentional: the routing signal cares whether *anyone* is watching a ticker"). The plan caveats this with "no PII, just public ticker symbols" and proposes documenting it inline.
- PRD-0034 §3 FR-T2-2 explicitly lists `watchlists` in the table set that gets RLS via `tenant_id = current_setting('app.tenant_id')` in **W8** (Week 5 of the same 6-week launch).
- PRD-0034 §7 sprint calendar has W8 *after* W5 (Week 5 vs Week 3) but both ship before public launch, so the cross-tenant query becomes a **silent breakage at W8 ship time**: queries on `watchlist_members` (joined to RLS-enforced `watchlists`) will return zero rows under the system JWT.

Even setting aside the routing breakage: even if the *intent* is "aggregate routing tickers across tenants," that is a security-design decision (it creates an internal endpoint that bypasses RLS by design). PRD-0034 §9 Security Analysis does not list a "cross-tenant routing signal endpoint" as an exception. Adding one without an architectural decision is a creeping security debt.

**Combined with B-1** (the entire watchlist-ticker mechanism is wrong): this is a moot point if B-1 is resolved by deleting T-W5-4-02. But if T-W5-4-02 survives in any form, the cross-tenant aggregation needs an explicit ADR.

**Required fix before /implement**:
1. Either drop T-W5-4-02 entirely (preferred, per B-1).
2. OR document a new ADR (`AD-W5-1: cross-tenant routing-signal endpoint exempt from RLS`) and update PRD-0034 §9 accordingly.
3. OR scope the endpoint per-tenant and accept the routing-signal becomes per-tenant (which weakens it but stays inside RLS).

### I-3 — Goal "+0.05 NDCG@10 lift" against an empty/baseline set is mathematically unstable

**Where**: PRD-0034 §3 FR-T1-2 + §4 NFR + PLAN-0063 W5-3-04.

**Issue**: The 50-query golden set is currently a stub (`relevant_doc_ids: []` on every row, verified). The plan correctly promotes labelling to W5-1-01 — good. But:
- **The ≥+0.05 absolute lift target was set in PRD-0034 before any baseline existed.** Without a known baseline, "+0.05" is not a defensible target — it could be trivial (if baseline ANN NDCG@10 is 0.20) or unreachable (if baseline is 0.85). The plan acknowledges this implicitly via "tuning fallback" (try k=30/80, asymmetric weighting) but the underlying target is ungrounded.
- The CI-gate target (regress ≥3% absolute) likewise was set without a baseline.
- The "every query must have ≥5 graded candidates and ≥1 relevance=3 row" is a quality bar — but without sample size analysis, 50 queries with 5 candidates each may not be statistically powered to detect a 0.05 lift (especially with intent-stratified analysis where an intent has only 4 queries: SIGNAL_INTEL).

**Fix**: After W5-1-03 captures the baseline, **re-evaluate** whether +0.05 is the right target. This should be an explicit task: "review baseline numbers, confirm or revise the FR-T1-2 lift target with PM/founder before T-W5-3-04 is run." If the baseline is 0.80+, +0.05 may be infeasible (Spärck Jones-Cormack ceiling effects). If the baseline is 0.30, +0.05 is too easy.

### I-5 — Missing test for the GENERATED column not being declared in the ORM (BP-NEW1 prevention is asserted but not tested)

**Where**: PLAN-0063 W5-2-01 acceptance + W5-5-03 (BP-NEW1 doc).

**Issue**: The plan correctly identifies that declaring `tsv` in `ChunkModel` would crash INSERTs (Postgres rejects writes to GENERATED columns). It documents this with a comment-block in `ChunkModel` and adds a new BP-NEW1.

But the **only** safety net is the comment. There is no automated test asserting "no SQLAlchemy model has a `tsv` column for the `chunks` table." A future refactor (e.g. someone running `sqlacodegen` to refresh ORM from schema) will silently regress this and only fail on the first INSERT.

**Fix**: add a unit test in T-W5-2-01 that asserts `ChunkModel.__mapper__.columns` does **not** contain `tsv`, with a docstring referencing BP-NEW1. This is a one-line test.

### I-7 — Cron timing for citation-accuracy: "first run on next service start" + Sunday 03:00 schedule

**Where**: PLAN-0063 W5-5-02.

**Issue**: The cron is registered inside `app.py:lifespan` with `asyncio.create_task(_run_periodically(...))`. The plan asks for both "first run on next service start" AND "every Sunday 03:00 UTC." This is two different schedules — what does "Sunday 03:00" mean if the service was last restarted Wednesday and the loop is "now and every 7 days"? Will the first-run-on-startup throw off the weekly cadence forever?

**Fix**: pick one cadence (recommend "every Sunday 03:00 UTC, computed from the next-aligned Sunday at process start; emit a single immediate run only on first-ever boot identified by an absent gauge value"). Document the implementation pattern (e.g. `asyncio.sleep(seconds_until_next_sunday_03_utc())`) so the cron doesn't drift.

Also: per R22, the lifespan-spawned task must use `asyncio.wait_for(timeout=...)` on shutdown, and the cron coroutine must be cancelled cleanly. The plan does not call this out.

## NICE-TO-HAVE

### N-1 — RRF k=60 default is justified by a citation, not by empirical evidence

**Where**: PLAN-0063 W5-3-02 / OQ-W5-4. The Cormack-Clarke-Buettcher 2009 paper used `k=60` on TREC web tracks. Our corpus (finance-specific filings + transcripts + news) is a different distribution. The plan authorises tuning at T-W5-3-04 — fine. But it could include a small ablation in the eval script (run k ∈ {30, 60, 100, 150} once and pick the best) so the choice is data-driven from day one rather than reactive.

### N-2 — OQ-W5-2 (backlog hand-labelling) — recommend resolving before W5-1-01 starts

The "raw queries only" recommendation is fine, but it should be a decision, not a recommendation. Resolve before /implement.

### N-3 — Per-leg timeout in `_execute_hybrid` lacks a concrete number

**Where**: W5-3-02 — the plan references BP-235 (httpx default 5s) and uses `asyncio.gather` (no per-leg timeout) but does not specify timeout. The outer `wait_for` in `_with_cb` caps total time, but if ANN takes 4s and lexical takes 4.5s and the outer cap is 5s, the slow leg is silently swallowed. Specify per-leg timeout (e.g. 3s for ANN, 1s for lexical) and add a unit test for the timeout path.

### N-4 — Documentation sweep should also update `docs/specs/0034-mvp-launch-readiness-program.md`

PRD-0034 §6 W5 still says "Existing PLAN-0058 Wave C+D" as the implementation target. Once PLAN-0063 ships, that PRD reference becomes stale. Add a single-line note at the bottom of the PRD's §6 W5 row pointing to PLAN-0063, so future readers don't go on a paper-chase through PLAN-0058 → PLAN-0060 → PLAN-0063.

## Top 3 findings (action priority)

1. **B-1 + B-2 — Wave W5-4 is built on a stale codebase model.** Both `compute_routing_score` source-reliability/document-type signals AND the watchlist Valkey set already exist in different shapes than the plan describes.
