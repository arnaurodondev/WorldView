### B-003 — UptimeRobot API key exposure on a public, unauthenticated `/status` page

**Location**: `docs/plans/0065-w9-regression-cleanup-observability-plan.md` Wave E T-E-02 Option A (lines ~717–727), §9 security analysis (PRD-0034 §9 row "Public status page leakage").

**Issue**: T-E-02 Option A description says the page "fetches UptimeRobot's `getMonitors` JSON via their read-only public API key" and writes the page as a Server Component fetching `https://api.uptimerobot.com/v2/getMonitors` with `api_key=<read-only>`. The plan claims this is safe because the key is "read-only."

UptimeRobot API key types:
- **Main account API key** — full read+write (NEVER expose).
- **Monitor-specific read-only API key** — read for that monitor only. Still **rate-limited per key**; abuse can exhaust your quota.
- The plan's intended choice is plausibly the read-only key, but this is not explicitly enumerated, and even read-only keys can be used to enumerate monitor configuration (alert contacts, custom uptime ratios, monitor URLs that may include internal probe paths).

If the page is implemented as a Server Component (recommended), the API key stays server-side — fine. **But T-E-02 doesn't explicitly require this**; a junior implementer could put the call in `'use client'` mode or pass it through to a client `useEffect` for "live updates," exposing the key in browser network logs. Worse, T-E-02's listed file `apps/worldview-web/app/status/api/uptime/route.ts` (a "proxy" route) is good — but the page file `app/status/page.tsx` is described as fetching directly.

**Why BLOCKING**: A leaked API key (even read-only) lets attackers call UptimeRobot rate-limit windows on your account, enumerate your alert contacts (PII), and potentially extract internal probe paths (which may leak architecture or staging hostnames). This is exactly the leakage class PRD §9 says we must prevent ("Status page shows uptime only, not error details or stack traces").

**Fix**:
1. T-E-02 must require the API call live in **only one place**: `app/status/api/uptime/route.ts` (Next.js Route Handler, Node runtime). The page (`app/status/page.tsx`) fetches `/api/uptime` from same-origin. The API key reads from `process.env.UPTIMEROBOT_READONLY_API_KEY` server-side only — never `NEXT_PUBLIC_*`.
2. Cache the route handler response 60s server-side (already in T-E-02 acceptance) and return only the whitelisted fields: `monitor_id`, `friendly_name`, `status`, `custom_uptime_ratio`, `daily_buckets`. Strip `url`, `alert_contacts`, `interval`, `keyword_value`, etc.
3. Add a unit test asserting the route handler does NOT echo `url` or `alert_contacts` even if UptimeRobot returns them (matches `test_status_page_strips_error_details` but at the API boundary, not the rendering boundary — defense in depth).
4. Document in `docs/runbooks/uptime-monitoring.md` that the API key is **monitor-scoped read-only** (not main account); rotation procedure; that it is never set as `NEXT_PUBLIC_*`.
5. Add an `UPTIMEROBOT_READONLY_API_KEY` env var to the new-env-vars list in §1 plan-summary table (currently lists 8; this should be the 9th).

### I-002 — Sentry free-tier 5K events/month quota assumption is unsafe at multi-service scale

**Location**: PRD-0034 §10 "Sentry quota exhausted | … Free tier 5K errors/month sufficient for MVP"; PLAN-0065 §6 risk row 1 (LOW likelihood); §5 cross-cutting "Free-tier dependency cost — $0 (Sentry free 5K events/month)."

**Issue**: 5K events/month = ~167/day = ~7/hour. With 10 backend services + 1 frontend, even a moderate steady-state error rate of 0.5 errors/hour/service × 11 surfaces = 5.5/hour ≈ 130/day, leaving thin headroom. A single misconfigured query that fires per request (e.g. F-CRIT-07 in S6 last quarter, BP-302 hang earlier this week) can exhaust 5K events in <30 min.

The plan's mitigation ("Sentry's own usage alert at 80%") is a one-way alert — by the time it fires you have already lost capture for the rest of the month.

**Why IMPORTANT**: Without per-fingerprint rate limiting in `before_send`, the first time a regression slips through (and history shows BP-302/F-CRIT-07-class regressions ~monthly), the platform will be effectively blind to subsequent unrelated incidents until the calendar rolls over.

**Fix**:
1. Add to T-C-01 `_strip_pii` (better: a sibling `_rate_limit_fingerprints`) `before_send` hook: per-process LRU keyed on `event["fingerprint"]` or `(exc_type, exc_value_truncated)` with token-bucket of N events/min/fingerprint. Drop excess. Log via structlog so the drop is observable in Loki even when Sentry is dropping.
2. PRD §10 amend: explicitly state "5K events/month free tier is a soft cap; production traffic above 50 RPS or any unbounded-error regression will exhaust the budget. `before_send` rate limiter required (W9 T-C-01)."
3. Add unit test in T-C-01 test list: `test_strip_pii_drops_excess_events_per_fingerprint` — fire 200 events with the same fingerprint in 1s; assert ≤N retained (where N is the configured per-minute cap).

### I-005 — `app/(app)/_dev/sentry-test` route placement collides with `(app)` group

**Location**: PLAN-0065 T-D-02 target file `apps/worldview-web/app/(app)/_dev/sentry-test/page.tsx`.

**Issue**: The path uses `(app)` as a route group AND `_dev` as a private folder. In Next.js 15:
- `(app)` is a group — its children inherit the group's `layout.tsx` and `loading.tsx`.
- `_dev` (underscore prefix) is a **private folder** — Next.js intentionally excludes it from routing.
- Therefore the path `(app)/_dev/sentry-test/page.tsx` is **not routable** — the underscore-prefixed segment makes the URL unreachable.

The plan intends the page to be reachable in dev so an operator can click a button to throw an error. With `_dev` it cannot be reached.

**Why IMPORTANT**: The synthetic-exception probe in T-E-04 (FR-T3-1 acceptance) cannot fire if the route doesn't render.

**Fix**:
1. Change path to `app/(app)/dev-tools/sentry-test/page.tsx` (regular folder, no underscore).
2. Keep the `if (process.env.NODE_ENV === "production") notFound()` guard inside the page component (server-component check at request time) — this is the correct way to make a route 404 in production while keeping it routable in dev.
3. Confirm the Vitest test `test_dev_route_404_in_production` covers this server-component branch (not a client-side guard).

### I-006 — `withSentryConfig` wrap conflicts with existing `next.config.ts` runtime checks

**Location**: PLAN-0065 T-D-01 ("wrap with `withSentryConfig`"); existing `apps/worldview-web/next.config.ts:11-39`.

**Issue**: Existing `next.config.ts` does runtime invariant enforcement at module load (`throw new Error` if `NEXT_PUBLIC_WS_BASE_URL` is `ws://` in production, a security hardening from PLAN-0059 W0). `withSentryConfig` is a function that takes the config object and returns a wrapped one. The order matters:
- If `withSentryConfig(nextConfig, …)` is called at the bottom of the file, the `throw` runs first — fine.
- But Sentry's `withSentryConfig` injects its own webpack plugin and may run side-effects (sourcemap upload, instrumentation injection) at build time. The plan does not specify whether `withSentryConfig` should be conditionally applied (only when `SENTRY_AUTH_TOKEN` is set, which it does say) AND whether the wrap is the last step or interacts with the existing throw-on-`ws://`.

**Why IMPORTANT**: Build can break silently if the Sentry wrap runs before the env-var validation. Worse, if a developer sets `SENTRY_AUTH_TOKEN` locally but not the WS URL, the build error becomes a Sentry-side error message (confusing).

**Fix**:
1. T-D-01 must specify the exact integration shape: `export default process.env.SENTRY_AUTH_TOKEN ? withSentryConfig(nextConfig, { … }) : nextConfig;` — and the existing `throw` for `ws://` runs **before** this line (since module top-level executes top-to-bottom).
2. Add a build smoke step in T-D-01 acceptance: "Run `pnpm build` (a) with both DSN + `SENTRY_AUTH_TOKEN` set, and (b) with neither set, and (c) with the bad `NEXT_PUBLIC_WS_BASE_URL=ws://...` — case (c) must still throw the WS hardening error, not a Sentry error."
