## Executive Summary

PLAN-0038 implements 5 waves adding Yahoo Finance and Finnhub free provider adapters, a shared observability layer (Prometheus + Loki), provider routing, and zero-bar failover to the market-ingestion service. The implementation is architecturally sound — hexagonal port/adapter pattern followed, BaseProviderAdapter provides consistent observability, and all 525 unit tests pass. However, the 5 specialist agents identified **3 CRITICAL issues** that must be fixed before merge: (1) the production worker never registers Yahoo/Finnhub adapters, making the entire routing feature dead code; (2) API keys are stored as plain `str` instead of `SecretStr`; and (3) the Valkey INCR+EXPIRE pair in the zero-bar tracker is non-atomic, risking orphan keys. Additionally, 8 MAJOR issues cover unhandled Valkey errors, missing shared metrics for EODHD errors, potential API key leakage in exception strings, and significant test coverage gaps. All test layers pass (4,584 backend + 376 frontend), but the CRITICAL findings require resolution.

## Issues — Full Investigation

## Issue F-001: Worker registry missing Yahoo/Finnhub — routing feature is dead code (CRITICAL)

### Summary
The `WorkerProcess._build_registry()` method in `worker.py` only registers `EODHDProviderAdapter`. It does not register `YahooFinanceProviderAdapter` or `FinnhubProviderAdapter`. Since the worker is the process that executes ingestion tasks, the provider routing logic in `_preferred_provider()` will always fall through to EODHD. The entire Waves A-2 through A-5 feature set (Finnhub, Yahoo, routing, zero-bar failover) is inoperative in the production worker.

### Severity / Confidence
**Severity**: CRITICAL
**Confidence**: HIGH
**Flagged by**: QA/Test Engineer, Distributed Systems Reviewer, Architecture Decision Lead

### Root Cause Analysis
- **What**: `worker.py:196-207` builds a `ProviderRegistry` with only EODHD. A separate `build_provider_registry()` function in `__init__.py` correctly registers all three providers, but the worker doesn't use it.
- **Why**: The worker's `_build_registry()` was written before PLAN-0038 and was never updated to call the shared `build_provider_registry()`. The PLAN-0038 waves added Yahoo/Finnhub to `__init__.py` but missed updating the worker path.
- **When**: Always — every task execution in the worker process uses the EODHD-only registry.
- **Where**: `infrastructure/workers/worker.py:196-207`
- **History**: Pre-existing worker code; PLAN-0038 Wave A-4 (routing) should have updated this but didn't.

### Evidence
```python
# worker.py:196-207 — only EODHD registered
def _build_registry(self) -> ProviderRegistry:
    registry = ProviderRegistry()
    timeout = getattr(self._settings, "provider_http_timeout_seconds", 30.0)
    client = httpx.AsyncClient(timeout=timeout)
    registry.register(
        EODHDProviderAdapter(
            api_key=self._settings.eodhd_api_key,
            client=client,
            base_url=self._settings.eodhd_base_url,
        ),
    )
    return registry  # Missing Yahoo + Finnhub!
```

### Impact
- **Immediate**: Provider routing never activates; all data fetched via EODHD (wasting credits)
- **Blast radius**: Only market-ingestion service
- **Data risk**: None — EODHD continues to work; just no credit savings
- **User impact**: No visible impact — EODHD still provides correct data

### Solution Options

#### Option A: Delegate to `build_provider_registry()`
**Description**: Have the worker call the shared registry builder from `__init__.py`.
**Changes required**:
- [ ] `worker.py` — replace `_build_registry()` body with `from market_ingestion.infrastructure.adapters.providers import build_provider_registry; return build_provider_registry(self._settings)`
- [ ] Add test verifying worker registry contains Yahoo and Finnhub when configured
**Benefits**: DRY — single registry construction path; future providers auto-included
**Drawbacks**: Worker shares httpx client config with API server path
**Effort**: Low
**Risk**: Low

#### Option B: Add Yahoo/Finnhub registration to `_build_registry()`
**Description**: Replicate the registration logic inline.
**Changes required**:
- [ ] `worker.py` — add Yahoo + conditional Finnhub registration matching `__init__.py`
**Benefits**: Worker maintains independent control of its registry
**Drawbacks**: Code duplication; must update both paths for future providers
**Effort**: Low
**Risk**: Low

### Recommended Option
**Option A** — eliminates duplication and ensures the worker always has the same provider set as the API.

### Verification Steps
- [ ] After fix, assert `registry.get(Provider.YAHOO_FINANCE)` succeeds in worker
- [ ] Assert `registry.get(Provider.FINNHUB)` succeeds when `finnhub_api_key` is set
- [ ] Run full market-ingestion unit test suite

## Issue F-002: API keys as `str` not `SecretStr` in config.py (CRITICAL)

### Summary
`finnhub_api_key`, `eodhd_api_key`, `polygon_api_key`, and `alpha_vantage_api_key` in `config.py` are typed as plain `str` instead of pydantic's `SecretStr`. This means they appear in full in `repr()`, `model_dump()`, error tracebacks, and diagnostic logging.

### Severity / Confidence
**Severity**: CRITICAL
**Confidence**: HIGH
**Flagged by**: Security Engineer

### Root Cause Analysis
- **What**: `config.py:51-54` — four API key fields are `str` type
- **Why**: Pre-existing pattern; `database_url` correctly uses `SecretStr` but API keys were added without following the same pattern
- **When**: Risk materializes when `settings.model_dump()` is called, or when pydantic validation errors include field values
- **Where**: `config.py` — affects all provider adapters that receive API keys

### Evidence
```python
# config.py:51-54
eodhd_api_key: str = "demo"
finnhub_api_key: str = ""
polygon_api_key: str = ""
alpha_vantage_api_key: str = ""
```

### Impact
- **Immediate**: API keys could leak in error logs, tracebacks, or diagnostic output
- **Blast radius**: All services reading these config values
- **Data risk**: None directly — credential exposure risk
- **User impact**: Potential unauthorized API usage if keys leaked

### Solution Options

#### Option A: Migrate to `SecretStr`
**Changes required**:
- [ ] `config.py` — change `str` to `SecretStr` for all 4 API key fields
- [ ] `__init__.py` — update `build_provider_registry()` to call `.get_secret_value()`
- [ ] `worker.py` — update `_build_registry()` to call `.get_secret_value()`
- [ ] Tests — update settings fixtures to use `SecretStr("test-key")`
**Effort**: Low | **Risk**: Low

### Recommended Option
**Option A** — straightforward migration, follows existing `database_url` pattern.

### Verification Steps
- [ ] `repr(settings)` does not contain API key values
- [ ] `settings.model_dump()` masks API keys with `**********`
- [ ] All adapter constructors receive the secret value correctly

## Issue F-003: Non-atomic INCR + EXPIRE in ValkeyZeroBarTracker (CRITICAL)

### Summary
`ValkeyZeroBarTracker.record_zero()` calls `INCR` and `EXPIRE` as two separate Valkey commands. If the process crashes between them, the key persists without TTL indefinitely, consuming memory and potentially triggering false failovers on restart.

### Severity / Confidence
**Severity**: CRITICAL
**Confidence**: HIGH
**Flagged by**: Data Platform Engineer, Distributed Systems Reviewer

### Root Cause Analysis
- **What**: `zero_bar_tracker.py:32-37` — INCR and EXPIRE are separate async calls
- **Why**: Valkey (like Redis) doesn't have a single command combining INCR + EXPIRE. Requires pipeline or Lua script for atomicity.
- **When**: On any process crash, kill signal, or connection drop between the two commands
- **Where**: `infrastructure/adapters/zero_bar_tracker.py`

### Evidence
```python
async def record_zero(self, provider, symbol, timeframe, dataset_type) -> int:
    key = self._key(provider, symbol, timeframe, dataset_type)
    streak = await self._valkey.incr(key)       # Step 1
    await self._valkey.expire(key, self._STREAK_TTL)  # Step 2 — not atomic with Step 1
    return int(streak)
```

### Impact
- **Immediate**: Orphan keys without TTL accumulate in Valkey
- **Blast radius**: Valkey memory; false failovers on service restart
- **Data risk**: No data corruption — only incorrect failover triggers

### Solution Options

#### Option A: Use MULTI/EXEC pipeline
**Changes required**:
- [ ] `zero_bar_tracker.py` — wrap INCR + EXPIRE in `pipeline(transaction=True)`
**Effort**: Low | **Risk**: Low

### Recommended Option
**Option A**

### Verification Steps
- [ ] Unit test verifying INCR and EXPIRE are in same pipeline
- [ ] Integration test with Valkey confirming TTL is set after INCR

## Issue F-006: httpx exception may leak Finnhub API key (MAJOR)

**Severity**: MAJOR
**File**: `services/market-ingestion/src/market_ingestion/infrastructure/adapters/providers/finnhub.py:215-218`
**Issue**: `f"Finnhub connection error: {exc}"` stringifies httpx exceptions which can include the full request URL with `token=SECRET` in query params. These propagate to structlog via `execute_task.py:193`.
**Fix**: Replace `{exc}` with `type(exc).__name__` or a sanitized message. Example: `raise ProviderUnavailable(f"Finnhub connection error on /{endpoint}")`.
**Auto-fixable**: YES

## Issue F-007: ValkeyClient created per-task in worker — connection pool leak (MAJOR)

**Severity**: MAJOR
**File**: `services/market-ingestion/src/market_ingestion/infrastructure/workers/worker.py:171-173`
**Issue**: `_build_circuit_breaker()` and `_build_zero_bar_tracker()` are called per-task, each creating a new `ValkeyClient`. With concurrency=4 and batch_size=10, this creates up to 20 clients per batch cycle, never closed.
**Fix**: Build ValkeyClient, CircuitBreaker, and ZeroBarTracker once in `__init__` and reuse across all tasks. Valkey operations (INCR, GET) are atomic and safe for concurrent use.
**Auto-fixable**: YES

## Issue F-008: Finnhub rate limit not coordinated across coroutines (MAJOR)

**Severity**: MAJOR
**File**: `services/market-ingestion/src/market_ingestion/infrastructure/adapters/providers/finnhub.py:103`
**Issue**: Per-request `asyncio.sleep(1.1)` is per-coroutine, not globally coordinated. With worker_concurrency=4, four concurrent Finnhub tasks can send ~218 req/min, exceeding the 60 req/min free tier limit.
**Fix**: Add a shared `asyncio.Lock()` or `asyncio.Semaphore(1)` at the adapter level to serialize all Finnhub requests within a process.
**Requires decision**: YES (single-process Lock vs distributed Valkey rate limiter)

## Issue F-009: yfinance executor call has no timeout (MAJOR)

**Severity**: MAJOR
**File**: `services/market-ingestion/src/market_ingestion/infrastructure/adapters/providers/yahoo.py:84-97`
**Issue**: `run_in_executor(None, ...)` with no timeout wrapping. yfinance makes synchronous HTTP requests that can hang indefinitely. A hung thread permanently occupies an executor slot.
**Fix**: Wrap with `asyncio.wait_for(..., timeout=30.0)`. Note: this unblocks the event loop but does not kill the thread.
