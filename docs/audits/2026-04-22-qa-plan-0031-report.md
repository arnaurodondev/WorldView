## Issues — Full Investigation

### F-001: D-004 intel commit failure — committed offset, lost intel writes (MAJOR)

**Severity**: MAJOR
**Confidence**: HIGH
**Flagged by**: Distributed Systems

**File**: `services/nlp-pipeline/src/nlp_pipeline/infrastructure/messaging/consumers/article_consumer.py:458-481`

**Root Cause Analysis**:
- **What**: When `nlp_session.commit()` succeeds (line 459) but `intel_session.commit()` fails (line 472), the error is caught and logged but NOT re-raised (line 479). The Kafka offset is then committed because `process_message` returns normally.
- **Why**: The D-004 fix intentionally prioritizes nlp consistency (the critical path) over intel writes (provisional entity queue, which is supplementary). The plan explicitly accepts eventual consistency for intel writes.
- **When**: Intel commit failure — rare (requires PostgreSQL connection loss or deadlock specifically on `intelligence_db` while `nlp_db` remains healthy).
- **Where**: `article_consumer.py:468-481` — the intel commit try/except block.
- **History**: This is the intentional design from PLAN-0031 Wave B-3 (T-B-3-01). The previous code (D-004 bug) had the inverse problem: intel committed before nlp, creating orphaned intel writes.

**Impact**:
- **Immediate**: Provisional entity queue entries for that article are lost. Entity resolution for mentioned entities will still happen on future articles mentioning the same entities.
- **Blast radius**: Limited to `provisional_entity_queue` — S6 core artifacts (sections, chunks, embeddings, routing) are safely committed.
- **Data risk**: Low — provisional queue is supplementary (triggers eventual entity creation). Entities will be created when next mentioned.
- **User impact**: None visible — entity creation is delayed, not prevented.

**Recommended Option**: Accept as-is. The plan explicitly documents this trade-off: "intel writes are idempotent on retry thanks to the UNIQUE constraint on (normalized_surface, mention_class)." The idempotency guard at line 198 prevents re-processing, but since intel writes are supplementary and entities will be created on next mention, this is acceptable for a thesis project.

**Requires decision**: YES — confirm acceptance of this documented trade-off.

### F-002: HALF_OPEN circuit breaker allows multiple concurrent probes (MAJOR)

**Severity**: MAJOR
**Confidence**: HIGH
**Flagged by**: Distributed Systems

**File**: `services/rag-chat/src/rag_chat/application/pipeline/circuit_breaker.py:60-72`

**Root Cause Analysis**:
- **What**: When the OPEN state key TTL expires, `is_open()` returns `False` for ALL concurrent callers. Multiple requests can simultaneously probe a degraded source.
- **Why**: The `is_open()` check only tests key existence — no atomic SETNX to claim a single probe slot.
- **When**: During the OPEN → HALF_OPEN transition (after `cool_down_seconds` expire), under concurrent request load.
- **Where**: `circuit_breaker.py:60-72` — `is_open()` method.

**Impact**:
- **Immediate**: N concurrent requests all probe a potentially degraded source instead of just 1.
- **Blast radius**: Limited to the S8 RAG pipeline. Each probe still has a 5s timeout.
- **User impact**: Slightly higher latency during probe phase if source is still down (N×5s timeout budget consumed instead of 1×5s).

**Solution Options**:

**Option A: Accept as-is**
- The 5s timeout protects against prolonged degradation. The breaker re-trips correctly after failed probes.
- **Effort**: None. **Risk**: Low.

**Option B: Add SETNX probe lock**
- `SET rag:cb:{source}:probe_lock 1 NX EX 10` — only the winner proceeds; others see OPEN.
- **Effort**: Low. **Risk**: Low.

**Recommended Option**: **Option A** — accept for thesis scope. The burst-probe behavior is benign (all probes timeout safely) and the breaker re-trips correctly.

**Requires decision**: YES — confirm Option A.

### F-003: TOCTOU idempotency check in article consumer (MAJOR)

**Severity**: MAJOR
**Confidence**: MEDIUM
**Flagged by**: Distributed Systems

**File**: `services/nlp-pipeline/src/nlp_pipeline/infrastructure/messaging/consumers/article_consumer.py:196-203`

**Root Cause Analysis**:
- **What**: The idempotency check uses a separate session (`check_session`) that is closed before the backpressure semaphore is acquired. Two concurrent deliveries of the same message could both pass the check.
- **Why**: The check is an optimization (skip before acquiring expensive semaphore slot), not the primary idempotency guard.
- **When**: Concurrent re-delivery of the same Kafka message during rebalance.

**Impact**:
- **Immediate**: The second execution would hit DB-level CONFLICT guards (section/chunk primary keys). The `routing_decisions` table should have a UNIQUE on `doc_id` as the real safety net.
- **Data risk**: None if DB constraints exist. Duplicate NLP processing wastes CPU/ML compute but doesn't corrupt data.

**Recommended Option**: Verify `routing_decisions` has UNIQUE on `doc_id`. If yes, this is a documented optimization with DB as the true guard. No code change needed.

**Requires decision**: YES — verify constraint exists.

### F-004: CB pipeline uses `transaction=False` (MINOR)

**Severity**: MINOR
**Confidence**: HIGH
**Flagged by**: Architecture, Distributed Systems (cross-agent signal)

**File**: `services/rag-chat/src/rag_chat/application/pipeline/circuit_breaker.py:81,97`

**Issue**: Both `record_success()` and `record_failure()` use `pipeline(transaction=False)`. The ZADD+ZREMRANGEBYSCORE+ZCARD+EXPIRE sequence is not atomic under MULTI/EXEC, allowing concurrent calls to see slightly stale counts.

**Fix**: Change to `pipeline(transaction=True)` for atomic MULTI/EXEC. Near-zero performance impact for 4 commands.

**Auto-fixable**: YES
