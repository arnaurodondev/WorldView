## Issue F-001: S10 InternalJWTMiddleware accepts unverified JWTs when JWKS unavailable (CRITICAL)

### Summary
When the S10 InternalJWTMiddleware cannot load the public key from S9's JWKS endpoint (startup failure, network issues), it falls back to decoding JWTs WITHOUT signature verification. Any attacker-crafted JWT would be accepted as valid, setting arbitrary user_id, tenant_id, and role claims.

### Severity / Confidence
**Severity**: CRITICAL
**Confidence**: HIGH
**Flagged by**: Security, Distributed Systems, Architecture (3/5 agents independently)

### Root Cause Analysis
- **What**: `services/alert/src/alert/infrastructure/middleware/internal_jwt.py:143-159` — when `self._public_key is None`, the middleware calls `jwt.decode(token, options={"verify_signature": False})`
- **Why**: Intentional graceful degradation for E2E testing and startup transients, as documented in the code comments
- **When**: During service startup before JWKS is fetched, or if S9 is unreachable
- **Where**: S10 Alert service infrastructure layer
- **History**: Pre-existing pattern, NOT introduced by S9-1..S9-3. The S9-2 wave modified this file to add WebSocket `?token=` support but did not introduce the bypass logic.

### Evidence
```python
if public_key is None:
    try:
        payload = jwt.decode(token, options={"verify_signature": False})
        request.state.tenant_id = payload.get("tenant_id", "")
        request.state.user_id = payload.get("sub", "")
        request.state.role = payload.get("role", "")
    except jwt.DecodeError:
        request.state.tenant_id = ""
```
- **File**: `services/alert/src/alert/infrastructure/middleware/internal_jwt.py:143-159`

### Impact
- **Immediate**: Authentication bypass if S9 JWKS is unreachable during S10 startup
- **Blast radius**: S10 alert service only (other services have the same middleware pattern)
- **Data risk**: Attacker could read/acknowledge alerts for any user
- **User impact**: Not visible unless exploited; silent security gap

### Solution Options

#### Option A: Fail-closed with 503
**Description**: Return HTTP 503 when public key is unavailable instead of accepting unverified tokens.
**Changes required**:
- [ ] `services/alert/src/alert/infrastructure/middleware/internal_jwt.py` — Replace unverified decode path with 503 response
- [ ] All other services with same middleware — Apply same fix
**Benefits**: Eliminates authentication bypass entirely; follows security best practice (fail-closed)
**Drawbacks**: Services become unavailable during S9 restart/JWKS transient failures; E2E tests need S9 running
**Effort**: Low | **Risk**: Medium (affects availability during deploys)

#### Option B: Add environment flag for bypass
**Description**: Only allow unverified decode when `INTERNAL_JWT_SKIP_VERIFICATION=true` (E2E/dev only).
**Changes required**:
- [ ] Add env flag to all services with InternalJWTMiddleware
- [ ] Default to `false` (fail-closed) in production
**Benefits**: Secure in production; flexible for testing
**Drawbacks**: Slight complexity increase; risk of flag being left on in production
**Effort**: Low | **Risk**: Low

### Recommended Option
**Option B** — Environment flag provides security in production while preserving E2E test flexibility. The flag default must be `false`.

### Verification Steps
- [ ] Set `INTERNAL_JWT_SKIP_VERIFICATION=false` and verify S10 returns 503 when JWKS is down
- [ ] Set `INTERNAL_JWT_SKIP_VERIFICATION=true` and verify E2E tests pass without JWKS

## Issue F-002: Missing downstream error handling tests (CRITICAL)

### Summary
All 40 proxy route tests only mock downstream services returning HTTP 200. No tests verify behavior when downstream services return 500/502/503 or timeout. While the proxy routes correctly forward error responses (Response(content=resp.content, status_code=resp.status_code)), this behavior is untested.

### Severity / Confidence
**Severity**: CRITICAL
**Confidence**: HIGH
**Flagged by**: QA/Test

### Root Cause Analysis
- **What**: `tests/test_s9_wave1_proxy.py`, `test_s9_wave2_proxy.py`, `test_s9_wave3_proxy.py` — all `_mock_response()` calls use status 200 or 201
- **Why**: Test plan focused on happy-path proxy behavior; error paths were not specified in acceptance criteria
- **When**: Always — any downstream outage will trigger untested code paths

### Impact
- **Immediate**: No regression safety net for error handling
- **Blast radius**: All 27 new proxy endpoints
- **Data risk**: None (proxy forwards errors transparently)

### Recommended Option
Add 10-15 error path tests covering downstream 500/503 for representative endpoints. Defer to T-1 wave or separate test PR.

### Verification Steps
- [ ] Add `test_ohlcv_proxy_downstream_500` — verify 500 is forwarded
- [ ] Add `test_heatmap_mixed_success_failure` — 5 sectors OK, 6 fail

## Issue F-003: WebSocket token appears in server logs via URL query param (MAJOR)

### Summary
The WS token issued by `GET /v1/auth/ws-token` is used in `ws://alert:8010/api/v1/alerts/stream?token={jwt}`. This JWT appears in reverse proxy and server access logs. While the 30s TTL limits exposure, log-based token extraction is possible within the window.

### Severity / Confidence
**Severity**: MAJOR
**Confidence**: HIGH
**Flagged by**: Security, Distributed Systems

### Root Cause Analysis
- **What**: Browser WebSocket API limitation — cannot set custom headers on `new WebSocket(url)`
- **Why**: Standard browser security model; no alternative for WS auth besides URL params or cookies
- **When**: Every WebSocket connection establishment
- **History**: Known design trade-off documented in jwt_utils.py:76-79

### Recommended Option
Accept as known trade-off. The 30s TTL is appropriate. Consider nginx log masking of `?token=` parameters if feasible. No code change needed — document in security model.

## Issue F-004: ws-token endpoint missing claim validation (MAJOR)

**Severity**: MAJOR
**File**: `services/api-gateway/src/api_gateway/routes/auth.py:529-531`
**Issue**: `issue_ws_jwt()` is called with `user.get("sub", user.get("user_id", ""))` without validating that required claims exist. If both keys are missing, a 30s token is issued with empty user_id.
**Fix**: Add explicit validation before token issuance:
```python
user_id = user.get("sub") or user.get("user_id")
tenant_id = user.get("tenant_id")
if not user_id or not tenant_id:
    return JSONResponse(status_code=401, content={"error": "incomplete_auth_claims"})
```

## Issue F-006: Heatmap _screener_for_sector may crash on malformed JSON (MAJOR)

**Severity**: MAJOR
**File**: `services/api-gateway/src/api_gateway/clients.py:163`
**Issue**: `resp.json()` is called without exception handling after the status code check passes. If S3 returns 200 with non-JSON body, this crashes.
**Fix**: Wrap `resp.json()` in try-except:
```python
try:
    return cast("dict[str, Any]", resp.json())
except Exception:
    return {"error": True, "sector": sector}
```

## Issue F-010: Economic calendar overwrites user query params (MAJOR)

**Severity**: MAJOR
**File**: `services/api-gateway/src/api_gateway/routes/proxy.py:579`
**Issue**: `params={"type": "economic", **dict(request.query_params)}` — if frontend sends `?type=custom`, the dict unpacking overwrites the hardcoded `"type": "economic"`.
**Fix**: Filter out `type` from user params:
```python
params = {"type": "economic"}
params.update({k: v for k, v in dict(request.query_params).items() if k != "type"})
```
