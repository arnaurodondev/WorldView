## 2. Attributing the Qwen3-235B usage — is 55k expected, or waste?

**The operator's Qwen3-235B is dominated by S6 EXTRACTION, not entity
descriptions.** This corrects the going-in hypothesis.

- **S6 nlp-pipeline extraction** runs `Qwen/Qwen3-235B-A22B-Instruct-2507` as the
  deep tier (`NLP_PIPELINE_EXTRACTION_API_MODEL_ID`, confirmed on the live pod;
  default set in `libs/ml-clients/src/ml_clients/model_registry.py:71`), fallback
  `openai/gpt-oss-20b`, `reasoning_effort=medium`, `max_tokens=8192`. It is driven
  by the multi-year **news backfill firehose** — ~4.5k extraction calls/day at
  ~6,300 input tokens each. Over ~a week this is the ~55k the operator saw.
- **Entity descriptions are NOT on 235B.** The KG config already migrated both
  `description_deepinfra_model_id` **and** `deepinfra_extraction_model_id` to
  `deepseek-ai/DeepSeek-V4-Flash-Thinking` (config.py:112/263). The stale comment
  at config.py:248 still says "Qwen3-235B-A22B primary" — code disagrees. In the
  2-day window there are **zero** description-capability rows; descriptions barely
  ran (90-day cadence, see below).

**Is it waste?** No call-count waste:
- nlp Qwen extraction: **100% success**, only 83 timeouts (fell to gpt-oss-20b).
- KG extraction: 1,059 / 39,123 fail = **2.7%** — normal, not a retry storm.
- The description worker (`definition_refresh.py`) already has **SHA-256
  content-hash dedup** (skips re-embed when `source_text` unchanged AND an
  embedding exists) and **`next_refresh_at = now + 90 days`**. No re-describe loop.

**The waste is model-tier over-provisioning, not volume.** Qwen3-235B
(~$0.24/Mtok effective) is the most expensive per-token model in the stack, used
as the *primary* extractor on a high-volume firehose whose fallback
(gpt-oss-20b, $0.06/Mtok) is 4× cheaper.

## 3. Is internal cost tracking accurate?

**Much improved** over the prior "S6 logs 100% $0 / S8 69% $0" finding — the
PLAN-0117 provider-cost path is now live and dominant:
- nlp_db: 18,021 / 18,383 rows use `cost_source="provider"` (DeepInfra
  self-reported, authoritative). Real $ captured.
- intel_db: 37,918 rows `cost_source="provider"`; the 1,059 `$0` are failed calls.
- The $0 rows that remain are legitimately local (Ollama/GLiNER) or failures.

**Remaining gaps the operator should know about:**
1. **KG model attribution is broken** (FIXED here). `intelligence_db.llm_usage_log`
   logged `model_id="deepinfra"` for every extraction row — the transport
   provider, not the model. Root cause: `fallback_chain.py:245`
   `getattr(client, "model_id", provider)` fell through because
   `DeepSeekExtractionAdapter` exposed no public `model_id`. You could not tell
   V4-Flash-Thinking from a 235B call in the KG DB. **Fixed** (§4).
2. **Llama-3.1-8B-Turbo rows log 0 tokens** (6,443 extraction + 2,482
   classification rows, tokens_in=out=0 but cost>0). Cost is captured via the
   provider path, but token COUNTS are unreliable for the 8B relevance/intent
   path — the adapter doesn't surface `usage.prompt_tokens`. Deferred (§5, D-3).
3. **Stale price matrix for Qwen3-235B** (FIXED here). Entry was $0.071/$0.10 —
   ~3× below the observed $0.24/Mtok blended and the repo's own documented
   $0.13/$0.60 list rate. Only affects the *fallback* path (when DeepInfra omits
   `usage.estimated_cost`), but would silently 3×-undercount there. **Fixed** (§4).
4. **rag_db `aggregate` duplicate rows**: 174 `aggregate` rows ($0) mirror the 174
   `provider` rows for the same tokens — token double-count if you `SUM(tokens)`
   without filtering `cost_source`. Cost is not double-counted. Deferred (§5, D-4).

Net: **$ totals are now trustworthy per-service; per-MODEL attribution inside KG
was not** (now fixed). Token counts are reliable except the Llama-8B path.

## 5. Deferred cost-reduction plan (prioritized by $/effort)

**D-1 — Right-size the S6 extraction model (BIGGEST LEVER). Est. save ~$150–180/mo.**
Qwen3-235B (~$0.24/Mtok) is the primary extractor on the firehose. Its own
fallback, `openai/gpt-oss-20b`, is $0.06/Mtok (4×) and already wired. Options:
- `gpt-oss-20b` primary → nlp extraction ~$219/mo → **~$55/mo** (save ~$165/mo).
- `gpt-oss-120b` ($0.09/$0.45) as a quality-balanced middle → ~$90/mo.
- `deepseek-ai/DeepSeek-V4-Flash` ($0.14/$0.28) → ~$120/mo.
**Do NOT flip blindly** — extraction quality drives KG relation precision (memory:
relation precision gates). Gate the swap on the existing harness:
`scripts/eval/extraction_quality_eval.py --models "Qwen/Qwen3-235B-A22B-Instruct-2507,openai/gpt-oss-120b,openai/gpt-oss-20b"`
and keep 235B as the escalation tier. This is a config/env change
(`NLP_PIPELINE_EXTRACTION_API_MODEL_ID`) once the eval clears — no code.

**D-2 — Trim extraction input tokens. Est. save ~$30–60/mo.**
Avg 6,300 input tokens/extraction × 9k/day is the cost base (input dominates:
56.8M in vs 3.7M out). Audit the extraction prompt + article-context window;
DeepInfra KV-prefix caching is already on (persistent client) but only helps the
static system-prompt prefix. Cap/whitelist the article context length. Quality-
sensitive — measure recall on the eval set first.

**D-3 — Capture Llama-8B token counts.** Wire `usage.prompt_tokens` /
`completion_tokens` through the 8B relevance/intent/classification adapter so
`llm_usage_log` stops logging 0 tokens for ~9k rows/2d. No $ savings; restores
token-count trust for the cheapest-but-highest-volume path.

**D-4 — Filter/rename rag_db `aggregate` rows** so `SUM(tokens_in)` dashboards
don't double-count. Cosmetic; a `WHERE cost_source <> 'aggregate'` in the cost
use-cases is the minimal fix.

**D-5 — Confirm the description model migration is intended** and fix the stale
config.py:248 comment (says 235B; code is V4-Flash-Thinking). If descriptions
should stay on a small model, no action beyond the comment. If quality regressed,
`description_deepinfra_model_id` is the knob (currently cheap already).

**D-6 — Refresh the whole price matrix** against the current DeepInfra list
(gpt-oss-120b/20b/Qwen3-235B all tagged "verify at OQ-1"). Reconcile matrix
fallback rates to observed provider $/Mtok quarterly.
