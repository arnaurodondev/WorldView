## Issues — Full Investigation

## Issue A-F-001: TanStack cache-key drift between usePortfolioMetrics and PortfolioSwitcher

### Summary
The new `PortfolioSwitcher` consumes the central `qk.portfolios.list()` query key from `lib/query/keys.ts`, but three older consumers (`usePortfolioMetrics`, `WorkspacePortfolioPanel`, `PortfolioSummary`) still hard-code the bare `["portfolios"]` tuple. TanStack treats these as different cache entries, so authentication-time the platform fires **two** separate `GET /v1/portfolios` calls instead of one shared fetch.

### Severity / Confidence
**Severity**: CRITICAL
**Confidence**: HIGH
**Flagged by**: Architecture reviewer (A-F-001)

### Root Cause Analysis
- **What**: Bare key `queryKey: ["portfolios"]` in 3 files vs factory `qk.portfolios.list()` returning `[QK_VERSION, "portfolios", "list"]` in 1 new file (PortfolioSwitcher). The `QK_VERSION` prefix ("v2") + the extra `"list"` segment guarantee non-equal tuples → no cache dedup.
- **Why**: W1 added the PortfolioSwitcher using the modern `qk.*` factory pattern (correct), but the consumer migration the plan §5 assumed had been done (`qk.portfolios.list() (existing)`) never actually happened — the legacy bare keys lived on across the codebase.
- **When**: Every authenticated page render where both the TopBar (which mounts PortfolioSwitcher) and a dashboard/workspace widget are on screen — i.e. every shell-rendered page.
- **Where**: Cross-cutting between hook + 2 widgets + 1 new component.
- **History**: Pre-W1 the platform shipped with the bare key everywhere and it didn't matter (every consumer agreed). PRD-0089 §5 documented the factory shape but skipped the migration step.

### Evidence
```
apps/worldview-web/components/workspace/WorkspacePortfolioPanel.tsx:31:    queryKey: ["portfolios"],
apps/worldview-web/components/dashboard/PortfolioSummary.tsx:52:    queryKey: ["portfolios"],
apps/worldview-web/hooks/usePortfolioMetrics.ts:82:    queryKey: ["portfolios"],
apps/worldview-web/components/shell/PortfolioSwitcher.tsx:74:    queryKey: qk.portfolios.list(),
```

### Impact
- **Immediate**: Authenticated pages fire 2× `/v1/portfolios` requests. Doubled S9/S1 load per authenticated session (rate impact: low — single call, infrequent — but a silent regression nonetheless).
- **Blast radius**: 3 widgets + 1 hook on every shell-rendered page.
- **Data risk**: None — both fetches return the same data.
- **User impact**: Imperceptible today; growing as more consumers are added.

### Solution Options

#### Option A: Migrate the 3 legacy consumers to `qk.portfolios.list()`
**Changes required**:
- [ ] `hooks/usePortfolioMetrics.ts` — `["portfolios"]` → `qk.portfolios.list()` + import qk
- [ ] `components/workspace/WorkspacePortfolioPanel.tsx` — same
- [ ] `components/dashboard/PortfolioSummary.tsx` — same
**Effort**: Low (~10 LOC each, mechanical)
**Risk**: Low — TanStack's existing cache for the old key will be orphaned (no harm) and re-populated under the new key

#### Option B: Add an ESLint rule banning inline `queryKey` literals
**Drawbacks**: Larger surface change; rule already exists per `lib/query/keys.ts` doc but is incompletely enforced.

### Recommended Option
**Option A** — applied during this QA pass (commit `bcfd6717`).

### Verification Steps
- [ ] grep `queryKey: ["portfolios"]` returns 0 hits in production code
- [ ] In dev tools Network tab, /v1/portfolios fires once per authenticated session, not twice

## Issue A-F-002: 3 widgets still pick `portfolios[0]` instead of respecting ActivePortfolioContext

### Summary
The W1.1 F-002 fix wired `usePortfolioMetrics` to consume `useActivePortfolio()` so the new PortfolioSwitcher chip controls the TopBar PortfolioRail. But three other widgets that pick a portfolio still hard-code `portfolios?.[0]`, ignoring the user's selection. From the user's POV the chip flips only the TopBar — Dashboard and Workspace panels stay on the first portfolio regardless.

### Severity / Confidence
**Severity**: CRITICAL
**Confidence**: HIGH
**Flagged by**: Architecture reviewer (A-F-002), corroborated by QA reviewer (F-003 — context branch not tested in usePortfolioMetrics)

### Root Cause Analysis
- **What**: 3 sites pick the first portfolio without checking the active-portfolio context:
  - `components/workspace/WorkspacePortfolioPanel.tsx:37` — `firstPortfolioId = portfolios?.[0]?.portfolio_id`
  - `components/dashboard/PortfolioSummary.tsx:58` — `firstPortfolio = portfolios?.[0]`
  - `components/dashboard/HoldingsMoversWidget.tsx:83` — `getPortfolios()` consumer that subsequently picks first
  - `components/chat/SlashCommandCard.tsx:207` — `portfolio = portfolios?.[0] ?? null`
- **Why**: W1.1 F-002 documented the wiring pattern in usePortfolioMetrics and said other consumers were "deferred to W4" but the user's direct feedback was that the chip "does nothing" — meaning the W4-deferral assumption is too conservative. The 3 widgets are the highest-traffic consumers; deferring them defeats the purpose of W1.1 F-002.
- **When**: Always — every Dashboard / Workspace page render.
- **Where**: 4 component files.
- **History**: Pre-W1 there was no ActivePortfolio concept; every consumer used `portfolios[0]` as "the user's portfolio" since most users had one. PortfolioSwitcher introduced multi-portfolio selection; consumers weren't updated.

### Evidence
```
components/workspace/WorkspacePortfolioPanel.tsx:37  const firstPortfolioId = portfolios?.[0]?.portfolio_id;
components/dashboard/PortfolioSummary.tsx:58         const firstPortfolio = portfolios?.[0];
components/dashboard/HoldingsMoversWidget.tsx:27..83 (relies on `getPortfolios()` then picks first)
components/chat/SlashCommandCard.tsx:207             const portfolio = portfolios?.[0] ?? null;
```

### Impact
- **Immediate**: Switching the chip flips TopBar rail but leaves Dashboard widgets, Workspace portfolio panel, and chat slash command targeting the FIRST portfolio.
- **Blast radius**: Every page that surfaces portfolio-scoped data.
- **Data risk**: None — just stale-selection UX.
- **User impact**: HIGH — direct contradiction of W1.1 F-002's user-visible intent.

### Solution Options

#### Option A: Wire each of the 4 widgets to `useActivePortfolio()` with `portfolios[0]` fallback
**Changes required (per widget, ~6 LOC each)**:
- [ ] Read `const { activePortfolioId } = useActivePortfolio();`
- [ ] Validate `persistedPortfolioExists = activePortfolioId && portfolios.some(...)`
- [ ] `firstPortfolioId = persistedPortfolioExists ? activePortfolioId : portfolios?.[0]?.portfolio_id`
- [ ] Same fallback pattern as usePortfolioMetrics.ts:93-98

Worth a small refactor: extract the selection-resolution into a tiny hook (`useResolvedPortfolioId(portfolios)`) so all consumers share the logic.

**Effort**: Low (~25 LOC for the hook + 4 × ~3 LOC for adoption)
**Risk**: Low — fallback to `portfolios[0]` preserves pre-W1 behaviour when context is unset

#### Option B: Defer to W4 (the original W1.1 plan)
**Drawbacks**: User has already noticed and reported it as "the chip does nothing" — deferring is now actively user-hostile.

### Recommended Option
**Option A** — applied during this QA pass (commit `bcfd6717`).

### Verification Steps
- [ ] Switch chip to non-default portfolio → Dashboard PortfolioSummary header value updates
- [ ] Switch chip → WorkspacePortfolioPanel reflects new portfolio's holdings
- [ ] grep `portfolios\?\.\[0\]\?\.portfolio_id` in components/dashboard + components/workspace returns 0 fresh hits (only present as fallback inside the new helper)
