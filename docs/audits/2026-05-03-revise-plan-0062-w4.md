### I-5: T-W4-B-03 "drop the bullet" path can produce a `BriefSection` with `bullets=[]`, violating the existing `min_length=1` schema constraint
**Severity**: IMPORTANT
**Location**: Plan T-W4-B-03 + T-W4-A-02 schema delta

**Issue**: Existing `BriefSection.bullets: list[str] = Field(..., min_length=1, max_length=8)` (schemas.py:119, verified). Plan T-W4-A-02 changes the inner type to `BriefBullet` but **does not change the `min_length=1` constraint on the section's bullets list**. T-W4-B-03's `_backfill_uncited_bullets` says: *"If `context_citations` is also empty, drop the bullet entirely. If the section becomes empty after dropping, drop the section."*

That "drop the section" path covers it — but only if the function correctly empties `bullets` first, then drops the parent in a second pass. If anywhere the code does `BriefSection(title=..., bullets=[])` (pre-drop), Pydantic raises `ValidationError` immediately. Worse, the function is described as in-place mutation in §6 ("returns `list[BriefSection]`"), but Python's frozen Pydantic models won't accept mutation — the function must build new objects.

**Required fix**:
- (a) Have T-W4-A-02 either keep `min_length=1` (and require `_backfill_uncited_bullets` to construct via list comprehension that filters BEFORE constructing `BriefSection`) OR relax to `min_length=0` and add a separate post-step to drop empty sections.
- (b) Add unit test: `BriefSection(title="t", bullets=[])` raises ValidationError — to lock down which interpretation we ship.

The plan currently leaves this ambiguous, and ambiguity around schema-violating intermediate states is a high-bug-likelihood pattern.

### I-6: Cache-key bump from `v1` → `v2` is asymmetric — only the writes change in the plan, not the reads
**Severity**: IMPORTANT
**Location**: Plan T-W4-C-01

**Issue**: Plan T-W4-C-01 says: *"Update the cached-response read path: when reading old `v1` keys would happen, the code now reads `v2` keys, so legacy entries are simply ignored and a fresh generation runs."*

Reality (verified `services/rag-chat/src/rag_chat/api/routes/public_briefings.py:87, 168`): the route uses **the SAME variable** `cache_key` for both read and write paths (lines 87, 92, 147 for morning; 168, 173, 235 for instrument). Bumping `cache_key` once changes both — that part is fine.

What the plan misses: the `try`/`except` around the cache read (lines 99-101) catches **all** exceptions (`except Exception`) and warns + falls through. So legacy `v1` entries that successfully decode-as-JSON but don't match the new `v2` shape will:
1. NOT crash the cache read (because `data["cached"] = True` followed by `PublicBriefingResponse(**data)` is **outside** the try/except). It IS inside the outer try/except — re-reading lines 90-101 confirms `PublicBriefingResponse(**data)` happens INSIDE the same try block. So a Pydantic ValidationError on the `v1` shape will be caught and warning-logged.

**HOWEVER**: This is reading the *new* key `v2`, so by definition the only data there is `v2`-shaped. The plan's mitigation is correct — but only because the bump means stale `v1` keys are never read. Good in practice.

**The real issue**: There is **no test** in the plan that exercises a stale `v1`-shape value at the new `v2` key (which can only happen if a deploy is rolled back mid-flight). Recommend adding a test in T-W4-C-01: *"Stale-shape cached value at v2 key → cache_read_failed warn-log + fresh generation."* Low effort, high regression value.
