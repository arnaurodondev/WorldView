**Issue**: Iter-1 F-005 was marked MAJOR — the RealizedPnL chart is a degenerate
2-point line. Iter-1 fix commit did not address it and the plan does not
document the limitation. The chart still ships with a misleading visualization
(diagonal between 0 and total) but no in-UI disclaimer and no plan-level
acknowledgement of the gap. Either:
- (a) add a plan-level note marking T-D-4-03 as PARTIAL with a follow-up
  reference (PLAN-0054 or similar), OR
- (b) render an in-UI "cumulative trace pending backend day-series"
  disclaimer below the chart so users understand the diagonal is an
  approximation.

This was the only iter-1 MAJOR not addressed by the fix commit. It does not
block ship — the chart itself is correct (the diagonal is mathematically the
correct cumulative for a single endpoint pair) — but the user expectation
gap is real.

**Fix**: 1-line plan annotation OR a 1-line component disclaimer string.

### F-iter2-003 (MINOR)
**File**: `apps/worldview-web/components/feedback/FeedbackButton.tsx:31`
**Issue**: After the F-010 fix, the line reads `useAuth();` — invoked purely
for side-effect with no destructured return. The retained comment on lines
27-30 explains the intent ("retained for future surface-tagging"), but the
bare call is a code smell — most ESLint configs flag this as
`@typescript-eslint/no-unused-expressions`. Current TSC pass is clean (the
hook has side effects so it's a valid statement), but a future contributor
will likely either remove the call (breaking when surface-tagging lands) or
add a useless lint-disable comment.

**Fix**: Either delete the call entirely (hook does no side-effect work
relevant to this component) and remove the explanatory comment, OR
destructure and `void user` to make the intent explicit:
```ts
const { user: _user } = useAuth(); // reserved for future pre-fill
```

### F-iter2-004 (NIT)
**File**: `apps/worldview-web/components/feedback/ScreenshotCapture.tsx:14-23`
**Issue**: The component's docstring still says:
> "the modal must POST the data URI to a presigned-URL upload route OR drop
> it … Until that upload route ships … we surface the preview but DO NOT
> include the URL in the submission"

This is now incorrect. The F-003 fix in `FeedbackModal.tsx:206-211` DOES
forward the data URI inline as a JSON field with a 1MB cap. The stale
docstring will mislead future maintainers about whether the screenshot
reaches the backend.

Additionally, when a screenshot exceeds 1MB the truncation flag
`screenshot_data_uri_truncated` is set on the submission but no UI
affordance tells the user their image was dropped. They see a preview,
press submit, and silently lose the bytes — same trust problem the
original F-003 was about, just at the size-limit edge.

**Fix**:
1. Update the ScreenshotCapture docstring to match reality (data URI is
   forwarded inline, capped at 1MB).
2. Surface the truncation case in the UI — when the captured `dataUrl.length
   > 1_048_576`, show a `text-warning` line under the preview: "Screenshot
   too large (>1MB) — submit will exclude image. Try the blur option to
   reduce size." Today it ships silently.

Niche edge case (html2canvas at scale=1 typically produces 200-500 KB), so
filed NIT not MAJOR. Real users at 4K resolution may hit it.
