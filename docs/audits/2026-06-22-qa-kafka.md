## Detail

### Wedge-class verification (the failure mode we hit before)
- All 27 consumer groups listed; every one has a CONSUMER-ID. No group is EMPTY with lag.
- Consumer-log scan (`--since 6–8m`, post-deploy) for `Connection setup timed out|transport failure|FetchRequest.*timed out|Disconnected|Receive failed`: **0 hits.**
- Broker log (`--since 15m`): one clean rebalance wave at 10:30:10–10:30:16Z, all groups `Stabilized ... with N members` + `Assignment received from leader`. No repeated rebalance churn afterward.
- The only `CONFWARN` lines are `metadata.request.timeout.ms is deprecated` — cosmetic librdkafka config warning, not a fault.

### #1 prediction-markets (MEDIUM, not HIGH)
Consumer `worldview-market-data-prediction-market-consumer-1` is actively emitting `prediction_market_consumer.materialised` at ~2/sec. Lag is flat at ~35k because the Polymarket producer writes to the single-partition `market.prediction.v1` at least as fast. This is a **capacity** issue (1 partition, no parallelism) not a wedge or broker fault. If real-time freshness of prediction snapshots matters, this group will never catch up without more throughput.

### #3 nlp-pipeline-group p0 (MEDIUM — overlaps Postgres agent)
`article-consumer-1` repeatedly hits:
```
asyncpg.exceptions.QueryCanceledError: canceling statement due to statement timeout
INSERT INTO provisional_entity_queue ... ON CONFLICT (normalized_surface, mention_class) ...
```
13 timeouts vs 8 processed in 6m on that consumer; consumer-2 (different partition) processes cleanly with 0 timeouts. The Kafka-visible symptom is that partition 0's offset barely advances. **Root cause is Postgres-side (statement_timeout under load) — owned by the Postgres/DB QA agent**; flagged here only because it manifests as Kafka lag. The consumer is not wedged; it is throttled by downstream DB cancellation. (Failed mentions are gracefully downgraded to `unresolved`, so no DLQ growth.)

### DLQ
`GetOffsetShell` on all 5 `*.dead-letter.v1` topics → offset 0. No dead-letter accumulation anywhere. `failed_tasks` Postgres tables intentionally NOT inspected here (owned by Postgres agent).

### Broker / Schema Registry
- `--under-replicated-partitions` and `--unavailable-partitions`: empty (clean).
- `docker stats`: kafka 4.41% CPU, 780 MiB/6 GiB. No GC/queue-overload entries in 15m of broker log.
- Schema Registry: 4 benign 404s, 59 successful registrations, no compatibility failures.
