### B-2 — `canonical_entities` view in `nlp_db` is fictional

**Category**: Stale codebase assumption (silent bug class)
**Locations**: PLAN-0064 §3 AD-W6-3, §7 Wave 2 T-W6-2-02 SQL Query 3 (lines 434-444), §13 OQ-W6-A.
**Evidence**:
- `services/nlp-pipeline/alembic/versions/0001_create_nlp_schema.py` defines `chunks`, `entity_mentions`, `chunk_entity_mentions`, `mention_resolutions`, `document_entity_stats`, `routing_decisions`, `sections`, `chunk_embeddings`, `section_embeddings`. **No** `canonical_entities`.
- `grep -rn "CREATE VIEW\|CREATE MATERIALIZED" services/nlp-pipeline/alembic/` returns only the `document_source_llm_latest` materialised view in 0013. No `canonical_entities` view.

- nlp-pipeline messaging consumers: only `article_consumer.py` and `watchlist_consumer.py` exist; no `entity_canonical_consumer.py` or similar.

  Option (b) is simpler and consistent with how titles are fetched. Pick one; the current plan has both as "primary" and "fallback" and silently ships option (a) which doesn't work.
**Bonus**: §13 OQ-W6-A claims this is "MEDIUM" severity verifiable in Wave 2 pre-read. Reality: it's BLOCKING because it changes the use-case shape (extra HTTP roundtrip + S7 client wiring), the latency budget calculation, and the "no migration owned by this plan" claim in §11.

### B-3 — Existing `RateLimitMiddleware` cannot enforce "100 search/day per tenant"

**Category**: Architecture compliance / stale assumption
**Locations**: PLAN-0064 §3 AD-W6-4 ("100 search/day per tenant"), §9 T-W6-4-01 acceptance criterion #6 ("test_search_proxy_rate_limited_after_100_in_24h"), §11 Cross-cutting "100/day key on /v1/search per tenant", §13 OQ-W6-C.
**Evidence**: `services/api-gateway/src/api_gateway/middleware.py:298-410`. `RateLimitMiddleware.__init__` takes only `max_requests: int = 100, window_seconds: int = 60`. There is one global instance constructed in `app.py:197`. The dispatch logic keys on `user_id` from `request.state.user`, falls back to IP hash for anonymous traffic. The middleware does not consult the route, does not read tenant_id, does not support multi-bucket configuration. There is a `is_public_feedback` special case that hard-codes a higher limit for `/v1/feedback/` paths — but it is a single-line if-branch, not a generic per-route capability.
**Why it matters**: The plan presents wiring as a one-line addition ("Use existing RateLimitMiddleware with the search-route key"). It is not. To honour PRD §4 NFR (100 search/day) and AD-W6-4 (per tenant), the work is at minimum:
  1. Extract a `(path_prefix, max, window_seconds, key_fn)` rule table from the middleware.
  2. Add a `tenant_id` extractor (today the middleware reads `user_id` only, not `tenant_id`).
  3. Add a 24h window for the new bucket.
  4. New tests, including verifying that the existing 100/min-per-user limit is not regressed.
This is ~0.5-1 dev-day of cross-cutting middleware work that does not exist in the plan estimate. PLAN-0064's 3.5 dev-day estimate excludes it.
**Fix**: Either (a) absorb the rate-limiter refactor into Wave 4 with explicit task breakdown (recommended — atomic with the new endpoint); (b) push it out to PLAN-0065 (W8 RLS + tier + Stripe — same workstream area) and have PLAN-0064 stop on the existing global per-user-per-minute behaviour, with an explicit acceptance gap noted. Whichever is chosen must update the plan's estimate.

**Category**: Architecture compliance / R14 boundary

### I-4 — Internal contradiction: snippet "HTML-stripped" in Wave 1 but rendered as HTML with `<mark>` in Wave 4

**Category**: Internal inconsistency (intra-plan)
**Location**: PLAN-0064 §6 T-W6-1-01 line 193: "snippet: str | None — top-ranked chunk's `ts_headline` (≤300 chars, HTML-stripped). Null when generation fails." vs §9 T-W6-4-02 line 746: "Renders each hit as a card: title, snippet (with `<mark>` highlights from `ts_headline`)" + §13 OQ-W6-E "HTML via ts_headline + DOMPurify — already decided".
**Why it matters**: If the use case strips HTML before returning, the frontend cannot render `<mark>` highlights — DOMPurify and the "dangerouslySetInnerHTML guard exists" test (line 765 `test_renders_snippet_with_marks`) become meaningless. **Fix**: Pick one. Update T-W6-1-01 description to "≤300 chars, contains only `<mark>` tags from ts_headline; sanitised by DOMPurify on the frontend".

### I-5 — Title field source: S5 `documents.title` vs S6 `document_source_metadata.published_at` — schema mismatch and 2nd HTTP call cost is unaccounted

**Category**: Internal consistency / latency NFR
**Location**: PLAN-0064 §7 T-W6-2-02 ("Title field source: documents table is in content_store_db. To return titles without a cross-DB query, the use case calls S5.POST /api/v1/documents/batch") + §4 row "POST /api/v1/documents/batch — exists, returns title, url, source_type, published_at".
**Evidence**: `services/content-store/src/content_store/api/documents.py:18` confirms the batch endpoint exists. But notice: `document_source_metadata` (in nlp_db, the table the SQL filters on for source_type and published_at) **also** has `published_at` and `source_type`. The plan therefore performs the source_type/date filter against nlp_db's `document_source_metadata` AND fetches the same fields from S5 for display. This is fine but:
  1. If the two diverge (e.g. S5 corrects a published_at after nlp_db's stale copy), filter results lie.
  2. The S5 batch call adds a ~50-200ms hop that is not budgeted in the latency NFR (≤500ms p95). For a 25-result page with 25 doc_ids, S5 must answer in well under 200ms or the budget blows.
**Why it matters**: The PRD §4 NFR table sets a hard p95 (500ms). The plan does not budget the S5 hop. If the plan's primary measurement (Wave 3 integration test latency) doesn't include the S5 hop, the acceptance gate is invalid.
**Fix**: Either (a) accept that the search query stays inside nlp_db only and **render the title from `document_source_metadata`** if you can guarantee it stays there (currently `document_source_metadata` does NOT carry a title field per `0002_add_document_source_metadata.py` — only `doc_id, source_type, published_at, ingested_at`); (b) add `title` to `document_source_metadata` (Alembic migration owned by this plan) so we can avoid the S5 hop; (c) accept the S5 hop and explicitly include it in the latency NFR test in T-W6-3-02 / T-W6-5-01. Recommend (b) — small, additive, removes a per-request roundtrip.

### I-6 — `chunks.tsv` GIN index + `chunks.doc_id` lookup may not be selective enough at MVP scale

**Category**: NFR feasibility / future risk
**Location**: PLAN-0064 §3 AD-W6-2 ("with GIN + ≤100K documents (MVP scale), p95 ≤500ms is achievable"), §4 NFR table, §7 T-W6-3-02 latency assertion.
**Evidence**: The Wave 2 SQL query plan does:
  1. `WITH ranked_chunks AS (SELECT … WHERE c.tsv @@ plainto_tsquery('english', :q))` — GIN index scan.
  2. `top_chunk_per_doc` — `DISTINCT ON (rc.doc_id) … ORDER BY rc.doc_id, rc.rank DESC` — sort.
  3. `EXISTS` correlated subqueries against `document_source_metadata` per filter (3 subqueries × N matched docs).
  Step 3 is potentially expensive: for a hot query like "apple" matching tens of thousands of chunks, the planner may turn `EXISTS` into a hash semi-join, but with three nested subqueries the cost is non-trivial. The integration test uses 1k chunks (T-W6-3-02) and the acceptance test uses 10k (T-W6-5-01); production with W2 universe expansion (S&P 500 × 200 articles each = ~100k+) will exceed this.
**Why it matters**: 500ms p95 is plausible at 1k chunks but optimistic at 100k. The plan acknowledges this in §12 (option to "reduce default page_size to 10" or "LIMIT 1000 early-cut") but does not include a fallback acceptance criterion.
**Fix**: Rewrite the query to push `document_source_metadata` joins into the CTE (single hash join), not three correlated EXISTS — and add an `EXPLAIN ANALYZE` assertion in T-W6-3-02 that ensures the plan is `Bitmap Index Scan + Hash Join`, never `Nested Loop + Index Scan` per filter. The plan already has the `EXPLAIN ANALYZE` assertion for GIN; extend it.

## NICE-TO-HAVE Issues

### N-1 — Migration ordering with PLAN-0063

PLAN-0063 §0 line 30-32 says: "the W6 plan is expected to add separate `documents.tsv` indexes". PLAN-0064 §3 AD-W6-3 explicitly says "We do **not** add a `documents.tsv` column on the S5 `documents` table". Direct contradiction in inter-plan expectations. PLAN-0064's decision is the right one (avoids new sync pipeline) but PLAN-0063 should be amended to reflect it. **Action**: open a PR in PLAN-0063 §0 noting the resolution; or include a note in PLAN-0064 §2 that "PLAN-0063 anticipates documents.tsv; this plan supersedes that expectation by aggregating chunks → documents at query time".

### N-2 — `entity_mentions.tenant_id` is nullable but not used by the search query

`0010_add_tenant_id_to_entity_mentions.py` added a nullable `tenant_id` to entity_mentions. The search query in T-W6-2-02 ignores it. Two options: (a) silently use the global view (correct for L1 — entity facets are public), (b) explicitly note that the route is tenant-agnostic and document the omission in §3 AD-W6-4 ("Tenancy: All searched content is global"). Currently (a) implicitly. Adding a one-line comment in the query and the audit doc would harden the assumption.

### N-3 — `mention_class` is plain TEXT, not an enum

T-W6-1-01 says `entity_type: str — GLiNER class`. Verified in 0001_create_nlp_schema.py: `mention_class TEXT NOT NULL`. There is no DB-level constraint on what value mention_class takes. If the seed pass (PRD §3 FR-T0-1 / PLAN-0057) introduced new GLiNER classes after the plan was written, the facet sidebar might display unfamiliar strings. Low-risk; document the ≥11 expected GLiNER classes in T-W6-4-02 frontend rendering for safe-fallback UI.

### N-4 — "Visible-feature inventory and Bloomberg theater cull" (FR-T0-3) interaction

PRD §3 FR-T0-3 mandates "Hide / remove from production routes any feature that is half-shipped or known-broken". The new `/search` page is half-shipped at the end of Wave 1-3 (returns 501 / partial functionality). PLAN-0064 §6 T-W6-1-02 places the route at `/v1/search/documents` (server-side) and registers it; PLAN-0064 Wave 4 routes the user to `/search` (frontend page). Until Wave 5 lands and PLAN-0055 universe expansion completes, the page shows "no results" for most S&P 500 tickers. Recommend: gate the frontend `/search` route behind a feature flag until acceptance passes, to avoid the "half-shipped" trap that FR-T0-3 explicitly cuts.
