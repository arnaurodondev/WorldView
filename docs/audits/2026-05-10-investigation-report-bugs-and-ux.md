## 1. ISSUE-1: Chat endpoint returns "Unable to retrieve relevant data"

### Root Cause
**This is a code wiring bug, not a data gap.** The pipeline resolves entities in Step 4 but never passes them to the ToolExecutor in Step 5.

`chat_orchestrator.py:156` correctly calls S6 entity resolution and gets back Apple and Anthropic with their entity UUIDs. But `chat_orchestrator.py:160-165` builds the ToolExecutor **without** the `entity_context=` argument:

```python
# Step 4 — entity resolution works correctly
entities = await p.resolve_entities(validated_message)  # [Apple ✓, Anthropic ✓]

# Step 5 — entity_context is NEVER passed → always None
tool_executor = self._tool_factory.for_request(
    user_id=request.user_id,
    tenant_id=request.tenant_id,
    internal_jwt=None,
    # entity_context= ← missing
)
```

Every entity-scoped tool handler in `tool_executor.py` has the same guard:

```python
if self._entity_context is not None:
    entity_id = self._entity_context.entity_id
else:
    log.warning("...", reason="no_entity_context_and_name_resolution_not_wired")
    return []  # ← fires for EVERY entity tool on EVERY request
```

Affected handlers: `_handle_get_entity_graph` (line 593), `_handle_traverse_graph` (line 694), `_handle_search_entity_relations` (line 780), `_handle_search_claims` (line 858), `_handle_get_contradictions` (line 954). All return `[]` silently.

The LLM calls one of these tools → gets `[]` → `all_tools_failed` guard at line 321 fires → "Unable to retrieve relevant data". This happens for **any** entity-scoped query, not just Apple/Anthropic.

### Evidence
- `chat_orchestrator.py:156` — `entities = await p.resolve_entities(validated_message)` resolves correctly
- `chat_orchestrator.py:160-165` — `for_request()` call is missing `entity_context=`
- `tool_executor.py:593-601` — `get_entity_graph` handler returns `[]` when `entity_context is None`
- `tool_executor.py:694-701` — `traverse_graph` handler returns `[]` when `entity_context is None`
- `tool_executor.py:780-787` — `search_entity_relations` handler returns `[]` when `entity_context is None`
- `tool_executor.py:858-866` — `search_claims` handler returns `[]` when `entity_context is None`
- `tool_executor.py:529-531` — `search_documents` also has a TODO: `entity_tickers` from LLM is silently dropped (entity-filtered doc search doesn't work either)
- `chat_orchestrator.py:321-328` — `all_tools_failed` guard fires, emits the user-facing error

### Required Changes

**Fix 1 — Wire entity_context into ToolExecutor** (`chat_orchestrator.py:160-165`):
After Step 4, build an `EntityContext` from the primary (highest confidence) resolved entity and pass it to `for_request()`. This fixes all single-entity queries:
```python
from rag_chat.application.pipeline.tool_executor import EntityContext
primary = entities[0] if entities else None
entity_context = EntityContext(
    entity_id=primary.entity_id,
    ticker=primary.ticker or "",
    name=primary.canonical_name,
) if primary else None
tool_executor = self._tool_factory.for_request(
    user_id=request.user_id,
    tenant_id=request.tenant_id,
    internal_jwt=None,
    entity_context=entity_context,
)
```

**Fix 2 — Inject all resolved entity UUIDs into the system prompt** (`chat_orchestrator.py:201-214`):
For cross-entity queries ("relation between Apple and Anthropic"), the LLM needs both UUIDs to pass as tool arguments. `_resolve_intel_entity_id` at `tool_executor.py:1212` already accepts a UUID string passed by the LLM — so injecting the UUID map into the system prompt lets the LLM supply the second entity's UUID to tools like `traverse_graph`:
```
Entities resolved from this query:
- "Apple Inc.": entity_id=<uuid> (type: financial_instrument, ticker: AAPL)
- "Anthropic": entity_id=<uuid> (type: organization)
```

**Fix 3 — Wire entity_ids into search_documents** (`tool_executor.py:529-531`):
The `entity_tickers` argument the LLM passes is accepted but silently ignored (TODO comment). Resolve the tickers to entity UUIDs (using the already-resolved entities) and forward to `ChunkSearchRequest.entity_ids`.

### Severity: CRITICAL | Effort: ~1.5h (pure code wiring, no data changes needed)

## 2. ISSUE-2: Daily brief news button — items not visible

### Root Cause
The "REMOVED" section seen in the screenshot is a **literal section header generated by the LLM briefing prompt** and rendered verbatim by `MorningBriefCard` → `StructuredBrief`. The prompt or LLM output is including "REMOVED" as a section title (possibly from a prompt artifact or hallucination). The items within that section are rendered with standard bullet styling and are visible but misrepresent "removed" news items.

Additionally, the expanded brief is rendered inside an `overflow-auto` container, but the section headings and bullets use standard text colors that may not contrast well at the brightness level of the brief panel.

### Evidence
- `apps/worldview-web/components/dashboard/MorningBriefCard.tsx:472` — expands `brief.sections` into `StructuredBrief`
- The brief API returns raw LLM markdown including "REMOVED" as a section header

### Required Changes
- In `MorningBriefCard`, filter out sections with title containing "REMOVED" before passing to `StructuredBrief`
- Update the LLM briefing prompt to explicitly prohibit "REMOVED" as a section heading
- Consider adding a visual distinction between "added" and contextual sections in the brief

### Severity: MEDIUM | Effort: ~1.5h

## 5. ISSUE-5: Knowledge graph — Apple shows 6 nodes, 0 at 2 hops

### Root Cause
**Two separate issues:**

**A) Frontend never sends `depth` parameter:** `apps/worldview-web/lib/api/knowledge-graph.ts:74-88` sends only `limit` to the backend — `depth=2` is derived only to compute limit but is never sent as a query param. The backend **does** support `depth: int` at `knowledge-graph/api/routes.py:220`, but receives `depth=1` (default) on every call.

**B) Data sparsity (content ceiling):** TRACKING.md notes "AAPL stuck at 6 edges — content-quality ceiling. 47-word Finnhub headlines yield 0-1 triples." Even with correct 2-hop queries, Apple's neighbors have few connections themselves, so depth=2 returns little.

### Evidence
- `knowledge-graph.ts:74-88` — never sends `depth` query param
- `routes.py:220` — backend accepts `depth: int = Query(default=1)`
- `routes.py:265-271` — silent fallback to depth=1 if `CYPHER_ENABLED=false`
- TRACKING.md — "AAPL stuck at 6 edges (content-quality ceiling)"

### Required Changes
- Update `knowledge-graph.ts:74-88` to pass `depth` as a query param: `params.set('depth', String(depth))`
- Verify `KNOWLEDGE_GRAPH_CYPHER_ENABLED=true` in docker-compose
- Add a depth slider (1/2/3 hops) to the Intelligence tab graph controls
- Wire the slider value to `getEntityGraph(entityId, depth)` call

### Severity: HIGH | Effort: ~1h

## 8. P1 Issues from Pre-Beta Fourth-Pass Audit

### P1-A: market-data ticker-path 500 (instead of 422)

**Root Cause:** `services/market-data/src/market_data/api/routers/quotes.py:73-83` — `instrument_id: str` with no UUID format guard. PostgreSQL raises `DataError` when a plain ticker string (e.g., "AAPL") is compared to a UUID column → unhandled → 500.

**Compare:** `fundamentals.py:44-46` has `_UUID_PATTERN` guard returning 422 for non-UUID input.

**Fix:** Add `pattern=_UUID_PATTERN` to `instrument_id` path param in quotes router (same 3-line pattern as fundamentals).

**Severity: HIGH | Effort: 30min**

---

### P1-B: rag-chat dev-mode CRITICAL log level

**Root Cause:** `services/rag-chat/src/rag_chat/infrastructure/middleware/internal_jwt.py:182` — `logger.critical("internal_jwt_unverified_decode", ...)` fires on every authenticated request when `skip_verification=True` (dev mode). This floods logs with false-positive CRITICAL alerts.

**Fix:** Change `logger.critical` → `logger.warning` (or `logger.debug`) at that line.

**Severity: MEDIUM | Effort: 15min**

---

### P1-C: SnapTrade /balances endpoint not wired

**Root Cause:** `IBrokerageClient` protocol (`portfolio/application/ports/brokerage_client.py:76`) has no `get_account_balance()` method. `SnapTradeClient` has no balance implementation. No S9 proxy route for `/v1/portfolios/brokerage/{id}/balances` exists.

**Fix:** Add `get_account_balance()` to the protocol, implement in `SnapTradeClient` via SnapTrade SDK account holdings API, add S1 port + S9 proxy route.

**Severity: HIGH | Effort: ~3h**

---

### P1-D: BP-443 pattern in ArticleConsumer (MissingGreenlet)

**Root Cause:** `services/content-store/src/content_store/infrastructure/messaging/consumers/article_consumer.py:81-83` — `_SessionUnitOfWork.__aexit__` does not call `await self.session.close()` before delegating to `_session_cm.__aexit__`. SQLAlchemy asyncpg pool reset path fires outside a `greenlet_spawn` context → `MissingGreenlet` error.

**Fix:** Add `await self.session.close()` wrapped in `contextlib.suppress(Exception)` before delegating to `_session_cm.__aexit__` (same fix as BP-443 in dedup consumer).

**Severity: CRITICAL | Effort: 30min**

---

### P1-E: Embedding worker first-tick delay

**Root Cause:** `services/knowledge-graph/src/knowledge_graph/infrastructure/scheduler/scheduler.py:149-158` — interval jobs for `definition_embedding`, `narrative_embedding`, `fundamentals_embedding` are registered without `next_run_time`. APScheduler waits a full interval (60–120 min) before first execution after restart.

**Compare:** `scheduler.py:189,203` — `narrative_generation` and `path_insight_seeder` already use explicit `next_run_time=now+timedelta(seconds=60)`.

**Fix:** Add `next_run_time=datetime.now(tz=UTC) + timedelta(seconds=120)` to the embedding job registrations in the scheduler loop.

**Severity: MEDIUM | Effort: 15min**

**Effort: 15min**

### P2-D: /v1/health cosmetic route missing
`GET /v1/health` returns 404 in S9 (api-gateway). External uptime monitors expect 200. **Fix:** Add a `GET /v1/health` route returning `{"status": "ok"}` to the api-gateway router.

**Effort: 15min**

### P2-E: SnapTrade description field mapping
`snaptrade_client.py` drops `description` and `settlement_date` from raw SnapTrade `Activity` objects during conversion. Fields exist in the SDK response but are not mapped to the domain `Transaction`. **Fix:** Add `description=act.description` and `settlement_date=_parse_date(act.settlement_date)` in the adapter conversion block.

**Effort: 30min**

### P2-F: Cluster-expand UI (chip clicks don't work)
The "+N sim" chip added in SA-7 is cosmetic — clicking it does nothing because `GET /v1/news/cluster/{cluster_id}` doesn't exist in S9 or S5. **Fix:** Implement the endpoint in content-store (S5), add S9 proxy route, and a frontend modal/drawer to display cluster articles.

**Effort: ~4h**
