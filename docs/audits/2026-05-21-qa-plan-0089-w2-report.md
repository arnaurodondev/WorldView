## Issues — Full Investigation

### F-SEC-001: URL Injection in ContributorsStrip.tsx ✅ FIXED

**Severity**: BLOCKING → FIXED
**File**: `apps/worldview-web/components/portfolio/ContributorsStrip.tsx:48,75`
**Confidence**: HIGH
**Flagged by**: Security agent

**Root Cause**: Both ticker `<Link>` elements in `ContributorsStrip` used template literals without `encodeURIComponent`. The `TickerLink.tsx` cell renderer (the correct pattern for AG Grid) properly encoded ticker values, but `ContributorsStrip` was written independently without applying the same guard. A ticker like `BRK.B` renders fine but a theoretically malformed ticker from a compromised API response (e.g. `../evil`) would produce a traversable URL.

**Fix Applied**:
```tsx
// Before
<Link href={`/instruments/${(entry as MoverEntry).ticker}`} ...>
// After
<Link href={`/instruments/${encodeURIComponent((entry as MoverEntry).ticker)}`} ...>
```

Both occurrences (line 48, line 75) fixed.

### F-QA-002: Performance Spec Lacks 100-Row Fixture and FPS ≥60 Test (C-37)

**Severity**: CRITICAL
**File**: `apps/worldview-web/e2e/portfolio-overview-perf.spec.ts`
**Confidence**: HIGH
**Flagged by**: QA/Test agent

**Root Cause**: The perf spec file contains 3 tests — "page loads without JS errors", "does not crash on 500", "unauthenticated redirect" — none of which measure scroll FPS. The plan §6.2 explicitly specified this as the sparkline perf canary for C-37 (AG Grid sparkline column with 100+ rows). The author wrote error-state tests instead.

**Impact**: Sparkline perf regression (C-37) is completely untested. If the sparkline canvas implementation causes frame drops, no automated gate catches it.

**Solution**: Add a test:
```typescript
test("scroll-fps ≥60 with 100-row fixture", async ({ page }) => {
  await page.setViewportSize({ width: 1440, height: 900 });
  // intercept holdings API with 100-row fixture
  await page.route('**/v1/portfolios/**/holdings', (route) =>
    route.fulfill({ json: { holdings: generateFixture(100) } }));
  await page.goto('/portfolio');
  // measure scroll performance
  const fps = await page.evaluate(async () => {
    let frames = 0;
    const start = performance.now();
    const id = requestAnimationFrame(function loop() { frames++; requestAnimationFrame(loop); });
    // scroll rapidly
    window.scrollBy(0, 5000);
    await new Promise(r => setTimeout(r, 1000));
    cancelAnimationFrame(id);
    return frames / ((performance.now() - start) / 1000);
  });
  expect(fps).toBeGreaterThanOrEqual(60);
});
```

### F-DS-001: PortfolioSwitcher Writes to Context; usePortfolioData Reads Local State — Portfolio Switching Inoperative

**Severity**: CRITICAL
**File**: `apps/worldview-web/features/portfolio/hooks/usePortfolioData.ts:164,198`
**Confidence**: HIGH
**Flagged by**: Distributed Systems agent

**Root Cause Analysis**:
- `PortfolioSwitcher.tsx:115` calls `setActivePortfolio(id)` → updates `ActivePortfolioContext`
- `usePortfolioData.ts:164` initialises `selectedPortfolioId` as `useState<string | null>(null)` — **no context import**
- `activePortfolioId = selectedPortfolioId ?? sortedPortfolios?.[0]?.portfolio_id` — derives from local state, not context
- When user clicks PortfolioSwitcher: context updates, but `usePortfolioData`'s state never changes → page continues showing old portfolio
- `useResolvedPortfolioId.ts` correctly reads from context (`useActivePortfolio`) but is **not used** in `usePortfolioData`
- Portfolio page passes `setSelectedPortfolioId` to `PortfolioPageHeader` (`onSelectPortfolio` prop), but W2 removed the dropdown from `PortfolioPageHeader` — so this is now dead

**When it manifests**: Always — every time the user uses the TopBar portfolio switcher on the `/portfolio` page.

**Evidence**:
```typescript
// usePortfolioData.ts:164 — local state, context-blind
const [selectedPortfolioId, setSelectedPortfolioId] = useState<string | null>(null);

// PortfolioSwitcher.tsx:115 — updates context, nobody in usePortfolioData listens
setActivePortfolio(id === ROOT_SENTINEL ? null : id);
```

**Solution Options**:

#### Option A: Wire usePortfolioData to read from ActivePortfolioContext
```typescript
// usePortfolioData.ts — add at top of function
const { activePortfolioId: contextPortfolioId, setActivePortfolio } = useActivePortfolio();

// Replace selectedPortfolioId state with context
const activePortfolioId = contextPortfolioId ?? sortedPortfolios?.[0]?.portfolio_id ?? null;
```
- **Benefits**: Single source of truth; no prop drilling; `setSelectedPortfolioId` can become `setActivePortfolio` (context-aware)
- **Drawbacks**: Creates a dependency on the context from a "data hook" (minor architecture concern)

#### Option B: Have PortfolioSwitcher call both context AND a registered callback
Register a listener in the portfolio page, dispatch to `setSelectedPortfolioId`. More indirection, not recommended.

**Recommended**: Option A — 3-line change, fully unblocked.

### F-DATA-001: benchmarkSeries Query Key Missing startDate — Stale Cache Across Date Boundary

**Severity**: CRITICAL
**File**: `apps/worldview-web/features/portfolio/hooks/useBenchmarkSeries.ts:46,54`
**Confidence**: HIGH
**Flagged by**: Data Platform agent

**Root Cause**: `startDate` is computed from `Date.now()` at render time but is not included in the TanStack Query cache key. When the browser clock crosses midnight UTC (or when the user's timezone rolls over), `startDate` changes but the cache key (`qk.market.benchmarkSeries(SPY_TICKER, period)`) stays the same. The chart will serve data from yesterday's range with today's cache hit.

**Evidence**:
```typescript
const startDate = new Date(Date.now() - days * 24 * 60 * 60 * 1000)
  .toISOString().slice(0, 10);  // computed at render, not in cache key

queryKey: qk.market.benchmarkSeries(SPY_TICKER, period),  // missing startDate
queryFn: async () => gw.getOHLCV(SPY_TICKER, { timeframe: "1D", start: startDate }),
```

**Fix**:
```typescript
queryKey: qk.market.benchmarkSeries(SPY_TICKER, period, startDate),
```
Update `lib/query/keys.ts:407` factory to accept optional 3rd `startDate` param.

### F-SEC-002: Hardcoded JWT Dev Key in brokerage_sync_worker.py

**Severity**: CRITICAL
**File**: `services/portfolio/src/portfolio/workers/brokerage_sync_worker.py:177`
**Confidence**: HIGH
**Flagged by**: Security agent

**Root Cause**: `_system_jwt_headers()` embeds `"dev-skip-verification-key-for-brokerage-sync-worker"` as the HS256 signing secret directly in source code. The comment documents this as dev-only, but there is no enforcement guard against production usage. If `skip_verification=True` is accidentally left on in production, this hardcoded key allows service token forgery.

**Evidence**:
```python
token = pyjwt.encode(
    { "iss": "worldview-gateway", "sub": "system:brokerage-sync", ... },
    "dev-skip-verification-key-for-brokerage-sync-worker",  # hardcoded
    algorithm="HS256",
)
```

**Solution**: Move to pydantic-settings:
```python
# config.py
brokerage_sync_jwt_secret: str = Field(
    default="dev-skip-verification-key-DO-NOT-USE-IN-PROD",
    description="HS256 key for dev-mode brokerage-sync JWTs only",
)
```
Add an `APP_ENV == "production"` guard that raises if `skip_verification` would be accepted.
