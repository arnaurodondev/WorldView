## Issues — Full Investigation

## Issue F-001: LAYER-APP-ISOLATION regression — ticker normalizer in `infrastructure/` imported from `application/`

### Summary
F1 introduced an automated architecture test
(`tests/architecture/test_layer_boundaries.py`) that enforces the hexagonal
invariant "`application/` must not import from `infrastructure/`". Step 7
(commit `4000ac1c`) placed the pure `_normalize_ticker` function at
`services/market-data/src/market_data/infrastructure/_ticker_normalize.py`
but `application/use_cases/on_demand_profile.py:27` imports it directly,
flipping the test from green to red.

### Severity / Confidence
**Severity**: BLOCKING — auto-enforced architecture invariant; merge gate.
**Confidence**: HIGH — directly reproducible with `python -m pytest tests/architecture/test_layer_boundaries.py -v`.
**Flagged by**: architecture test (LAYER-APP-ISOLATION + LAYER-BOUNDARY; both rules fire on the same import).

### Root Cause Analysis
- **What**: `from market_data.infrastructure._ticker_normalize import normalize_ticker` at `services/market-data/src/market_data/application/use_cases/on_demand_profile.py:27`.
- **Why**: `_normalize_ticker` is a pure-string helper with zero infrastructure dependencies (no I/O, no DB, no HTTP). It was placed under `infrastructure/` because Step 7 also wires it into adapters (eodhd, ohlcv consumer, quotes consumer, fundamentals consumer, intraday resampling consumer) — those are infrastructure, so colocating the helper there minimised step diff. The application-layer use case `on_demand_profile` also calls it for the ticker-input normalisation contract; that crosses the layer boundary.
- **When**: Always — `pytest tests/architecture` deterministically fails.
- **Where**: `services/market-data/src/market_data/application/use_cases/on_demand_profile.py:27` (importer); `services/market-data/src/market_data/infrastructure/_ticker_normalize.py` (importee).
- **History**: Introduced by commit `4000ac1c` (Step 7 / s2-normalize). The F1 layer-boundary test pre-dates this commit and was green on `feat/plan-0089-f1`.

### Evidence
```
[LAYER-APP-ISOLATION] Architecture violations (1):
  Service:  market-data
  File:     services/market-data/src/market_data/application/use_cases/on_demand_profile.py:27
  Rule:     LAYER-APP-ISOLATION
  Detail:   application/ imports from infrastructure/ (`market_data.infrastructure._ticker_normalize`).
            Application must only depend on domain and use port interfaces.
```
File reference: `services/market-data/src/market_data/application/use_cases/on_demand_profile.py:27` and `tests/architecture/test_layer_boundaries.py:99` / `:136`.

### Impact
- **Immediate**: F1's automated layer-boundary invariant is broken. Pre-PR hook (`scripts/hooks/pre-pr-checklist.sh` if it runs the architecture suite) will reject the merge.
- **Blast radius**: Only the one import line and its single helper module; no behavioural impact on running code.
- **Data risk**: None.
- **User impact**: None at runtime. CI/dev-experience impact only.

### Solution Options

#### Option A (recommended): Move `_ticker_normalize.py` to `domain/` (or a shared `common/` location)
**Description**: The helper is a pure string transformation with no infrastructure or framework dependencies, so it belongs in the domain layer (or in `libs/common` if cross-service sharing becomes attractive). Move the file; update the five infrastructure adapters + the one application use case to import from the new location.
**Changes required**:
- [ ] `services/market-data/src/market_data/infrastructure/_ticker_normalize.py` → `services/market-data/src/market_data/domain/_ticker_normalize.py` (or `domain/ticker.py` if you prefer a non-underscored module).
- [ ] 6 import sites: `application/use_cases/on_demand_profile.py:27` and the 5 infrastructure consumers/adapters.
- [ ] `services/market-data/tests/test_ticker_normalize.py` — fix the one import path.
**Benefits**: Restores hexagonal invariant; the helper now lives where its purity declares it belongs; cross-layer use is correct (domain → app/infra is the legal direction).
**Drawbacks**: None.
**Effort**: Low (10 minutes including running the 18-test suite).
**Risk**: Low.

#### Option B: Define a port in `application/ports/` and let infrastructure provide the impl
**Description**: Treat ticker normalisation as a port (`ITickerNormalizer`) with an in-memory infra implementation. `on_demand_profile` accepts the port via constructor injection.
**Drawbacks**: Heavy machinery for a 30-LOC pure function; introduces a port that has exactly one zero-state implementation; violates Standards §11 "no premature abstraction".
**Effort**: Medium.
**Risk**: Low.

#### Option C: Loosen the architecture test
**Description**: Add an allowlist for `_ticker_normalize` to the architecture test.
**Drawbacks**: Trains future authors that the rule is negotiable; the rule is correct and the violation is real.
**Effort**: Low.
**Risk**: Medium (precedent).

### Recommended Option
**Option A** — the helper is pure and belongs in domain; moving it is mechanical, restores the invariant, and avoids unnecessary indirection.

### Verification Steps
- [ ] `python -m pytest tests/architecture/test_layer_boundaries.py -v` returns 0 failures.
- [ ] `cd services/market-data && python -m pytest tests/test_ticker_normalize.py -v` still 18/18 green.
