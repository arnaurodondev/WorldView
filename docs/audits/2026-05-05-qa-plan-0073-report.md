## Executive Summary

Five specialist agents (QA/Test, Security, Data Platform, Distributed Systems, Architecture) reviewed the full PLAN-0073 implementation. PLAN-0073 introduces Worker 13J (StructuredEnrichmentWorker) plus unified `/instruments/lookup` and `/on-demand-profile` endpoints in S3, a new `GET /entities/{entity_id}` route in S7, an S9 proxy, and a frontend description panel.

**Overall health**: The schema migrations (0022–0024 + 016) are well-structured, R25/R27 boundaries hold for the new entity-detail route, and many architectural invariants are correctly maintained. **However, three independent agents converge on three BLOCKING wire-up defects that mean the plan's central feature does not actually function in production** — `entity.dirtied.v1` is never produced, `MarketDataClient.internal_jwt=""` returns 401 against S3 outside dev, and the hot-path consumer enriches all entity types in violation of FR-04. A fourth BLOCKING defect (`RetryableEnrichmentError` silently dropped by Kafka consumer) makes the only retry mechanism non-functional. Additionally, migration 0023's seed UPDATE matches **zero rows** because canonical_type values are uppercased while the existing registry uses lowercase — a silent no-op that defeats the structural-relation seeding goal entirely.

Beyond the BLOCKING issues, 16 CRITICAL findings cover prompt-injection mitigation never wired in (`sanitize_entity_name` is dead code), SSRF route-level pattern guards missing on `/on-demand-profile`, R25 session lifecycle violation in `OnDemandProfileUseCase` (DB session held over 10s EODHD HTTP call), and the entire infrastructure layer (worker, consumer, adapter, MarketDataClient, frontend panel) shipping with zero direct unit tests.

**Deployment readiness**: NOT MERGE-READY. BLOCKING and CRITICAL findings must be resolved before this plan can be considered complete.

## BLOCKING Issues — Full Investigation

### Issue F-X01 (BLOCKING): RetryableEnrichmentError silently dropped by Kafka consumer

**Severity**: BLOCKING **Confidence**: HIGH **Flagged by**: Distributed Systems

**Summary**: The structured enrichment Kafka consumer's `RetryableEnrichmentError` (raised on EODHD 429 / LLM timeout) re-raises into the run-loop, which calls `_handle_failure` → `store_failure` (logs only) and continues to the next message. The next successful poll commits a higher offset, permanently skipping the retryable failure until consumer restart.

**Root cause**: `StructuredEnrichmentConsumer.process_message_from_failure`, `get_pending_retries`, and `store_failure` are all log-only stubs. There is no DB-backed retry queue, no consumer.seek() to redeliver, no busy-loop with backoff.

**File**: `services/knowledge-graph/src/knowledge_graph/infrastructure/messaging/consumers/structured_enrichment_consumer.py:147-160`, plus `libs/messaging/.../consumer/base.py:416-418, 612-625`.

**Impact**: PRD §13.2 promised "a 429 aborts the current sweep iteration; Kafka redelivers the message". This redelivery does not happen. The retry path is purely cosmetic.

**Recommended option**: In `_handle_message` (or via override in `StructuredEnrichmentConsumer`), when the exception is `RetryableEnrichmentError`, call `self._consumer.seek(msg.partition, msg.offset)` so the broker re-delivers on next poll. Alternatively, implement actual retry-queue persistence in `store_failure` / `get_pending_retries`.

### Issue F-X02 / F-A01 (BLOCKING): `entity.dirtied.v1` never produced

**Severity**: BLOCKING **Confidence**: HIGH **Flagged by**: Distributed Systems + Architecture

**Summary**: `StructuredEnrichmentUseCase.__init__` accepts `direct_producer: DirectProducerProtocol | None = None`. Both bootstrap call sites (`scheduler.py:_add_structured_enrichment_worker` and `structured_enrichment_consumer_main.main`) construct the use case **without** passing a producer. The use case's Step 8 (`if self._producer is not None: ...`) never executes. PRD §10.1 step 8 + §13.7 recovery flow are dead code.

**File**: `services/knowledge-graph/src/knowledge_graph/infrastructure/scheduler/scheduler.py:338-343` and `services/knowledge-graph/src/knowledge_graph/infrastructure/messaging/consumers/structured_enrichment_consumer_main.py:72-77`.

**Impact**: Entities enriched by Worker 13J never emit `entity.dirtied.v1`. The downstream `EmbeddingRetryWorker` falls back to a watermark-based scan (PRD §13.7 fallback path), so embeddings eventually refresh — but the primary signal documented in the PRD is missing entirely. Any future consumer of `entity.dirtied.v1` would silently miss Worker 13J output.

**Recommended option**: In both bootstrap paths, instantiate a `ConfluentDirectProducer` (existing pattern in `macro_indicator_dataset_consumer_main.py:75-81`) and pass `direct_producer=...`. Add startup `WARNING` log when the producer is None, mirroring `provisional_enrichment.py:175-178`.

### Issue F-X03 / F-A03 / F-Q07 (BLOCKING): FR-04 entity_type filter missing in hot-path consumer

**Severity**: BLOCKING **Confidence**: HIGH **Flagged by**: DS + Architecture + QA

**Summary**: PRD-0073 §3.1 FR-04 explicitly mandates: "Hot-path enrichment MUST skip entity types other than `financial_instrument` and `company`". Plan T-C-2-03 acceptance criterion: "Consumer skips non-`financial_instrument`/`company` entity types". Implementation does not enforce this; consumer's `process_message` runs `enrich()` for every entity_type. The consumer docstring even contradicts the PRD ("for all entity types").

**File**: `services/knowledge-graph/src/knowledge_graph/infrastructure/messaging/consumers/structured_enrichment_consumer.py:88-148`.

**Impact**: Every `person/concept/location/event` entity creation triggers a synchronous LLM call in the hot path. NLP-pipeline ingestion bursts → LLM stampede; NFR-01 30s p95 budget violated; ADR-0073-002 cost rationale defeated.

**Recommended option**: Add early-return after entity load:
```python
if entity.entity_type not in ("financial_instrument", "company"):
    logger.debug("structured_enrichment_consumer_skip_non_financial", ...)
    return
```

### Issue F-X06 / F-A02 / F-S02 (BLOCKING): MarketDataClient internal_jwt=""

**Severity**: BLOCKING **Confidence**: HIGH **Flagged by**: DS + Architecture + Security

**Summary**: Both bootstrap paths pass `internal_jwt=""` to `MarketDataClient`. S3's `/on-demand-profile` is gated by `require_internal_jwt`. Empty header passes only when `MARKET_DATA_INTERNAL_JWT_SKIP_VERIFICATION=true` (dev-only); in production every call returns 401. PRD §12 explicitly requires Worker 13J to sign an RS256 JWT using `settings.internal_jwt_private_key`.

**File**: `services/knowledge-graph/src/knowledge_graph/infrastructure/scheduler/scheduler.py:332-335` and `services/knowledge-graph/src/knowledge_graph/infrastructure/messaging/consumers/structured_enrichment_consumer_main.py:66-69`.

**Impact**: Step 2 of the cascade (EODHD on-demand-profile) is non-functional in any environment with proper JWT verification. The full cascade degrades to "DB lookup → LLM" — bypassing EODHD entirely. ADR-0073-001 cascade design unimplemented in prod.

**Recommended option**: Mirror `FundamentalsRefreshWorker`'s pattern (`scheduler.py:283-285`) which already reads `settings.internal_jwt_private_key` and signs an RS256 JWT. Inject the signed token into both `MarketDataClient` constructions. Use a per-request signer (callable) so JTI is fresh.

### Issue F-D01 (BLOCKING): Migration 0023 seed UPDATE matches zero rows (case mismatch)

**Severity**: BLOCKING **Confidence**: HIGH **Flagged by**: Data Platform

**Summary**: Seed UPDATE uses UPPERCASE canonical_types (`OPERATES_IN_SECTOR`, `OPERATES_IN_INDUSTRY`, `HEADQUARTERED_IN`, `LISTED_ON`), but every existing `relation_type_registry` row uses LOWERCASE (per migrations 0001/0002/0004). The `WHERE canonical_type = :type AND data_source IS NULL` predicate matches **zero rows** — the entire seed is a silent no-op. Downstream, `EntityEnrichmentAdapter.seed_relations` queries `WHERE data_source = 'market_data'` and finds nothing → Worker 13J never seeds any structural relations.

**File**: `services/intelligence-migrations/alembic/versions/0023_add_source_fields_to_relation_type_registry.py:44-51`.

**Impact**: The central goal of the plan (seeding `OPERATES_IN_SECTOR`, `HEADQUARTERED_IN`, etc.) silently fails. Worker 13J writes descriptions but never seeds the relation graph.

**Recommended option**: (1) Use lowercase canonical_types matching existing seeds; (2) replace `OPERATES_IN_SECTOR`/`OPERATES_IN_INDUSTRY` with `is_in_sector`/`is_in_industry` (the actual existing names); (3) update `_CANONICAL_TYPE_OBJECT_ENTITY_TYPES` in `entity_enrichment_adapter.py` to match.

## CRITICAL Issues

### F-S01 / F-A05: Prompt-injection sanitizer dead code
- `libs/prompts/.../entity_enrichment.py` exports `sanitize_entity_name` and `build_entity_enrichment_prompt`, but `DeepInfraDescriptionAdapter._build_prompt` ignores them. PRD §12 prompt-injection mitigation unmet. Wire `sanitize_entity_name` into `_build_prompt`; wrap entity name in `<entity>...</entity>` delimiters.

### F-S03: SSRF route-level pattern guard missing on /on-demand-profile
- Ticker/ISIN have `min_length`/`max_length` but no `pattern=` regex at FastAPI boundary; only the use case validates. Defence-in-depth gap. Add `pattern=` to Query params on `/instruments/on-demand-profile`.

### F-D02: R25 session lifecycle violation in OnDemandProfileUseCase
- Use case opens UoW, performs DB lookup, then issues EODHD HTTP call (10s timeout) **while holding the session**, then writes via the same UoW. Read-replica connection pinned for up to 10s per call. Refactor to 3-phase: (1) read+release, (2) HTTP call, (3) re-acquire+write+commit.

### F-D03: OPERATES_IN_SECTOR/INDUSTRY don't exist in any case
- Independent of F-D01: `OPERATES_IN_SECTOR` and `OPERATES_IN_INDUSTRY` are not in the registry under ANY case. Use `is_in_sector` / `is_in_industry` (PG seed names from migration 0002).

### F-X04: Sweep no backoff on consecutive 429s
- Worker continues to next entity after 429 → re-hits 429 immediately. Burns through 50× failed S3 calls in seconds. Add consecutive-failure tracking + exponential backoff or circuit breaker; abort batch after N consecutive retryables.

### F-X05: Empty event_id collapses Valkey dedup keys
- `extract_event_id` returns `""` when payload missing → all such events share one dedup key → silent data loss after first event. Raise `MalformedDataError` on missing/empty event_id.

### F-X07: BP-064 partial-commit ordering risk
- `enrich()` owns its own transaction boundary; `_handle_message` calls `mark_processed` *after* enrich returns. Documented but not enforced.

### F-A04: enrichment_llm_model_id config field is dead; fallback model differs from ADR-0073-006
- Field defined but never read. Actual config: `description_deepinfra_*` defaults fallback to `Qwen3-32B`. ADR-0073-006 mandates `meta-llama/Meta-Llama-3.1-8B-Instruct-Turbo`. Decide: delete unused fields and align defaults, OR wire to Worker-13J-specific path.

### F-Q03: EntityEnrichmentAdapter zero unit tests
- All four methods contain raw SQL; `seed_relations` JSONB merge and INSERT...ON CONFLICT untested. Add `tests/integration/intelligence_db/test_entity_enrichment_adapter.py`.

### F-Q04: 7 of 16 promised T-C-2-02 tests missing
- Including: `test_relation_seeding_eodhd_sector`, `test_relation_seeding_skips_missing_object_entity`, `test_entity_dirtied_produce_failure_does_not_raise`, `test_llm_prompt_sanitizes_entity_name`, `test_market_data_connect_error_falls_through`, `test_llm_fallback_on_primary_404`, `test_enrichment_attempts_incremented_on_llm_short_response`.

### F-Q05: EntityDescriptionPanel.tsx zero unit tests
- All 4 render states (loading skeleton, null, populated, completeness bar) unverified. Add `__tests__/entity-description-panel.test.tsx`.

### F-Q06: MarketDataClient zero unit tests
- 404→None vs 429→raise discrimination, header injection, query-param mapping all unverified. Add `tests/unit/infrastructure/http/test_market_data_client.py` using `respx`.

### F-Q08: Fallback model retry logic missing
- PRD §13.4 mandates "primary→fallback model retry once". Use case has no such logic; trusts adapter. Either implement or update plan.
