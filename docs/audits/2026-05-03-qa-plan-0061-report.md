## MAJOR Issues (Deferred)

### F-003 ┬╖ MAJOR ┬╖ `provisional_enrichment_core.py:apply_retry_transition`
**Category**: Distributed Systems / Non-atomic decision
**Confidence**: MEDIUM
**Issue**: `apply_retry_transition` reads `retry_count` as a parameter (not re-queried from DB), then decides `pending` vs `failed` based on that value. Concurrent consumers could theoretically read the same `retry_count` and both choose `pending`, exceeding `max_retries`. Functionally safe in practice because `FOR UPDATE SKIP LOCKED` prevents concurrent processing of the same row тАФ only one consumer holds the lock at a time.
**Decision**: Accept. `FOR UPDATE SKIP LOCKED` already prevents the race. Non-atomic pattern is safe here.

### F-004 ┬╖ MAJOR ┬╖ `provisional_enrichment_core.py` тАФ no direct unit tests
**Category**: Test Coverage
**Confidence**: MEDIUM
**Issue**: `extract_entity_profile`, `compute_embedding`, `persist_enrichment`, `apply_retry_transition` are tested indirectly via `test_provisional_queued_consumer.py` patches, but no dedicated unit tests for the core functions themselves (e.g., malformed LLM response handling in `persist_enrichment`, the `max_retries` boundary in `apply_retry_transition`).
**Decision**: Deferred. Indirect test coverage is sufficient for the current reliability bar; add dedicated tests in a follow-up enrichment quality pass.

### F-005 ┬╖ MAJOR ┬╖ `_fail_safe_retry` in consumer тАФ stale `processing` row on permanent failure
**Category**: Distributed Systems / Catch-Up Gap
**Confidence**: MEDIUM
**Issue**: If `_fail_safe_retry` itself fails (DB down), the row remains in `processing` state permanently. The catch-up sweep (`ProvisionalEnrichmentWorker`) does not requeue `processing` rows that have been stuck for >N minutes. A stuck `processing` row from the hot path can prevent the catch-up sweep from enriching the same entity.
**Decision**: Deferred. `_fail_safe_retry` already increments `s7_provisional_queue_stuck_total` for observability. A dead-letter sweep (reset `processing`тЖТ`pending` after timeout) is a good PLAN-0062 candidate.

### ml-clients Fix: `TestDeepSeekExtractionAdapter` (4 tests)

**Root cause**: `DeepSeekExtractionAdapter.__init__` does an eager `import openai` and creates `self._client = openai.AsyncOpenAI(...)`. Tests were creating the adapter *outside* `patch.dict("sys.modules", {"openai": mock_openai})`. When `extract()` ran inside the mock context, the real `openai.AsyncOpenAI.chat` `cached_property` fired, doing `from .resources.chat import AsyncChat` while `sys.modules["openai"]` was the MagicMock тАФ raising `ModuleNotFoundError: 'openai' is not a package`. This was caught by the broad `except Exception` handler and re-raised as `FatalError` instead of the expected `RetryableError`/`FatalError` from the fake error classes.

**Fix** (`libs/ml-clients/tests/test_adapters.py`): Moved `adapter = DeepSeekExtractionAdapter(...)` inside the `with patch.dict(...)` context for all 4 tests. This ensures `__init__` binds `self._openai` and `self._client` to the mock. Pattern follows `ChatGPTExtractionAdapter` which defers `import openai` into `extract()` and naturally avoids this issue.

**Classification**: BP-306 тАФ Eager-import adapters must be instantiated inside `patch.dict` mock context; adapters with deferred `import` in the called method can be instantiated outside.
