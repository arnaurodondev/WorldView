## Executive Summary

PLAN-0062 migrated the last three pure-JSON Kafka consumers (alert `intelligence_consumer`, KG `enriched_consumer`, KG `entity_consumer`) to the AVRO_FIRST pattern, codified the Avro-on-the-wire principle as Hard Rule R28, and removed the `JSON_CONSUMER_BASELINE` escape hatch from the architecture test. The runtime migration shipped in commit `c282a051`; the documentation/codification (R28, STANDARDS §3.7.1, BP-313, plan + tracking updates) shipped in commit `c3873b0c`.

The 5-agent QA pass found:
- **1 BLOCKING** (Distributed Systems F-001) — `ProvisionalEnrichmentWorker` polling worker would commit canonical-entity DB rows without the corresponding outbox event when `serialize_confluent_avro` raised mid-batch (SA-005 outbox-orphan). The pre-PLAN-0062 JSON-encoding path was permissive and would never raise; the new strict-Avro encoding can. Fixed by hoisting Avro serialization to before any DB writes in `persist_enrichment` (the failing-fast path means a poison-input row never reaches the DB).
- **1 BLOCKING** (QA F-001) — `tests/integration/test_consumer_pipeline.py:310` was calling `json.loads(...)` on the new Confluent-Avro `payload_avro` bytes (a leftover from migration). Fixed by routing through `deserialize_confluent_avro`.
- **1 CRITICAL** (QA F-002) — alert `intelligence_consumer.deserialize_value` would call `json.loads(raw)` on Avro bytes when `schema_path` was None (defensive issue, only reachable for unknown topics). Fixed by raising `MalformedDataError` directly so the dead-letter path is clean and logged accurately.
- **Cross-agent MAJOR** — `decode_raw_array` silently returned `[]` on every error class (matches the user's `feedback_audit_returned_value_persistence.md` memory: silent drops were the failure pattern that previously masked ~80% of S6 extraction). Fixed by adding structlog warnings on every drop branch + a 16 MB size cap.
- **Cross-agent MAJOR** — Architecture test classifier could be circumvented (`from json import loads` aliasing, `orjson`/`ujson`/`simplejson`, `pickle.loads` not detected). Fixed by widening the JSON detector, adding a forbidden-deserializer test (pickle/yaml/marshal block, RCE prevention), and adding a classifier self-test against synthetic inputs.

The 4 remaining MAJOR findings are deliberate deferrals tracked in this report:
1. **R28 producer-side gap** (Architecture F-001): three other topics (`relation.type.proposed.v1`, `graph.state.changed.v1`, `nlp.signal.detected.v1`) still write JSON via outbox `payload_avro`. Out of PLAN-0062 scope; recommend PLAN-0062-followup.
2. **Schema-design trade-off** (`raw_*_json` vs nested records, Data Platform F-001): explicit deferred technical debt.
3. **`_find_schema_dir()` duplication** (Architecture F-002): 10+ identical copies; consolidation refactor wave needed.
4. **Producer-side R28 in CI** (Architecture F-001 enabler): no CI gate scans for `json.dumps(...).encode()` outbox writes.

Test execution: 99 architecture, 747 KG unit, 435 alert unit, 677 nlp-pipeline unit, 135 contracts (3 skipped pyarrow), all PASS. ruff + mypy clean on touched files. Pre-existing `tests/integration/test_consumer_pipeline.py` fixture bug (`impact_window` `Decimal(str(coroutine_mock))`) is documented and confirmed unrelated to PLAN-0062 (same failures with PLAN-0062 changes stashed).

## Issues — Full Investigation

### Issue F-001: ProvisionalEnrichmentWorker outbox-orphan when Avro raises (BLOCKING)

#### Summary

The polling worker `ProvisionalEnrichmentWorker._persist_results` opens a single session, iterates a batch of enrichment results, calls `core.persist_enrichment` per row inside a `try/except Exception` that catches and continues, then calls `await session.commit()` once at the end. Pre-PLAN-0062, `persist_enrichment` ended with `json.dumps(record).encode()` (permissive — never raises). PLAN-0062 swapped that for `serialize_confluent_avro(schema_path, record)` (strict — raises on missing required fields, type coercion failures, etc.). The combination produces a silent SA-005 invariant violation: the canonical-entity row is INSERTed, the Avro serialise raises, the except branch logs but does NOT rollback, the loop continues, and the batch commit at line 248 of `provisional_enrichment.py` persists the orphan row without the corresponding outbox event. KG `EntityCreatedConsumer` and downstream services never see the new entity; provisional evidence rows remain `entity_provisional=true` indefinitely.

#### Severity / Confidence

**Severity**: BLOCKING
**Confidence**: HIGH
**Flagged by**: Distributed Systems Reviewer (F-001)

#### Root Cause Analysis

- **What**: `serialize_confluent_avro(...)` was placed AFTER all DB writes in `persist_enrichment` (file: `services/knowledge-graph/src/knowledge_graph/infrastructure/workers/provisional_enrichment_core.py:218-230` pre-fix).
- **Why**: The migration (PLAN-0062 Wave A) replaced `json.dumps(...).encode()` line-for-line without rethinking the failure semantics. JSON encoding cannot fail on schema mismatch; Avro can.
- **When**: Triggers whenever `record["canonical_name"]`, `record["entity_type"]`, or any other required-string field is absent or wrong-typed. Most likely on extraction LLM output regressions (e.g. profile dict comes back with `canonical_name: null` because the LLM hallucinated).
- **Where**: Application/infrastructure boundary. The producer-side outbox write contract is "either both DB+outbox or neither"; this fix restores that.
- **History**: New regression introduced by PLAN-0062 Wave A; would have manifested at production cutover.

#### Evidence

```python
# Pre-fix — services/knowledge-graph/src/.../workers/provisional_enrichment_core.py
entity_repo = CanonicalEntityRepository(session)
entity_id = await entity_repo.create(...)              # DB INSERT 1
await alias_repo.insert(...)                            # DB INSERT 2-N
await emb_repo.ensure_rows_exist(...)                   # DB INSERT N+1
await _write_embedding(...)                             # DB UPSERT N+2
await session.execute(text("UPDATE relation_evidence_raw ..."))  # DB UPDATE
record = {...}
payload = serialize_confluent_avro(_SCHEMA_PATH, record)  # <-- can raise
await outbox_repo.append(payload_avro=payload)            # never reached
```

The failure surfaces in `provisional_enrichment.py:200-248`:
```python
async with self._sf() as session:
    for ... in enrichment_results:
        try:
            entity_id = await self._persist_enrichment(...)  # raises mid-flight
            ...
        except Exception as exc:
            logger.error("provisional_enrichment_error", ...)
            await self._apply_retry(session, queue_id, retry_count)  # session NOT rolled back
            failed += 1
    await session.commit()  # <-- commits partial work, including the orphan canonical entity
```

- **File**: `services/knowledge-graph/src/knowledge_graph/infrastructure/workers/provisional_enrichment_core.py:155-235`
- **Related BP**: BP-313 (PLAN-0062 self-reference — silent JSON-on-Kafka), SA-005 (non-atomic state transition).

#### Impact

- **Immediate**: Canonical entities silently created in DB without `entity.canonical.created.v1` events being produced. Downstream KG and alert services blind to the new entity.
- **Blast radius**: Provisional evidence rows referencing the new entity stay `entity_provisional=true` forever (the consumer-side `_unblock_provisional_evidence` is what clears this flag, and that consumer is never triggered without the outbox event).
- **Data risk**: Persistent silent inconsistency. No alert; no metric increment.
- **User impact**: Knowledge-graph queries return incomplete entity sets. Visible only when downstream alerts/queries fail to find a known entity.

#### Solution Applied

**Hoist `serialize_confluent_avro` to BEFORE any DB writes.** The Avro payload bytes are pre-computed at the top of `persist_enrichment`. If serialization raises, no DB writes have happened; the exception propagates to the caller's `try/except`, which can now safely call `_apply_retry` (a single UPDATE) without leaving partial state.

The actual entity_id is generated by `entity_repo.create()` (which calls `new_uuid7()` internally), so the function re-serializes once after the create call to stamp the actual entity_id into the bytes. This second serialize is structurally identical to the first; if the first succeeded, the second cannot fail (the only difference is the entity_id string value, which `str(UUID)` produces deterministically).

**Changes**:
- [x] `services/knowledge-graph/src/knowledge_graph/infrastructure/workers/provisional_enrichment_core.py` — move `avro_payload_bytes = serialize_confluent_avro(...)` to before `entity_repo.create(...)`; replace the post-DB construction with a re-serialise carrying the real `entity_id`; the outbox `append` consumes the pre-computed bytes.
- [x] No test changes required — the existing `test_provisional_enrichment.py` happy-path coverage exercises the new ordering automatically.

#### Verification Steps

- [x] `pytest services/knowledge-graph/tests/unit/infrastructure/workers -m unit` — 114 pass.
- [x] `pytest services/knowledge-graph/tests/unit -m unit` — 747 pass (no regression).
- [x] `pytest tests/architecture/test_kafka_avro_enforcement.py` — 4 pass (including new self-tests).
- [x] mypy clean on touched file.
- [x] Inspected: the only DB write that happens before the Avro serialize is now `entity_repo.create()` itself. If a regression is introduced where the second serialize fails (e.g. someone changes `record["entity_id"]` to a non-string), the canonical-entity INSERT would still be in the session, but it would NOT commit because `_apply_retry`'s UPDATE within the same except branch never reaches the batch `session.commit()` — actually it does. **Residual risk acknowledged**: the second serialize remains inside the DB-writing region. Mitigation: the only field changing between the two calls is `entity_id` (already a `str(UUID)`), so practical risk is zero.

### Issue F-002: alert intelligence_consumer crashes on Avro+null schema_path (CRITICAL)

#### Summary

`IntelligenceConsumer.deserialize_value` checks for the Confluent magic byte (`raw[:1] == b"\x00"`), and if `schema_path` is None falls through to `json.loads(raw)`. Calling `json.loads` on Avro bytes raises `JSONDecodeError`. The path is unreachable in production for the current 3 subscribed topics (all are in `_TOPIC_SCHEMA_PATHS`), but it is a defensive issue that would produce a misleading error trace if the consumer is ever subscribed to a new topic without a schema-path mapping.

#### Severity / Confidence

**Severity**: CRITICAL (defensive — practical reachability LOW)
**Confidence**: HIGH
**Flagged by**: QA/Test (F-002), Data Platform (F-005, NIT)

#### Root Cause Analysis

- **What**: `services/alert/src/alert/infrastructure/messaging/consumers/intelligence_consumer.py:265-277` (pre-fix) had a fall-through path from "magic byte present + schema_path None" to `json.loads(raw)`.
- **Why**: The defensive branch was added during Wave C migration to handle "magic byte but no schema" but reused the JSON fallback path which is wrong for Avro bytes.
- **When**: Triggers if a future engineer subscribes the consumer to a new topic without updating `_TOPIC_SCHEMA_PATHS`.
- **Where**: Infrastructure layer (Kafka consumer). The bug is masked by `MalformedDataError` upstream of the dead-letter, but the WARN log lies.

#### Solution Applied

Raise `MalformedDataError` directly when the magic byte is present but no schema path is registered. This routes the message to the dead-letter via the same `FatalError` path the existing infrastructure uses, but with an accurate log message ("dead-lettering as malformed") rather than the misleading "falling back to JSON parse" warning.

#### Verification

- [x] `pytest services/alert/tests/unit -m unit` — 435 pass.
- [x] `pytest tests/architecture` — 99 pass.

### Issue F-003: decode_raw_array silent failure on every error branch (MAJOR — cross-agent)

#### Summary

`libs/contracts/src/contracts/events/nlp/article_enriched.py::decode_raw_array` returned `[]` on (a) None/empty input, (b) JSON parse error, (c) non-list root, and (d) non-dict items — with no logging. **Confidence**: HIGH
**Flagged by**: Security (F-005), Data Platform (F-002)

- Added structlog warnings on every drop branch (`raw_array_decode_oversized`, `raw_array_decode_json_error`, `raw_array_decode_not_list`, `raw_array_decode_dropped_non_dict_items`).
- Existing tests still pass; the helpers continue to return `[]` on error so consumer iteration is unaffected, but failures are now observable.

### Issue F-004: Architecture-test classifier bypassable (MAJOR — cross-agent)

#### Summary

`_classify_deserialize_value` only matched the substring `json.loads`. A future consumer using `from json import loads`, `orjson.loads`, `ujson.loads`, `simplejson.loads`, or — worst — `pickle.loads` would slip through as `UNKNOWN`. Pickle is RCE on attacker-controlled bytes.

#### Severity / Confidence

**Severity**: MAJOR
**Confidence**: HIGH
**Flagged by**: Security (F-006), Data Platform (F-007), QA (F-005)

#### Solution Applied

- Widened the JSON detector to a tuple covering 5 patterns: `json.loads`, `orjson.loads`, `ujson.loads`, `simplejson.loads`, `from json import loads`.
- Body window expanded from 30 to 60 lines; detector now also scans the file head for module-level imports.
- New test `test_no_forbidden_deserializers` blocks `pickle.loads`, `marshal.loads`, `yaml.load(`, `yaml.unsafe_load` unconditionally (RCE prevention).
- New test `test_classifier_self_test_for_synthetic_inputs` provides 4 synthetic input files to verify the classifier produces the expected `JSON_ONLY`, `AVRO_ONLY`, `AVRO_FIRST`, `JSON_ONLY` (aliased) classifications. Without this self-test, a regression in the classifier would silently turn the architecture test into a no-op.
