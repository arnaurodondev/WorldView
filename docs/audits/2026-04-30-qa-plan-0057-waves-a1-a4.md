### Issue F-DATA-01: Wave A-2 pre-clean DELETE uses `alias_id` for ordering, but `gen_random_uuid()` is NOT monotonic

#### Summary
The pre-clean DELETE in `0008_alias_unique_per_entity.py:47-55` keeps the row with the smallest `alias_id` for each `(entity_id, normalized_alias_text, alias_type)` triple, intending to preserve "the oldest". `entity_aliases.alias_id` is `gen_random_uuid()` (UUIDv4), which is NOT monotonic — so the row preserved is **arbitrary**, not the actually-oldest.

#### Severity / Confidence
**Severity**: MEDIUM (re-classified from data-platform agent's CRITICAL — production impact is minimal since the migration runs once, but the logic is unsound)
**Confidence**: HIGH
**Flagged by**: Data Platform Engineer

#### Root Cause Analysis
- **What**: `WHERE a.alias_id > b.alias_id` in the DELETE clause assumes `alias_id` is monotonic.
- **Why**: The migration comment claims "alias_id is monotonic (UUIDv7-like)" but the schema at `0001_create_intelligence_db.py:142` declares `alias_id UUID PRIMARY KEY DEFAULT gen_random_uuid()` which produces UUIDv4 (random).
- **When**: Only on first migration application against a database with duplicate aliases (the dev DB at session start had 32 of 38 alias rows as 4× duplicates).
- **Where**: intelligence-migrations 0008 pre-clean phase.
- **History**: Mistake stems from confusing the project's UUIDv7 default for IDs (`common.ids.new_uuid7()` used elsewhere) with `gen_random_uuid()` used in this specific schema.

#### Evidence
```sql
DELETE FROM entity_aliases a
USING entity_aliases b
WHERE a.alias_id > b.alias_id
  AND a.entity_id              = b.entity_id
  AND a.normalized_alias_text  = b.normalized_alias_text
  AND a.alias_type             = b.alias_type
```
- File: `services/intelligence-migrations/alembic/versions/0008_alias_unique_per_entity.py:47-55`

#### Impact
- **Immediate**: On migration run, duplicate aliases collapse correctly to **one row per triple** — but it is an arbitrary row, not the oldest. The kept row's `created_at` may be later than the deleted siblings'.
- **Blast radius**: Zero in production (migration runs once; if no duplicates exist, no rows are deleted; if duplicates exist, the result is still de-duplicated — just not optimally). Some loss of audit trail accuracy on the `created_at` of seed_demo TICKER aliases.
- **Data risk**: None — uniqueness invariant is enforced after migration regardless of which sibling was preserved.
- **User impact**: None (alias resolution doesn't depend on `created_at`).

#### Solution

**Replace ordering by `alias_id` with ordering by `created_at` (which IS reliably oldest-first):**

```sql
DELETE FROM entity_aliases a
USING entity_aliases b
WHERE (a.created_at, a.alias_id) > (b.created_at, b.alias_id)
  AND a.entity_id              = b.entity_id
  AND a.normalized_alias_text  = b.normalized_alias_text
  AND a.alias_type             = b.alias_type
```

The lexicographic `(created_at, alias_id)` ordering keeps the actually-oldest row; the secondary `alias_id` tiebreaker ensures determinism if two rows share a `created_at` timestamp.

**Effort**: Low (one migration line)
**Risk**: Low (no schema change, just ordering change)
**Recommended**: APPLY before the next migration run (no production rollback needed since the migration was authored just now).

#### Verification Steps
- [ ] Migration round-trip test still passes
- [ ] Pre-clean dedup test still passes (regardless of which sibling is kept, only one survives)

### Issue F-DATA-02: Wave A-3 alias INSERT uses bare `ON CONFLICT DO NOTHING` without explicit constraint target

#### Summary
The seed migration's alias inserts (`_emit_alias_block` in `0009_seed_canonicals_bootstrap.py`) use bare `ON CONFLICT DO NOTHING`. PostgreSQL accepts this and auto-resolves to whichever index/constraint matches, but the resolution depends on **which UNIQUE indexes exist** at migration-apply time. If migrations run out of order (0009 before 0008), the new `uidx_entity_aliases_entity_norm_type` index doesn't exist, and `ON CONFLICT` silently succeeds even when re-running the seed (no constraint matched), allowing duplicates to accumulate. The 0008 migration's pre-clean would then need to handle the post-0009 duplicates — circular dependency.

#### Severity / Confidence
**Severity**: MEDIUM
**Confidence**: HIGH
**Flagged by**: Data Platform Engineer

#### Root Cause Analysis
- **What**: `ON CONFLICT DO NOTHING` (bare form) at `_emit_alias_block` in `services/intelligence-migrations/alembic/versions/0009_seed_canonicals_bootstrap.py:1180-1184`.
- **Why**: Author wrote idiomatic PostgreSQL but didn't pin to a specific constraint, leaving the conflict-resolution path implicit.
- **When**: Only matters on re-apply or out-of-order migration scenarios; in the normal case (0008 then 0009) both `uidx_entity_aliases_entity_norm_type` and `uidx_entity_aliases_normalized` are present, and Postgres picks an applicable one.
- **Where**: intelligence-migrations 0009.

#### Evidence
```python
op.execute(
    f"-- F-CRIT-10 alias seed: {label}\n"
    "INSERT INTO entity_aliases "
    "(entity_id, alias_text, normalized_alias_text, alias_type, is_active, source) "
    f"VALUES {_values_clause_alias(rows)} "
    "ON CONFLICT DO NOTHING"  # ← no constraint target
)
```

#### Impact
- **Immediate**: None in the normal sequential migration path.
- **Blast radius**: A future migration that re-orders or replays could silently let duplicates accumulate.
- **Data risk**: Low (downstream queries tolerate duplicate aliases via DISTINCT or grouped JOINs).
- **User impact**: None.

#### Solution

**Specify explicit conflict target:**

```python
"ON CONFLICT (entity_id, normalized_alias_text, alias_type) "
"WHERE is_active = true "
"DO NOTHING"
```

This makes the dependency on the Wave A-2 index explicit and produces a clear error if the migration runs in the wrong order. Note: PostgreSQL allows `ON CONFLICT` with constraint columns even if the index doesn't exist yet, IF a UNIQUE constraint with those exact columns is defined — but since we use a UNIQUE INDEX (not a CONSTRAINT), this option requires the index to exist first. Net effect: the migration becomes order-dependent on 0008, which is the correct dependency.

**Effort**: Low (one migration patch)
**Risk**: Low

#### Verification Steps
- [ ] Re-apply 0009 against a database that already has 0008 applied → no error.
- [ ] Idempotency test still passes.
