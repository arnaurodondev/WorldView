## Issues — Full Investigation

### Issue F-001: E-5 startup repair silently misses ~50% of canonicals on multi-page runs

#### Summary
The Wave E-5 `repair_missing_embedding_state` function used `LIMIT :limit OFFSET :offset` pagination against a query whose result-set shrinks as the loop INSERTs missing rows. After page 1 inserted rows for the first 500 canonicals, those canonicals were removed from the gap-set; page 2's `OFFSET 500` then skipped the next 500 canonicals that DO have gaps. Net effect: silent ~50% data loss in the repair task — defeating the entire purpose.

#### Severity / Confidence
**Severity**: CRITICAL
**Confidence**: HIGH
**Flagged by**: Distributed Systems agent (DS-002)

#### Root Cause Analysis
- **What**: `embedding_state_repair.py` lines 88-126 used `OFFSET page_offset` against a `_GAP_QUERY` that only returns canonicals with missing rows. `page_offset` was incremented by `_PAGE_SIZE` after each iteration.
- **Why**: Adapted from a generic keyset-pagination pattern without recognising that the gap-set is monotonically shrinking — OFFSET semantics assume a stable result-set.
- **When**: Manifests whenever the gap-set exceeds a single page (`_PAGE_SIZE = 500`). At ~250 canonicals (current dev scale) the bug is invisible. At ~1000+ canonicals (post-A-3 seeds + future growth) ~50% of repairs would silently fail.
- **Where**: `services/knowledge-graph/src/knowledge_graph/infrastructure/workers/embedding_state_repair.py`
- **History**: Introduced 2026-05-01 in commit `df74f2a8` (PLAN-0057 QA iter-1 fix for M-2 — the optimisation that replaced per-row counts with a single GROUP BY). The OFFSET was a copy-paste artefact from the keyset-pagination version.

#### Evidence
```python
result = await session.execute(text(_GAP_QUERY + " LIMIT :limit OFFSET :offset"), {
    "limit": _PAGE_SIZE,
    "offset": page_offset,
})
...
page_offset += _PAGE_SIZE  # ← increments OFFSET against shrinking result-set
```
- **File**: `services/knowledge-graph/src/knowledge_graph/infrastructure/workers/embedding_state_repair.py:88-126` (pre-fix)

#### Impact
- **Immediate**: For canonical catalogues > 500 entities, ~50% of canonicals with gaps would be skipped per startup run.
- **Blast radius**: Embedding refresh workers cannot pick up entities without `entity_embedding_state` rows → embeddings stay NULL → ANN search degraded for those entities → silent quality degradation in entity resolution.
- **Data risk**: No corruption (entries that were inserted are correct; entries that were skipped just don't exist), but recovery requires restart cycles or live-write path coverage.
- **User impact**: Invisible until ANN search returns wrong nearest neighbours.

#### Solution Options

**Option A (CHOSEN): Drop OFFSET, re-query from top**
- Replace OFFSET pagination with always re-querying `LIMIT :limit` (no offset). The gap-set monotonically shrinks, so the loop terminates when the gap query returns zero rows. Add an iteration cap (100) as a safety net for permanent insert failures.
- **Changes**: `embedding_state_repair.py:60-126` (rewrite loop body); `test_embedding_state_repair.py` add 2 new regression tests (multi-page + iteration cap).
- **Benefits**: Correct semantically; resilient to concurrent inserts; iteration cap protects against runaway loops.
- **Drawbacks**: Re-runs the gap query each page (acceptable; query is fast on indexed columns).
- **Effort**: Low. **Risk**: Low.

**Option B (rejected): Track inserted entity_ids in memory, exclude in WHERE clause**
- Maintain a Python set of processed entity_ids; `WHERE NOT IN (:ids)`.
- **Drawbacks**: Memory grows O(canonicals); WHERE NOT IN with a large list is slow in Postgres; complexity not justified.

#### Recommended Option
**Option A** — minimal diff, correct semantics, defended by 2 new regression tests.

#### Verification Steps
- [x] `python -m pytest tests/unit/infrastructure/workers/test_embedding_state_repair.py -v` — 7/7 pass (2 new tests cover the multi-page + iteration-cap branches)
- [x] Full KG suite: 680/680 pass

### Issue F-002: InstrumentRepository.upsert clears flags under concurrent INSERT race

#### Summary
When ohlcv_consumer and quotes_consumer both call `upsert(Instrument(flags=...))` for the same (symbol, exchange) pair (after both seeing `find_by_symbol_exchange == None`), the second INSERT becomes an UPDATE that uses unconditional `set_={...}`. This silently clears the flag the first INSERT just set.

#### Severity / Confidence
**Severity**: MAJOR
**Confidence**: HIGH
**Flagged by**: Distributed Systems agent (DS-001) + (Data Platform agent disagreed with PASS — verified DS was correct via direct code reading)

#### Root Cause Analysis
- **What**: `instrument_repo.py:162-187`'s `on_conflict_do_update` used `set_={"has_ohlcv": instrument.flags.has_ohlcv, ...}` — overwriting the existing row's flags with the incoming values.
- **Why**: Author copied the field list from the INSERT clause, missing that ON CONFLICT semantics need OR-merge for boolean flags. The adjacent `update_flags()` correctly uses `case((flags.has_ohlcv, True), else_=Model.has_ohlcv)`.
- **When**: Race window between `find_by_symbol_exchange` returning None and the consumer's `upsert` call. Bounded by single Kafka consumer's poll loop, but two consumers (ohlcv + quotes) compete.
- **Where**: `services/market-data/src/market_data/infrastructure/db/repositories/instrument_repo.py`
- **History**: Pre-PLAN-0057 (likely original PLAN-0010 / market-data wave). Surfaced by F-DS-06 deferral in C+D iter-1 audit.

#### Evidence
Pre-fix:
```python
.on_conflict_do_update(
    constraint="uq_instruments_symbol_exchange",
    set_={"has_ohlcv": instrument.flags.has_ohlcv, ...},  # not OR-merged
)
```
Adjacent (already-correct) pattern in `update_flags`:
```python
.values(has_ohlcv=case((flags.has_ohlcv, True), else_=InstrumentModel.has_ohlcv), ...)
```

#### Impact
- **Immediate**: For symbols that get exactly one OHLCV event AND one quotes event simultaneously on first observation, the `has_ohlcv` OR `has_quotes` flag may be wrong.
- **Blast radius**: Internal market-data flag-based queries see stale flags. Cross-service `InstrumentDiscovered` event payload does NOT include flags, so downstream services unaffected.
- **Data risk**: Self-healing on next event for that symbol (the elif branch at `ohlcv_consumer.py:230` calls `update_flags` which uses OR-merge correctly). Symbols that get ONLY one event of each type stay wrong indefinitely.
- **User impact**: Negligible for well-traded symbols; potential edge for thinly-traded instruments.

#### Solution Options

**Option A (CHOSEN): Mirror `update_flags`' atomic OR-merge in `upsert`**
- Use `case((excluded.has_ohlcv, True), else_=InstrumentModel.has_ohlcv)` for each flag in the `set_={}` dict.
- **Changes**: `instrument_repo.py:162-187` only.
- **Benefits**: Atomic at the SQL level; consistent with adjacent code; no caller changes needed.
- **Effort**: Low. **Risk**: Low.

#### Recommended Option
**Option A** — adopted.

#### Verification Steps
- [x] market-data unit suite: 556/556 pass
- [x] Existing tests still green (none asserted the buggy behaviour)

### Issue F-004: KG embedding-state repair lifespan call is unbounded

#### Summary
The Wave E-5 `repair_missing_embedding_state` ran inside the FastAPI lifespan as a synchronous `await` with no timeout. A slow `intelligence_db` (lock contention, replica lag, pgvector hot loop) would block API readiness past the k8s probe window → pod restart → repair re-runs against the same slow DB → crash loop.

#### Severity / Confidence
**Severity**: MAJOR
**Confidence**: HIGH
**Flagged by**: Distributed Systems agent (DS-004)

#### Fix Applied
Wrapped the await with `asyncio.wait_for(..., timeout=30.0)`. Initial fix used `asyncio.create_task` to background the work, but that violated R22 / TOPO-LIFESPAN architecture rule (no background tasks in lifespan). Final fix: bounded synchronous await with explicit `TimeoutError` handler that logs `kg_embedding_state_repair_timeout` and continues startup. Live-write path (InstrumentEntityConsumer / InstrumentDiscoveredConsumer) covers any gap.
