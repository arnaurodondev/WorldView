## Findings

### B-1 (BLOCKING) — `chunk_text_key` is a MinIO path, not chunk text

**Discovery**: applied alembic 0017 to live `nlp_db` (9706 chunks). Schema was correct
(2 denorm columns + 2 GENERATED tsvectors + 2 GIN indexes). But:

```
SELECT count(*) FROM chunks WHERE tsv_english @@ websearch_to_tsquery('english', 'Apple');
-- returns 0
```

Inspecting `chunks.chunk_text_key`:
```
SELECT chunk_text_key FROM chunks LIMIT 1;
-- "nlp-pipeline/chunk-text/<uuid>/<uuid>/body/v1.txt"
```

`chunk_text_key` holds a MinIO object path. The GENERATED tsvectors were tokenizing
file paths (UUIDs and `body/v1.txt`), not chunk content.

**Why tests didn't catch it**: integration test fixture `_seed_chunk` did
`chunk_text_key=<actual chunk text>` (line ~270 of `test_chunk_lexical_search.py`)
to make the GENERATED tsvector see real text. The 11 lexical-search tests passed
green against this fake fixture — masking that the production data path was broken.
This is an R19-spirit violation: the test does not exercise the production code path.

**Root cause**: the plan body itself (PLAN-0063 §5 W5-2-01) and §0-bis L7 specify
`chunk_text_key` as the FTS source, missing that chunk text was offloaded to MinIO
in an earlier infra optimisation. The subagent followed the plan literally.

**Fix**: added a denormalised `chunk_text TEXT` column to `chunks`, populated from
`Chunk.text` at insert time by `ChunkRepository.add()`. GENERATED `tsv_english` and
`tsv_simple` now reference `chunk_text`. MinIO retains the canonical body via
`chunk_text_key`; the new column is ~10-50MB per 50K chunks (acceptable for MVP).

**Files touched in fix**:
- `services/nlp-pipeline/alembic/versions/0017_add_chunks_tsv_english_gin.py` —
  added `chunk_text TEXT` column + updated GENERATED expressions to reference it
  + extended downgrade to drop `chunk_text` after the GENERATED columns
- `services/nlp-pipeline/src/nlp_pipeline/infrastructure/nlp_db/models.py` —
  added `chunk_text` Mapped column + updated docstring
- `services/nlp-pipeline/src/nlp_pipeline/infrastructure/nlp_db/repositories/chunk.py` —
  pass `chunk_text=chunk.text` in `pg_insert`
- `services/nlp-pipeline/src/nlp_pipeline/infrastructure/nlp_db/repositories/chunk_search.py` —
  added `chunk_text` to lexical_search SELECT + result dict (snippet without MinIO fetch)
- `services/nlp-pipeline/tests/integration/test_chunk_lexical_search.py` —
  updated bootstrap to add `chunk_text` and reference it in GENERATED expr; updated
  `_seed_chunk` to use `chunk_text=<body>` and stuff a placeholder MinIO path in
  `chunk_text_key`

### B-2 (subtle) — Stale GENERATED expression in long-lived test DB

**Discovery**: after the B-1 fix landed in source, integration tests still failed
because the test DB (`worldview-test-postgres-1`) had `tsv_english` from an earlier
bootstrap with the old expression. `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` is a
no-op when the column already exists, so the test bootstrap silently kept the old
GENERATED expression.

**Fix**: the bootstrap now drops `tsv_english` / `tsv_simple` + their indexes
**before** re-creating them, so each test session starts from clean DDL. Added
inline comment explaining the trap.

**One-time DB fix**: dropped the stale columns manually on the test DB so the next
bootstrap recreated them with the correct expression.

**BUG_PATTERNS**: filed as BP-402.
