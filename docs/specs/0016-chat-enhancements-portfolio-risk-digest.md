## 1. Problem Statement

PRD-0015 designs a unified LLM prompt for all query types. In practice, financial questions have radically different answer structures: factual lookups need concise citations; relationship queries need hop-by-hop graph traversal; financial data queries need structured tables. A single prompt produces mediocre answers across all types.

Multi-turn conversations re-execute the full retrieval pipeline on every turn, including for follow-up questions about the same entity/topic. This wastes latency and LLM inference budget when the same chunks are retrieved repeatedly.

Portfolio users have no proactive intelligence delivery. They must actively query the chat to discover risk signals that may already exist in the knowledge graph. A weekly automated digest would surface this information without requiring user-initiated queries.

## 3. Functional Requirements

### 3.1 Must-Have (v1)

| F02 | Add `GENERAL` as a 9th query intent — smart fallback for ambiguous/educational queries |

### 6.7 Data Flow

#### Weekly Email Digest Flow
```
S10 EmailScheduler (weekly cron)
  → Query email_preferences WHERE weekly_digest_enabled=true AND send_day_of_week=today
  → For each user:
      1. GET /internal/v1/portfolio/context from S1 (holdings, watchlist, alerts)
      2. GET /api/v1/ohlcv/bulk from S3 (7-day bars for each held instrument_id)
      3. GET /api/v1/fundamentals from S3 (fundamentals for held instruments)
      4. POST /internal/v1/briefings to S8 (portfolio_context + market_snapshots + active_signals)
      5. Receive: narrative + risk_summary + citations
      6. Render HTML email template with sections:
           [Risk Overview] → risk_summary.top_risk_signals
           [Portfolio Positions] → concentration + YTD performance from OHLCV
           [Recent News] → citations from briefing
           [Market Fundamentals] → key metrics from risk_summary
      7. EmailProvider.send(to=user_email, subject="Your weekly portfolio brief", ...)
      8. INSERT INTO email_log
      9. UPDATE email_preferences SET last_digest_sent_at = now()
      10. Produce alert.email.sent.v1 via outbox
```

#### 3-Layer Context Assembly Flow (S8)
```
User query (turn N)
  → Intent classification (Qwen2.5:3b)
  → Entity resolution (S6)
  → Check chunk cache: load s8:ctx:chunks:{thread_id}:{N-1}
      → If (entity_overlap>50% AND intent=prev_intent AND query_sim>0.85):
            REUSE chunks from Valkey → skip retrieval
        Else:
            Full retrieval (S6 ANN + S7 graph + S3 SQL)
  → Load turn summaries: s8:ctx:summary:{thread_id}:1 ... {N-2}
  → Load last turn verbatim: messages JOIN where turn_num = N-1
  → Assemble ConversationContext (≤6000 tokens)
  → LLM completion stream (DeepSeek R1 Distill 32B via DeepInfra; fallback: OpenRouter → Ollama local)
  → Stream tokens to client via SSE
  → Post-stream (async, non-blocking):
      a. Store chunks → s8:ctx:chunks:{thread_id}:{N} (TTL 4h)
      b. LLM call: generate turn summary → s8:ctx:summary:{thread_id}:{N} (TTL 24h)
      c. Persist message to rag_db (content, citations, context_valkey_key, summary_valkey_key)
```

## 7. Architecture Decisions

### AD-01: Provider-Agnostic Email Interface
**Decision**: Define `EmailProvider` as a Python Protocol. Factory selects adapter from `ALERT_EMAIL_PROVIDER` env var. Default: Resend. No code changes needed to switch providers.
**Alternatives considered**: (A) Direct Resend SDK dependency — faster but creates vendor lock-in. (B) Abstract base class — heavier than Protocol.
**Rationale**: Protocol is the correct Python pattern for structural typing. Resend's free tier (3K/month) covers thesis and early startup. Switch to SendGrid or SES by changing env var when volume grows.

### AD-02: Triple-Condition Chunk Cache Reuse
**Decision**: All three conditions required: entity overlap > 50%, same intent class, query semantic similarity > 0.85.
**Rationale**: Entity overlap alone is insufficient — different questions about the same company need different retrieval (FACTUAL vs RELATIONSHIP). The 0.85 semantic threshold ensures genuine follow-up detection. False positives (reusing wrong chunks) degrade response quality more than false negatives (redundant retrieval) degrade performance.

### AD-03: Async Turn Summaries
**Decision**: Turn summary generation fires after streaming completes, not before. Not awaited by the request handler.
**Rationale**: Adds 2-3 seconds of latency to each turn if synchronous. Async means the user receives their answer immediately. Risk: if the process crashes before summary completes, the next turn uses one fewer compressed turn (graceful degradation to fresh retrieval).

### AD-04: External-First LLM Provider Chain
**Decision**: Primary chat completion is DeepSeek R1 Distill 32B via DeepInfra API (REST, no VRAM requirement). Provider fallback chain: DeepInfra → OpenRouter (same model: `deepseek/deepseek-r1-distill-qwen-32b`) → Ollama local (emergency last resort). Only two external providers are used. Config: `RAG_CHAT_COMPLETION_PROVIDER` (`deepinfra` | `openrouter` | `ollama`) + `RAG_CHAT_COMPLETION_MODEL` (model ID per provider).
**Alternatives considered**: (A) Qwen2.5:32b local-first (18.5GB VRAM fits production server but unavailable in dev/cloud) — creates environment disparity; (B) OpenAI GPT-4o — higher cost, no reasoning trace; (C) Anthropic Claude — preference for Gemini for descriptions, DeepSeek for inference.
**Rationale**: External-first eliminates cold-start latency (Ollama model loading), removes VRAM constraint from primary path, and enables dev/prod parity. DeepInfra offers DeepSeek R1 Distill at <$1/M tokens — cost-competitive with local inference at reasonable throughput.

**Prefix caching benefit**: External providers (DeepInfra, OpenRouter) auto-cache prompt prefixes >1024 tokens at ~50% token cost discount. The 3-layer context assembly order is optimized for caching: (1) static system prompt [cacheable] → (2) turn summaries [cacheable once computed] → (3) dynamic retrieval context + current query [never cached]. This structure maximizes cache hits on long conversations.

**Entity description model**: `gemini-3.1-flash-lite` via Google AI Studio (see PRD-0017). Gemini chosen for lowest latency + cost for batch world-knowledge lookups. NOT configurable (hardcoded in S7 `DefinitionRefreshWorker`).

## 8. Security Analysis

| Threat | Mitigation |
|--------|-----------|
| Email preferences visible to other users | `X-Tenant-ID + X-User-ID` must match DB row; 404 if not owned |
| S8 briefing endpoint called by external | `X-Internal-Token` required; token rotated via env var |
| Turn summaries leaking cross-user | Valkey key includes `thread_id` (UUIDv7, not guessable); S8 verifies thread ownership |
| Email log reveals financial positions | Log table stores only metadata (email_type, status, subject) — no portfolio data |
| Prompt injection via portfolio data | Portfolio data is injected into EMAIL_DEEP_BRIEF prompt as XML-wrapped system context, not as user input |

## 9. Failure Modes

| Dependency | Failure | Behavior |
|------------|---------|----------|
| **Email provider (Resend) down** | `EmailProviderError` | Retry 3× with exponential backoff (1s, 2s, 4s). Log status=failed. Skip user for this week. |
| **S8 briefing endpoint timeout** | HTTP 503 / timeout > 90s | Send email with partial data sections + notice: "AI narrative unavailable this week." Log error. |
| **S1 portfolio context unavailable** | HTTP 503 | Skip user for this week. Log status=skipped with reason. |
| **Valkey chunk cache miss** | Cache miss | Full retrieval (degraded performance, correct behavior). |
| **Turn summary LLM call fails** | Exception | Log warning. Context assembly for next turn uses one fewer compressed turn — falls back to older turns verbatim. |
| **S3 OHLCV unavailable** | HTTP 503 | Email sent without OHLCV section. Risk section still sends. |

## 10. Scalability

- **Email digest batch**: Sequential per-user processing (not concurrent) prevents S8 briefing rate limit exhaustion. Expected users: <1000 for thesis/early startup → 1000 × 90s = 25 hours theoretical max, but realistically <100 users → <2.5 hours batch window (overnight Saturday → Sunday morning).
- **Chunk cache memory**: 12 chunks × ~300 bytes × 4h TTL × concurrent users. At 100 concurrent users with 5-turn conversations: ~18MB. Negligible.
- **Turn summary storage**: 150 tokens × ~6 bytes/token × 24h TTL × 100 users × 5 turns: ~450KB. Negligible.
- **Context token budget**: Hard cap at 6000 tokens prevents model context overflow regardless of conversation length.
