## 2. Wave A — Pre-Flight & Live-Stack Warm-Up

**Goal**: Bring local stack to "audit-ready" state — populated, healthy, observable.
**Depends on**: none.
**Estimated effort**: 1h.
**Architecture layer**: infrastructure / ops.
**Parallelism**: sequential.

### Tasks

#### T-A-1 — Verify branch state + clean working tree
- **Type**: ops
- **depends_on**: none
- **Steps**:
  - `git status` clean; on `feat/content-ingestion-wave-a1` or your demo branch
  - `git pull` to sync with remote
- **Acceptance**: `git status` reports clean working tree and a known branch HEAD.

#### T-A-2 — Refresh secrets + start full stack
- **Type**: ops
- **depends_on**: T-A-1
- **Steps**:
  - `make fetch-secrets`
  - `make dev-rebuild` (if last build is stale) or `make dev`
  - Wait for boot; `docker compose ps` and confirm ≥54 containers `healthy`
- **Acceptance**: All expected containers healthy; no `unhealthy` or `restarting`.

#### T-A-3 — Seed corpus + warm pipelines
- **Type**: ops
- **depends_on**: T-A-2
- **Steps**:
  - `make seed`
  - Trigger ingestion for at least 5 source feeds; wait until `articles` count grows by ≥200
  - Trigger top-50 entity enrichment via KG scheduler
  - Verify embeddings populated for the same top-50
- **Acceptance**: per-table sanity counts:
  - `articles` ≥500 in last 24h
  - `entities` (canonicals) ≥400
  - `entity_summaries` rows for top-50 entities ≥45 (allow ≤5 misses)
  - `chunk_embedding_state` populated; `entity_embedding_state` HNSW index in use
  - `prediction_market` rows refreshed
  - `economic_event` rows for next 7d
  - OHLCV bars present for top-25 tickers (≥30 bars each)

#### T-A-4 — Verify worldview-web container builds and serves
- **Type**: ops
- **depends_on**: T-A-2
- **Steps**:
  - Confirm `worldview-web` container exists in `infra/compose/docker-compose.yml`; if it doesn't, raise as immediate defect (D-INIT-1) and block — without the container we can't rehearse against the production build per PRD locked decision.
  - `curl -fsS http://localhost:3001/healthz` (or the configured port)
  - Open `http://localhost:3001` in Chromium; confirm SSR HTML returns; no FOUC
- **Acceptance**: 200 on healthz; landing renders.

#### T-A-5 — Open the live observability tabs
- **Type**: ops
- **depends_on**: T-A-2
- **Steps**:
  - Grafana dashboards: trace_id-injected logs visible (PLAN-0054 artefact); Kafka lag panel green
  - Tail ERROR / Traceback log filter via Loki (one query saved)
- **Acceptance**: tail visible and ready for Wave B's runtime monitoring sweep.

#### T-A-6 — Initialise Defect Register
- **Type**: docs
- **depends_on**: none
- **Steps**:
  - Create `docs/audits/2026-05-09-pre-demo-qa-defect-register.md` with the schema from PRD §11
  - Add the §11.2 summary tables (severity counts + per-VA coverage), zeroed out
- **Acceptance**: file exists; agents in W-B can append to it.

### Validation gate (W-A exit)
- All `make dev` containers healthy
- Live data populated to the §T-A-3 thresholds
- worldview-web container reachable
- Defect Register file ready

## 3. Wave B — Audit (Parallel Swarms)

**Goal**: Find every defect on the demo path. Produce a populated Defect Register.
**Depends on**: W-A.
**Estimated effort**: 6h wall clock with 10 agents in flight.
**Architecture layer**: read-only investigation.
**Parallelism**: 10 agents in parallel (Swarm 1 R1–R4, Swarm 2 F1–F3, Swarm 3 P1–P3).

### Dispatch protocol (executed by main session)
Spawn all 10 agents in a single message via 10 `Agent` tool calls with:
- `subagent_type: general-purpose`
- `isolation: "worktree"` (enables independent file-system state for any read-only fixtures they need)
- `run_in_background: true`
- Each prompt is self-contained: links PRD-0087 §6 swarm spec, the Defect Register path, the agent's specific VA scope, and the report-back deliverable schema (1 register entry per finding).

## 4. Wave C — Cross-Cutting Sweeps + Defect Consolidation

**Goal**: Fill in the gaps Swarm 1–3 couldn't cover; consolidate the register; assign triage labels.
**Depends on**: W-B.
**Estimated effort**: 4h.
**Parallelism**: 2 agents (sweep A + sweep B).

### Tasks

#### T-C-1 — Endpoint smoke matrix (Sweep A)
- **Type**: audit
- **depends_on**: W-B
- **Agent**: spawned via Agent tool (general-purpose, background, no worktree needed)
- **Scope**: Build the call list from `apps/worldview-web/lib/api/*.ts` (every imported function); for each, hit S9 with realistic auth + payload; record status, latency p50/p95, response size; flag any 4xx/5xx, p95 >2s, empty response when data should exist.
- **Deliverable**: `docs/audits/2026-05-09-endpoint-smoke-matrix.md`.

#### T-C-2 — Live runtime monitor sweep (Sweep B)
- **Type**: audit
- **depends_on**: W-B (runs concurrently with T-C-1)
- **Agent**: spawned via Agent tool
- **Scope**: For the duration of W-B (which already happened), tail container logs via Loki; collect ERROR/Traceback/panic; collect Kafka consumer-group lag >1k; collect DB slow queries >500ms; collect outbox backlog >100.
- **Deliverable**: `docs/audits/2026-05-09-runtime-anomaly-log.md`.

#### T-C-3 — Defect Register consolidation
- **Type**: triage
- **depends_on**: T-C-1, T-C-2
- **Done by**: main session (me)
- **Steps**:
  - De-duplicate register entries (same root cause across multiple agents)
  - Re-classify any entry where severity is ambiguous
  - Map each entry to (a) one VA, (b) one or zero demo path surfaces, (c) one severity code
  - Update the §11.2 summary tables
- **Acceptance**: register has zero "untriaged" rows; severity counts match; per-VA coverage table accurate.

### Validation gate (W-C exit, == G1)
PRD §12.1 conditions:
- All 7 primary + 3 frontend agents submitted
- Endpoint smoke matrix complete
- Defect Register fully populated and triaged
- Go/no-go answered: **can demo path be made green within 18 hours?**

If G1 fails: extend by ≤2h; if still failing, escalate to user with full state.

This is a QA program. Avro / API / DB schemas should not change in fix waves except to fix bugs. If a fix requires a schema change (rare), that fix is automatically a subagent escalation (>4h almost always).

## 14. Risk Assessment

### 14.1 Critical path
W-A → W-B → W-C → W-D → (W-E, W-F, W-G in parallel) → W-H → W-I → W-J → W-K.
The critical path runs through W-B (audit must complete before fixes can be triaged) and W-H (G2 gate; can't enter rehearsal if HF still open).

### 14.2 Highest-risk waves
- **W-B** — if agent quality is uneven, defects may be missed; mitigation: cross-cutting sweeps (W-C) catch what swarms missed
- **W-G** — frontend container rebuild after every batch is slow; mitigation: batch frontend commits in larger chunks
- **W-H** — subagent merges are where regressions sneak in; mitigation: mandatory `/review` and re-test of original failure

### 14.3 Rollback strategy
Every fix is a single commit on a feature branch. If a fix introduces regression: `git revert <SHA>`. Spawned subagents are in worktrees; cancellation is `git worktree remove --force <path>`.

### 14.4 Testing gaps
Real-time observability (Grafana panels) is part of the demo surface but not under test. Mitigation: T-A-5 confirms the panels work; W-K pre-flight re-verifies.
