# Test Execution Report Guide

## Overview

The Test Execution Report (TEST_EXECUTION_REPORT.md) is auto-generated by `scripts/test-full.sh` and provides a comprehensive view of test suite health across the system. This guide describes the report structure and how to interpret the metrics.

## Report Sections

### 1. Header & Environment
- **Generated at**: UTC timestamp of report generation
- **Run ID**: Unique identifier for this test run (format: YYYYMMDDTHHMMSSz)
- **Run artifacts**: Directory containing detailed logs and metadata
- **Run duration**: Total time for all test suites (wall clock, not cumulative)
- **Environment**: Python version, Docker version, compose version, git branch/commit
- **Integration mode**: Sequential (one suite at a time) or parallel-safe (with safety checks)

### 2. Summary (High-Level Counts)
| Metric | Meaning |
|--------|---------|
| `Test suites passed` | Number of suite entries with status=passed |
| `Test suites failed` | Number of suite entries with status=failed |
| `Test suites skipped` | Number of suite entries with status=skipped or failure_type=no_tests |
| `Total collected tests` | Aggregate test count across all discovered test suites |
| `Total failed tests` | Aggregate count of individual failing tests from JUnit XML |

**Important**: "suites" ≠ "tests". A suite is a collection of tests (e.g., "market-data:unit"). The `Total collected tests` is an aggregation of all pytest discoveries. The `Total failed tests` is extracted from JUnit XML files.

### 3. Aggregated Metrics

#### By Layer
Metrics rolled up by test layer. Each row represents one layer (architecture, libs, unit, contract, integration, e2e, infra).

| Column | Meaning |
|--------|---------|
| Suites (P/F/S) | Passed/Failed/Skipped count of **suites** in this layer |
| Collected Tests | Sum of all tests discovered in this layer |
| Failed Tests | Sum of all individual test failures in this layer |
| Duration (s) | Total wall-clock time for this layer |

**Pass rate = (Collected Tests - Failed Tests) / Collected Tests**

Example: if layer has 100 collected tests and 2 failed, pass rate is 98%.

#### By Service
Metrics rolled up by service (market-data, market-ingestion, portfolio, etc.). Each row represents one service or system component.

| Column | Meaning |
|--------|---------|
| Suites (P/F/S) | Passed/Failed/Skipped count of **suites** for this service |
| Collected Tests | Sum of tests for all layers of this service |
| Failed Tests | Sum of failures across all test layers/types |
| Duration (s) | Total execution time for all tests of this service |

**Service Health**:
- **Green** (✓): All suites passed, no failed tests
- **Yellow** (⊘): Some suites skipped (common for services with no tests for a layer)
- **Red** (✗): Failed suites or failed tests

### 4. Failure Hotspots
Ranked list of suites with the most failures. Helps prioritize debugging.

```
## Failure Hotspots
- market-data:e2e: 19 failed tests       ← most failures first
- api-gateway:unit: 1 failed tests
- portfolio:e2e: 0 failed tests          ← no failures in this suite
```

### 5. Infra Status
Docker compose health and artifact location. If status=failed, no tests ran; check compose logs in the ``infra/`` subdirectory.

### 6. Suite Results
Detailed one-line entry for each suite executed. Format:

```
- <service>:<layer>: <status> (layer=<layer>, type=<type>, collected=<N>, duration=<S>s[, failure_type=<reason>]) [- <reason>]
```

Example:
```
- market-data:unit: passed (layer=unit, type=pytest, collected=248, duration=7s)
- api-gateway:unit: failed (layer=unit, type=pytest, collected=0, duration=0s, failure_type=script_failure) - pytest exited with code 4
- content-ingestion:e2e: skipped (layer=e2e, type=pytest, collected=0, duration=0s, failure_type=no_tests) - no e2e tests
```

### 7. Failed Tests (Details)
For each individual failing test, includes:
- **Test name**: Fully qualified test path
- **Suite**: Which suite (e.g., market-data:e2e)
- **Kind**: `error` (setup/teardown), `failure` (assertion), `script_failure` (pytest crashed)
- **Reason**: Short description of the failure
- **Log reference**: Path to detailed suite log
- **Trace excerpt**: First 50-100 lines of Python traceback (if available)

## Key Metrics to Monitor

### Per-Service Health Checklist
- [ ] All suites have at least "contract" or "unit" tests
- [ ] Integration/e2e should have >0 collected tests or explicit skip reason
- [ ] Failed test count should be 0 or declining
- [ ] Duration should be <10s for unit, <60s for integration/e2e

### Cross-Layer Health
- **Architecture**: Should always pass (system design tests)
- **Libs**: Should always pass (shared code guardrails)
- **Unit**: Should have >95% pass rate (individual component tests)
- **Contract**: Should pass (API/event contract validation)
- **Integration**: Should pass (full-service tests)
- **E2E**: Should have high pass rate; transient failures are common

### Red Flags
1. `api-gateway:unit: failed (failure_type=script_failure)` → pytest couldn't run, check service config
2. `market-data:e2e: failed` with 15+ failures → likely fixture/setup issue or data race
3. Any layer with duration >120s → performance regression or stuck test
4. High variance in duration between runs → non-deterministic test behavior

## Interpreting Common Failures

### "fixture_failure" / "OSError: Connect call failed"
Database isn't ready. Usually happens when docker-compose failed to reach the "healthy" state. Check `infra/compose.all.log`.

### "pytest exited with code 4"
Pytest couldn't import the test module or find tests. Usually indicates missing dependency or import error. Check the suite log file.

### "UniqueViolationError" / "duplicate key"
Test fixture didn't clean up data properly, or previous test left data behind. Check fixture isolation and transaction handling.

### "httpx.ConnectError" / "Connection refused"
API service didn't start or is no longer listening. Common during compose startup race conditions. Rerun the test; usually transient.

## Artifacts
Each run generates a directory in `docs/testing/test-runs/<RUN_ID>/`:
- `TEST_EXECUTION_REPORT.md` - Full report (copy of latest)
- `TEST_EXECUTION_SUMMARY.json` - Machine-readable metrics
- `suites/` - Individual test logs per suite (one file per layer/service combination)
- `junit/` - JUnit XML files (used for failure extraction)
- `infra/` - Docker Compose output, health checks, diagnostic info

## Best Practices

1. **Before committing**: Run `scripts/test-full.sh` to ensure comprehensive test health
2. **On CI**: Set `--retain-logs always` and check full report before merge
3. **During development**: Run targeted `pytest` for quick feedback; use full suite for final validation
4. **When investigating failures**: Always check the full trace in `suites/` directory, not just the excerpt
5. **For flaky tests**: Look for duration variance and connection errors; consider retry logic or better setup/teardown
